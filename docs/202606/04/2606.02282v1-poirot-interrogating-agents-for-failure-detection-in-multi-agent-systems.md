## Abstract
Orchestrating Large Language Models into Multi-Agent Systems (LLM-MAS) has unlocked remarkable reasoning capabilities, yet emergent failures and hallucinations that resist characterisation block their deployment in safety-critical domains -- a gap made legally untenable by emerging AI regulation. Existing evaluation paradigms share a common flaw: centralised judgment creates single points of failure and demands domain-specific expertise. Here we present POIROT, a protocol that repurposes a system's own agents as its diagnostic layer, leveraging the epistemic diversity already present in the architecture. Across evaluated settings, POIROT outperforms single-LLM evaluator baselines, with gains that scale with problem complexity (OR = 1.60, $p = 0.008$), agent count, and fault dimensionality, persisting under compound fault conditions. These results demonstrate that safety oversight need not be externalised: the agents executing a role carry sufficient collective intelligence to audit it. We release POIROT as an open-source library alongside BLAME, a benchmark for fault attribution in safety-critical multi-agent systems.

## 2. 方法论：核心思想与技术细节

**核心思想：**
将多智能体系统自身的agents重新利用为诊断层，通过结构化的同伴质询和危险感知投票聚合，实现分布式故障诊断。

**协议架构（POIROT）：**
系统被表示为元组 $S = (A, C, H, U)$，其中：
- $A = \{A_1, ..., A_p\}$：agents集合
- $C$：软件组件
- $H$：硬件元素
- $U$：人类交互者

**五阶段流程：**

| 阶段 | 名称 | 描述 |
|------|------|------|
| Phase I | 危险空间定义 | POIROT空间构造器构建N维向量空间 $H_{haz} \subseteq R^N$，每个轴对应一个结构化风险源 |
| Phase II | 自我评估 | 每个agent独立重建会话轨迹，生成定性报告识别异常或确认无异常 |
| Phase III | 同伴质询 | agents交换报告，通过结构化对话对比解释，收集超出本地视角的上下文信息；此阶段观察与归因分离，不进行公开投票 |
| Phase IV | 私人投票 | 每个agent发出二进制归因向量 $v_i \in \{0,1\}^N$，$v_i[j]=1$ 表示该agent将故障归因于组件j；投票隐私防止策略性对齐 |
| Phase V | 危险感知聚合 | 通过距离加权共识机制聚合投票，赋予与危险维度结构性接近的agents更大影响力 |

**权重计算公式：**
```python
# 代理Ai的位置向量 pi 为单热编码（自身位置为1）
# 归因向量 vi 由代理投票产生
# 使用汉明距离测量结构差异
d_H(p_i, v_i) = Σ_{k=1}^N [p_i[k] ≠ v_i[k]]

# 权重计算
w_i = β + 1/2 × (1 - d_H(p_i, v_i)/N)

# 其中 β ∈ [0, 0.5] 为基线项，确保每个agent最小影响力
# 设定 β = 0.25，权重范围 [0.25, 0.75]

# 最终归因分数
S_j = Σ_{i=1}^p w_i × v_i[j]
```

**关键特性：**
- 自归因获得最大影响力，反映对局部故障源的直接认知访问
- 权重与结构距离线性关联，模拟信息亲近度降低
- 与均匀多数投票不同，距离加权聚合将认知影响力与操作接近度对齐

**实现方式：**
- 使用LangGraph构建为StateGraph多智能体框架
- 每个agent封装为LLM节点和动作节点
- 专用poirot_server节点中介所有代理间通信
- 每agent最多8次LLM调用限制防止无限循环

## 6. 主要结论与发现

### 6.1 性能提升
- POIROT在所有评估设置中一致优于单LLM评估器基线
- 优势随问题复杂度增加而扩大（OR=1.60，p=0.008）
- 在CORTEX中，总体增益范围为+6.7pp至+25.6pp
- 在TradingAgents中，POIROT在所有模型配置下均超过最佳基线

### 6.2 复杂度效应
- 低复杂度（Q1/Q2）：POIROT提供中等改进（12%-28%）
- 高复杂度（Q4）：POIROT一致优于基线，DeepSeek提升33%，Gemini提升46%
- 问题复杂度是POIROT相对于基线优势的主导驱动因素

### 6.3 Agent数量效应
- 在高复杂度案例（Q3/Q4）中，agent数量效应估计增强（OR=2.36）
- 在整体分析中，agent数量效应未达显著水平

### 6.4 复合故障挑战
- 多故障条件下准确率显著下降（CORTEX最佳结果28.8%）
- POIROT在3/4模型中仍保持优势
- 组合难度反映安全关键临床系统中的实际挑战

### 6.5 个体vs聚合性能
- 在所有配置中，最佳单个agent的准确率低于POIROT聚合分数
- 差距从14.3pp到21.3pp不等
- 直接量化了自评方法的局限性

### 6.6 模型特异性
- 无单一模型在所有故障场景中占主导
- 较小开源模型可优于较大闭源模型（特定故障类型）
- 表明评估管道设计需考虑故障类型特异性

## 7. 优点

1. **去中心化评估架构**：消除单点故障，将诊断功能内化到系统本身
2. **无需外部专家**：危险空间基于系统结构定义，避免了领域特定评估器的手工设计

## 8. 不足与局限

### 8.1 实验覆盖局限
- TradingAgents仅使用单故障注入（受上下文窗口限制）
- 复合故障场景的准确率较低（最佳28.8%），需进一步研究
- 某些故障类型（如Depressed Bull）在所有模型中准确率均为0%

### 8.2 模型能力依赖
- 协议有效性受限于参与agents的基础推理能力
- 较低能力模型（GPT-oss 20B/120B）在某些场景中表现不稳定
- GPT-oss 120B在多故障场景中甚至略逊于基线

### 8.3 应用限制
- 尚未在真实安全关键部署中验证，仅基于模拟环境
- 未与人类专家判断进行对照验证
- 协议需要系统内agents参与，可能不适用于完全黑盒系统

### 8.4 潜在偏差风险
- 危险空间定义可能遗漏非预期故障源
- 同伴质询过程中的交流可能引入新的偏见
- 权重机制中的β参数固定，可能不是所有场景的最优选择

### 8.5 可解释性挑战
- 多agent交互的累积效应难以追踪
- 投票聚合的合理性依赖于危险空间定义的准确性
- 复杂场景下的归因解释可能不够直观

### 8.6 泛化性验证不足
- 仅在两个安全关键领域（医疗/金融）验证
- CORTEX基于特定硬件平台（Discover2Walk外骨骼）
- 缺乏跨领域泛化的系统评估
