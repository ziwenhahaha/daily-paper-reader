## 摘要
现代深度学习工作负载日益表现出动态的、由元数据驱动的执行特征，即由运行时生成的信息来决定内存分配与内核启动决策。在基于采样的图神经网络（GNN）训练中，这种行为使得CPU成为关键路径，引入了持续的主机-设备协调开销以及频繁的GPU-CPU同步；当GPU计算量较小时，这些开销将主导端到端的运行时间。现有的方法（包括CUDA Graphs和GPU动态并行机制）未能解决这一问题，因为由元数据驱动的控制循环仍由主机主导，且执行结构在不同迭代间存在差异。我们提出了ZEROGNN系统，该系统将主机从由元数据驱动的控制循环中移除，并支持在动态行为下实现完全常驻GPU的执行。ZEROGNN将运行时元数据保留在设备端，在固定的启动结构内协调动态执行，并提供保守但紧凑的执行包络，从而恢复CUDA Graph的可重放性。在基于采样的GNN工作负载上的实验表明，ZEROGNN实现了高达5.28倍的端到端加速，接近100%的GPU执行占比，以及与理想元数据已知分配相当的内存效率；同时，通过消除主机端瓶颈，实现了强大的多GPU扩展能力。

## Abstract
Modern deep learning workloads increasingly exhibit dynamic, metadata-driven execution, where runtime-generated information determines memory provisioning and kernel launch decisions. In sampling-based graph neural network (GNN) training, this behavior places the CPU on the critical path, introducing persistent host-device orchestration overhead and frequent GPU-CPU synchronization, which dominate end-to-end runtime when GPU computation is small. Existing approaches, including CUDA Graphs and GPU dynamic parallelism, fail to address this problem because the metadata-driven control loop remains host-mediated, and execution structure varies across iterations. We present ZEROGNN, a system that removes the host from the metadata-driven control loop and enables fully GPU-resident execution under dynamic behavior. ZEROGNN keeps runtime metadata on-device, mediates dynamic execution within a fixed launch structure, and provisions a conservative yet tight execution envelope to restore CUDA Graph replayability. Experiments on sampling-based GNN workloads show that ZEROGNN achieves up to 5.28 x end-to-end speedup, near 100% GPU execution fraction, and memory efficiency comparable to ideal metadata-informed allocation, while enabling strong multi-GPU scaling by eliminating host-side bottlenecks.

### 1. 论文的核心问题与整体含义（研究动机和背景）
* **研究背景**：现代基于采样的图神经网络（GNN）训练（如GraphSAGE）采用小批量训练，每次迭代会动态采样生成不规则的子图。这种动态性导致系统表现出“由元数据驱动的执行”特征。
* **核心问题**：在训练过程中，GPU生成的运行时元数据（如采样的顶点/边数量）必须频繁地传回CPU，以决定后续的内存分配和内核启动参数。这导致CPU处于关键路径上，引发了严重的**主机-设备协调开销（HDOO）**和**主机介导的依赖屏障（HMDB）**。
* **现有方案的局限**：当GPU计算量较小时，这种CPU-GPU同步和调度开销将主导端到端的运行时间。而现有的优化技术（如CUDA Graphs和GPU动态并行）由于依赖静态可预测的执行流或存在巨大的嵌套启动开销，无法解决这种跨迭代动态变化的宿主机开销。

### 2. 论文提出的方法论：核心思想、关键技术细节
* **核心思想**：提出 **ZEROGNN** 系统，将主机（CPU）从元数据驱动的控制循环中彻底移除。通过将元数据保留在GPU端，并在固定的启动结构内协调动态执行，实现完全由GPU驻留的动态执行，从而恢复CUDA Graph的捕获与重放能力。
* **关键技术细节**：
  * **设备端驻留元数据缓冲区 (DRMB)**：通过内存间接引用，将采样产生的元数据（如顶点数 $|V|$ 和边数 $|E|$）保留在GPU内存中。下游内核直接通过设备指针解引用获取这些数据，彻底消除了GPU到CPU的元数据往返传输。
  * **设备端启动协调 (DLM)**：解决内核启动依赖问题。主机端使用保守的静态上限发起内核（保持对CUDA Graph友好），GPU内核在运行时从DRMB读取真实的元数据大小，多余的线程块直接提前退出，实现动态特化而无需CPU干预。
  * **无元数据调度器 (MFD)**：解决动态内存分配问题。利用统计学原理（引理4.1：去重后的采样节点规模服从泊松-二项分布，高度集中在均值附近），系统在初始化时一次性分配一个“保守但紧凑的执行包络”。这避免了每次迭代向CPU请求动态分配内存。对于极小概率的溢出情况，系统会回退执行一个预缓存的安全图以保证正确性。
  * **端到端CUDA Graph重放**：结合上述技术，ZEROGNN满足了CUDA Graph所需的稳定内存地址和固定控制流前提。通过“预热-捕获-重放”机制，每次迭代仅需CPU发起一次重放指令，其余计算全在GPU异步完成。
