## Abstract
Orchestrating Large Language Models into Multi-Agent Systems (LLM-MAS) has unlocked remarkable reasoning capabilities, yet emergent failures and hallucinations that resist characterisation block their deployment in safety-critical domains -- a gap made legally untenable by emerging AI regulation. Existing evaluation paradigms share a common flaw: centralised judgment creates single points of failure and demands domain-specific expertise. Here we present POIROT, a protocol that repurposes a system's own agents as its diagnostic layer, leveraging the epistemic diversity already present in the architecture. Across evaluated settings, POIROT outperforms single-LLM evaluator baselines, with gains that scale with problem complexity (OR = 1.60, $p = 0.008$), agent count, and fault dimensionality, persisting under compound fault conditions. These results demonstrate that safety oversight need not be externalised: the agents executing a role carry sufficient collective intelligence to audit it. We release POIROT as an open-source library alongside BLAME, a benchmark for fault attribution in safety-critical multi-agent systems.

## 1. 论文核心问题与整体含义（研究动机和背景）

- **问题领域**：大型语言模型多智能体系统（LLM-MAS）在安全关键领域（如医疗、金融）的部署面临瓶颈，因为其输出易产生幻觉、突发性失败和泛化崩溃，难以被检测和归因。

- **监管压力**：欧盟AI法案（Regulation 2024/1689）对高风险AI系统提出了明确的风险管理、追溯记录和人类监督要求，但现有评估范式难以满足这些合规需求。

- **现有方法的共同缺陷**：所有现有评估范式（人工评估、LLM-as-a-judge、单模型自评估）都依赖**中心化判断**，引入单点故障、需领域专家知识、且在长上下文下存在记忆衰减和判断幻觉风险。

- **研究空白**：缺乏一种**去中心化、可扩展、无需外部专家**的故障检测机制，能够系统性地识别LLM-MAS中的失败模式和安全隐患。

- **研究目标**：提出POIROT（Peer-Opinion Interrogation and Review Outcome Tracking）协议，利用系统自身的agents作为诊断层，将故障检测本身转化为一种**内化的分布式诊断过程**。

## 2. 论文方法论

### 2.1 核心思想

POIROT将多智能体系统表示为元组 **S = (A, C, H, U)**，其中：
- **A**：agents集合
- **C**：软件组件
- **H**：硬件元素
- **U**：人类参与者

核心洞察：**不同agents间的认知多样性本身即是一种诊断资源**，而非需要消除的噪声。通过让agents相互审问、对比证据，可以分布式地定位故障源。

### 2.2 五阶段协议流程

| 阶段 | 名称 | 关键操作 |
|------|------|----------|
| **Phase I** | 危险空间定义 | 从系统结构描述构建N维向量空间，轴对应结构性风险源（agents、软件、硬件、人类） |
| **Phase II** | 自我评估 | 每个agent独立回顾自己的行为轨迹，生成定性报告识别异常或确认无异常 |
| **Phase III** | 同行审问 | Agents交换报告，通过结构化对话收集上下文信息、验证假设，但不公开投票 |
| **Phase IV** | 私密投票 | 每个agent发射二进制归因向量 **vi ∈ {0,1}^N**，私密性防止策略性对齐 |
| **Phase V** | 危险感知聚合 | 通过距离加权共识机制聚合投票：距离近的agent影响权重更大 |

### 2.3 危险感知加权聚合公式

1. **Hamming距离**：衡量agent位置与其投票的结构差异
   > d_H(p_i, v_i) = Σ |(p_i)_k ≠ (v_i)_k|

2. **权重函数**：
   > w_i = β + ½ × (1 − d_H(p_i, v_i) / N)

   其中β ∈ [0, 0.5]是基线项，固定β = 0.25，使权重范围为[0.25, 0.75]。

3. **最终归因得分**：
   > S_j = Σ w_i × (v_i)_j

   得分S_j最高的维度即为POIROT的最终故障归因。

### 2.4 技术实现

- 基于**LangGraph**构建，每个agent作为独立节点运行
- 通过`talk_to_{target}`工具实现all-to-all通信
- 每个agent限制最大8次LLM调用，超限强制投票
- 消息根据接收者重新类型化为AIMessage/HumanMessage以保持轮次结构

## 3. 实验设计

### 3.1 使用的基准测试

#### Who&When Benchmark
- **来源**：Zhang et al. (2025) 提出的外部基准
- **规模**：122个算法生成系统，包含2-4个专业agents
- **特点**：每个案例有单一注入故障，提供受控的归因真值
- **用于**：评估跨异构多智能体配置的故障归因能力

#### BLAME Benchmark（自研）
作者专门开发的**BLAME**（Benchmark for Localising Agent Malfunctions Effectively）包含两个安全关键场景：

##### CORTEX（医疗康复）
- **场景**：儿童脑瘫康复外骨骼治疗的多智能体临床决策系统
- **Agents**：诊断医生、康复主任、理疗师（3个）
- **危险空间**：7维（医生、康复主任、理疗师、外骨骼、患者、家长、通讯通道）
- **故障注入**：5种单故障向量 + 多种复合故障组合（包括五故障同时注入）
- **难度**：2^7 = 128种可能定位，随机正确率<0.8%

##### TradingAgents（金融交易）
- **场景**：专业交易公司的多智能体投资流水线
- **结构**：12个agents，分为四层（分析师团队、研究团队、交易员、风险管理团队）
- **危险空间**：15维
- **故障注入**：6种单故障向量（如市场分析师技术分析完全反向、新闻分析师捏造丑闻等）
- **难度**：2^15 = 32,768种可能定位，上下文窗口饱和

### 3.2 对比基线

- **单LLM评估器**：单一外部模型，在all-at-once模式下接收完整上下文（系统提示、消息日志、用户输入）
- **测试模型**：4种 backbone LLMs
  - 闭源API：DeepSeek V3.2 Reasoner、Gemini 2.5 Pro
  - 开源本地：GPT-oss 120B、GPT-oss 20B（使用vLLM 0.16.0）

### 3.3 评估指标

- **精确匹配准确率**：归因向量与真实故障向量完全一致的比例
- **分层分析**：按问题复杂度（四分位数Q1-Q4）、agent数量、故障维度分组

## 7. 优点

### 7.1 方法亮点

1. **去中心化架构**：彻底消除中心化评估的单点故障风险，将安全监督内化为系统固有能力
2. **危险空间的结构化表示**：将故障归因限制在系统架构维度，避免抽象语义判断，增强可解释性
3. **私密投票机制**：防止策略性对齐，保留认知独立性
4. **距离加权聚合**：根据结构邻近性分配影响权重，相比均匀投票更符合故障传播逻辑
5. **无需外部专家**：消除对领域特定评估器的依赖，提高泛化性

### 7.2 实验设计亮点

1. **跨能力谱系测试**：同时验证闭源API模型和开源本地模型，结果更具说服力
2. **几何可视化**：图5的二维投影直观展示归因质量（共识点与真值距离）和智能体分歧程度
3. **分层统计检验**：使用逻辑回归分析交互效应，提供了比简单准确率比较更严格的统计论证
4. **BLAME基准开源**：模块化设计允许社区扩展，促进可复现性和后续研究

### 7.3 工程贡献

1. **完整开源实现**：pip install poirot-framework，与LangChain/LangGraph生态兼容

## 8. 不足与局限

### 8.1 实验层面

1. **复合故障准确率过低**：CORTEX多故障条件下最高仅28.8%，在真实安全关键场景中可能仍不够可靠
2. **Depressed Bull场景系统性失败**：所有模型在TradingAgents的"看涨分析师反向角色"场景均得0%，暗示某些故障类型存在系统性识别困难
3. **缺乏Human Expert对照**：声称POIROT是替代人工评估的方案，但未提供与人类专家的直接比较数据
4. **无跨领域泛化测试**：仅在医疗和金融两个领域验证，对于机器人控制、自动驾驶等场景的适用性未知
5. **闭源模型API调用不确定性**：DeepSeek和Gemini的API可能存在版本漂移，影响可复现性

### 8.2 方法论局限

1. **基线选择可能不公平**：单LLM all-at-once模式是相对弱的基线，未与few-shot prompting、chain-of-thought等增强方法对比
2. **β参数固定为0.25**：未充分消融此超参数对不同场景的影响
3. **通信开销**：全互联通信在agent数量增加时可能造成O(n²)开销，限制了向百agent级系统的扩展
4. **恶意行为未考虑**：假设agents是善意的，未测试对抗性agents或collusion攻击场景
5. **上下文长度限制**：每个agent的上下文窗口仍受限，对于超长session可能失效

### 8.3 应用限制

1. **实时性开销**：五阶段协议需额外LLM调用，对于延迟敏感场景可能不适用
2. **安全关键场景的监管接受度**：虽然符合EU AI Act的设计思路，但监管机构是否认可"agents自审计"尚未可知
3. **闭源模型依赖**：最强结果来自闭源API，存在供应商锁定风险
