## 摘要
现有的多智能体大语言模型（LLM）编排方法——从暴力集成到学习型路由器——根据任务和模型特征选择模型与拓扑结构。然而，这些方法并未考虑服务基础设施的运行时状态。在共享GPU集群上并发负载的情况下，这种对基础设施的盲目性导致系统性资源利用不足：首选模型积压大量请求队列，而能力相当的其他模型却闲置。在多智能体流水线中，每个查询都会触发多个顺序的模型调用，这些延迟会在每个下游步骤中叠加放大。弥合这一差距颇具挑战性，因为相关的基础设施信号（队列深度、KV缓存压力、延迟）具有动态性和噪声，且必须驱动三类不同的决策：规划、每步路由和调度。我们提出了INFRAMIND，一个使整个多智能体技术栈具备基础设施感知能力的框架。基础设施感知的规划器依据实时系统负载和剩余预算来调节拓扑与角色选择，在拥塞时倾向于更简单的图结构，在负载低时倾向于更丰富的图结构。随后，基础设施感知的执行器在每个智能体步骤观察各模型的队列深度、缓存利用率和响应延迟，以决定调用哪个模型以及进行多深的推理；预算感知的调度器进一步对每个模型的队列进行重新排序，使紧急请求优先得到服务。该系统被建模为层次约束马尔可夫决策过程（MDP），并通过端到端强化学习求解，自动学习在质量与延迟之间取得平衡。在五个基准测试中，INFRAMIND在低负载下实现了最高提升7.6个百分点（相比先前基线）准确率，同时延迟降低至原来的七分之一；在高负载下保持高达99.9%的SLO达标率，而所有基线方法均降至50%以下。

## Abstract
Existing multi-agent LLM orchestration methods, ranging from brute-force ensembles to learned routers, select models and topologies based on task and model features. However, these methods do not consider the runtime state of the serving infrastructure. On shared GPU clusters under concurrent load, this infrastructure blindness causes systematic resource underutilization: preferred models accumulate deep request queues while equally capable alternatives sit idle. In multi-agent pipelines, where each query triggers multiple sequential model calls, these delays then compound across every downstream step. Closing this gap is challenging because the relevant infrastructure signals (queue depths, KV-cache pressure, latencies) are dynamic and noisy, and they must drive three different decisions: planning, per-step routing, and scheduling. We introduce INFRAMIND, a framework that makes the entire multi-agent stack infrastructure-aware. An infra-aware planner conditions topology and role selection on real-time system load and remaining budget, biasing toward simpler graphs under congestion and richer ones at low load. An infra-aware executor then observes per-model queue depths, cache utilization, and response latencies at each agent step to decide which model to call and how deeply to reason; a budget-aware scheduler further reorders each model's queue so that urgent requests are served first. Cast as a hierarchical constrained MDP and solved end-to-end via reinforcement learning, the system learns to balance quality against latency automatically. Across five benchmarks, INFRAMIND delivers up to +7.6 pp accuracy over the prior baseline at low load with up to 7x lower latency, and sustains up to 99.9% SLO compliance under high load where every baseline drops below 50%.

## 2. 论文提出的方法论

### 核心思想

将基础设施状态（队列深度、KV缓存压力、延迟）作为**第一公民信号**，嵌入多智能体编排的每个决策层，实现"感知-决策-调度"的闭环。

### 系统架构（三个组件）

#### 2.1 基础设施感知规划器（Infra-Aware Planner）

- **功能**：在查询到达时，选择拓扑（Chain/CoT/Debate/FullConnected）、智能体数量、角色分配
- **输入**：查询语义 + 系统摘要 $z_0 = [b_0, d_{queue}, d_{e2e}, d_{kv}]$
- **技术**：基于MasRouter的级联控制器，注入FiLM调制：
  - $\tilde{e}_q = \gamma(z_0) \odot e_q + \beta(z_0)$
- **策略**：拥塞时倾向简单链式拓扑，空闲时启用丰富辩论结构

#### 2.2 基础设施感知执行器（Infra-Aware Executor）

- **功能**：每步决定调用哪个模型 + 采用何种推理深度
- **状态**：$s_k = [e_q, e_{rk}, b_k, d_{queue}, d_{e2e}, d_{kv}]$
- **架构**：双路径设计
  - **语义路径**：处理查询+角色嵌入 → $h_{sem} \in \mathbb{R}^{128}$
  - **资源路径**：预算头 + 系统头 → $h_{res} \in \mathbb{R}^{64}$
  - **决策**：$h = \text{ReLU}(W_{merge}[h_{sem} \| h_{res}])$ → softmax输出 $N \times |S|$ 联合动作
- **推理策略**（3级）：
  - Flash：直接回答
  - Concise：2-3步CoT
  - DeepThink：深度推理+验证

#### 2.3 预算感知调度器（Budget-Aware EDF Scheduler）

- **功能**：防止 tight-budget 请求被 relaxed 请求阻塞
- **机制**：为每个请求附加截止时间 $t_{arrive} + \beta$，每个模型队列按 **Earliest-Deadline-First** 排序
- **分层**：跨模型负载均衡由执行器负责，单模型内优先级由EDF保证

### 优化框架

- **建模**：层次约束MDP（Constrained MDP）
- **目标**：$\pi^* = \arg\max_\pi \mathbb{E}[P_k R_k]$，约束为 $\mathbb{E}[P_k C_k] \leq 1$（$C_k = \ell_k / \beta$）
- **训练**：单一Lagrange乘子 $\lambda$ 将约束转为奖励：$L(\pi, \lambda) = \mathbb{E}[P_k R_k - \lambda \cdot C_k] + \lambda$
- **算法**：
  - 执行器：PPO clipped surrogate（per-step）
  - 规划器：REINFORCE + baseline normalization（episode-level）
  - 双重更新：$\lambda \leftarrow \text{clip}(\lambda + \eta_\lambda(\bar{C} - 1), 0, \lambda_{max})$

## 6. 论文的主要结论与发现

### 核心发现

| 场景 | INFRAMIND表现 | 基线表现 |
|------|---------------|----------|
| **低负载准确率** | 最高+7.6pp（MATH: 82.0%） | MoA: 74.4%，MasRouter: 67.4% |
| **低负载延迟** | 最高14×加速（HumanEval: 5s vs 70s） | MASRouter: 70s |
| **高负载SLO合规** | 最高99.9%（HumanEval） | 所有基线 <50% |
| **高负载延迟** | 远低于500s阈值 | MoA/GPTSwarm常超1000s |

### 关键结论

1. **基础设施感知是必要的**：在共享GPU集群上，忽视运行时状态的路由会产生系统性资源利用不足
2. **三组件互补**：基础设施感知路由、预算感知调度、自适应推理深度各自解决不同失效模式
3. **质量-延迟权衡自动学习**：端到端RL自动发现跨预算层级的最优权衡，无需手工规则
4. **扩展到黑盒可行**：客户端侧延迟/RPM代理即可支持纯API或混合部署

### 应用层面

- **模型池静态性**：假设固定模型集，不支持运行中添加/移除模型（Autoscaling场景）
- **冷启动问题**：新模型加入时缺乏训练数据，基础设施感知策略可能失效
- **安全考量**：未讨论对抗性请求注入、预算耗尽攻击等恶意场景
