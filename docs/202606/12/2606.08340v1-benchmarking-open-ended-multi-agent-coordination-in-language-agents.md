## Abstract
As language models are increasingly deployed as autonomous agents, they must coordinate with others over long horizons in open-ended interactive tasks. Yet existing evaluations rarely test these demands together, instead emphasising single-agent tasks, short interactions, or highly structured multi-agent settings. We introduce $alem$, a JAX-based benchmark for open-ended multi-agent coordination built on Craftax-like dynamics. Alem embeds procedurally generated coordination tasks, soft specialisation, communication, and controllable coordination difficulty into a long-horizon survival world with exploration, crafting, trading, and combat. We evaluate $13$ modern LLMs zero-shot within homogeneous teams, with trained MARL agents as reference points. Current LLM agents remain far from solving alem, averaging only ~6% normalised return, but their failures are not uniform. On the hardest coordination setting, zero-shot Gemini-3.1-Pro-High approaches MARL agents trained for one billion steps, while GPT-5.4-High achieves strong base-task reward but much lower coordination reward. This contrast shows that individual task competence does not imply coordination competence. Ablations show that communication is the largest contributor to coordination, while memory and reasoning help when used to maintain multi-step plans. Overall, our results identify coordination as a distinct bottleneck for frontier LLM agents, separate from single-agent capabilities. Alem makes this bottleneck measurable and provides a controlled testbed for developing agents that communicate, allocate roles, and execute shared plans. Code is available at https://github.com/alem-world/alem-env.

### 1. 核心问题与整体含义（研究动机和背景）

随着大语言模型（LLM）被部署为自主智能体，它们必须与他人进行长期协作以完成开放式任务。然而，现有评估存在明显短板：

- 大多数基准聚焦于**单智能体**场景，忽略了多智能体协作的核心挑战；
- 多智能体基准（如 Collab-Overcooked、MultiAgentBench）通常只覆盖**短期交互**，缺乏长期 horizon 的测试；
- 现有环境缺乏**显式、可控且程序化生成**的协调难度调节机制；
- 社区尚不清楚当前 LLM 智能体在真实长 horizon、开放世界中的协调能力究竟如何。

本论文指出，**协调（coordination）是独立于单智能体能力的独立瓶颈**，而非水到渠成的副产物。

### 2. 论文提出的方法论

#### 核心思想

提出 **ALEM**（Alem 为阿姆哈拉语中"world"之意），一个基于 JAX 编写、构建在 Craftax 类动力学之上的开放式多智能体协调基准。ALEM 将协调难度作为**显式可调参数**，在保持底层世界结构不变的前提下，调节协调执行的严苛程度。

#### 关键技术细节

- **协调类型谱（Coordination Spectrum）**：任务按耦合动作间的时间间隔 Δt = |t − t′| 划分为三类：
  - **同步（Synchronous）**：Δt ≈ 0，多个智能体必须在同一时间步对同一目标执行动作；
  - **交接（Handover）**：小 Δt，一个智能体启动任务，另一个须在有限时间窗口内完成；
  - **长程依赖（Long-range）**：大 Δt，资源采集与后续使用之间相隔多个时间步。
- **程序化协调生成**：每个 episode 中，挖掘、建筑、战斗、制作等活动独立采样协调类型（同步或交接）和所需智能体数量，防止智能体记忆固定模式。
- **软专业化（Soft Specialisation）**：引入角色（Forager、Miner、Warrior），专业智能体执行角色限制动作成功率 100%，非专业智能体成功率为 1 − α，不设硬性角色门限。
- **协调难度控制**：单一标量参数 α ∈ [0, 1] 控制难度：
  - Easy（α = 0.3）：宽松交接窗口、高非专业成功率；
  - Medium（α = 0.6）；
  - Hard（α = 0.9）：严格交接窗口（仅 3–6 步）、极低非专业效率、更多 N-agent 强制要求。
- **通信通道**：智能体可通过离散广播消息进行交流，支持资源请求、行动协调和意图表达。
- **环境接口**：提供像素观测、符号观测和文本接口三种模式，其中文本接口用于 LLM 智能体评估。

### 3. 实验设计

#### 评估场景

- **9 个程序化生成的关卡**，涵盖地上世界、矿井、火焰领域等；
- 活动类别包括：挖掘、建筑、战斗、制作；
- 三种协调难度（Easy / Medium / Hard）；
- 93 项成就（achievements），其中 27 项为新增的协调成就。

#### 对比方法

- **13 个现代 LLM**：包括 Gemini-3.1-Pro-High、GPT-5.4-High、Gemma-4 系列（E4B/26B/31B）、Qwen-3.5/3.6 系列（9B/27B/35B/122B）、Llama-3.1/3.3 系列；
- **4 个 MARL 基线**：IPPO、MAPPO、PQN-VDN、HyperMARL-IPPO，分别在 100M 和 1B 环境步数下训练；
- 异构团队实验：同族（Gemini 家族）与跨族（Gemini + GPT + Gemma）混合团队。

#### 指标

- 归一化 episode return，分解为 **Base%（基础奖励）**、**Coord.%（协调奖励）**、**Total%（总奖励）**，三者独立归一化；
- 使用 rliable 库计算 95% 分层自助置信区间。

### 6. 论文的主要结论与发现

1. **当前 LLM 智能体远未解决 ALEM**：所有模型平均仅获得约 6% 的归一化总奖励；
2. **基础任务能力 ≠ 协调能力**：GPT-5.4-High 基础奖励名列前茅，但协调奖励远低于更小的 Gemma-4-26B 模型；
3. **Gemini-3.1-Pro-High 表现最优**：在 Hard 难度下协调奖励达 17.5%，与训练 1B 步的 MARL 基线（17.6%）相当；
4. **通信是协调的最大贡献因素**：移除通信后，Gemini 协调奖励从 17.5% 跌至 5.3%，Gemma 从 8.8% 跌至 3.8%；
5. **记忆的作用取决于使用方式**：Gemini 用 scratchpad 做前瞻性规划（多行、带时间戳的行动序列），效果显著；Gemma 仅用于状态摘要，效果有限；
6. **推理同时影响基础和协调表现**：降低推理能力后两项指标均下降；
7. **异构团队性能接近同构团队均值**：混合团队不会自动匹配最强成员的性能，也不会崩溃至最弱成员；
8. **同步-困难任务最具挑战性**：即使最强 LLM 在该类别覆盖也仅约 18%；
9. **建筑类长程协调几乎无人完成**：稀疏、长 horizon 的协调难以通过试错发现；
10. **规模本身不能解释协调表现**：同族模型中性能与参数量不呈单调关系。

### 7. 优点

- **可控的协调难度参数**：单一 α 参数实现了协调难度与底层世界复杂度的解耦，是极为优雅的设计；
- **程序化协调生成**：防止记忆化，强制智能体从观测中推断协调需求，而非依赖固定模式；
- **多接口支持**：像素 / 符号 / 文本三种接口，MARL 和 LLM 智能体可公平对比；
- **信息论诊断验证**：通过 HAR、PIF、AA、DAI 等指标证明 ALEM 确实引入了真实的智能体间依赖，而非简单增加单智能体难度；
- **消融实验设计严谨**：对通信、记忆、推理三维度进行独立消融，结合定性分析（词汇检测、行为模式），实验与解读互相印证；
- **高效 JAX 实现**：支持单 GPU 两天内完成 1B 步训练，降低了研究门槛；
- **公开代码**：提供完整实现和基线，促进社区复现和跟进。

### 8. 不足与局限

- **闭源模型评估规模有限**：Gemini 和 GPT 因 API 成本仅评估 10 seeds，置信区间较宽；
- **仅评估文本 LLM**：未覆盖视觉-语言模型（VLM）和 Claude 系列模型，未来工作留待扩展；
- **同构团队为主**：异构团队实验仅两组（各一种配置），尚不足以得出一般性结论；
- **缺乏跨 episode 记忆**：当前评估仅覆盖单 episode，尚未测试跨 episode 的终身学习能力；
- **动作空间受限**：60 维离散动作空间可能无法覆盖所有真实世界协调场景；
- **协调成功可能存在天花板**：即使 MARL 训练 3B 步后仍未饱和，说明该基准极具挑战性，但也意味着可能需要更强大的算法才能取得实质进展；
- **定性分析的局限性**：词汇检测器为正则表达式近似，可能存在误分类，但已通过多重指标相互印证。
