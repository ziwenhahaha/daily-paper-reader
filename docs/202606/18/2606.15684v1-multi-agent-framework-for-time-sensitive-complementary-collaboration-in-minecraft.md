## 摘要
我们提出了 TickingCollabBench，一个基于 Minecraft 的多智能体基准测试，针对一类新颖的时间敏感互补协作任务。我们的基准测试反映了真实世界协作的四个核心特征：智能体异构性、强制性协作、动态环境以及伴随失败风险的严格实时约束。为此，我们开发了 TickingCollab 框架，它支持生成多样化的动态环境，并将 Minecraft 的底层 API 抽象为声明式 YAML 任务规范，以组合这些事件。在此基础上，我们设计了一个可行性感知的自动化基准生成流程，其中大语言模型（LLM）起草结构多样的任务配置，随后可行性验证器使用近似约束过滤无效配置。评估表明，由于部分可观测性和智能体异构性带来的协调困难以及大延迟，LLM 在动态环境中经常失败，并显著低于全局知识神谕的表现。

## Abstract
We present TickingCollabBench, a Minecraft-based multi-agent benchmark for a novel class of time-sensitive complementary collaboration tasks. Our benchmark reflects four core characteristics of real-world collaboration: agent heterogeneity, mandatory collaboration, dynamic environments, and strict real-time constraints with failure risks. To enable this, we develop the TickingCollab framework, which supports the generation of diverse dynamic environments and abstracts Minecraft's primitive APIs to enable declarative YAML task specifications for composing these events. Building on this, we design a feasibility-aware automated benchmark generation pipeline, where an LLM drafts structurally diverse task configurations and feasibility verifier filters out invalid ones using approximate constraints. Evaluations demonstrate that lang latency and inherent difficulty of coordinating under partial observability and agent heterogeneity cause LLMs to frequently fail under dynamic environments and fall significantly short of a global-knowledge oracle.

## 2. 论文提出的方法论

### 核心思想
设计一个完整的框架和基准测试，通过声明式配置自动生成多样化的动态协作任务，并量化评估指标，从而系统性地研究 LLM 在时间敏感多智能体协作中的能力与局限。

### 关键技术细节

#### 2.1 TickingCollab 框架三大功能模块

**（1）动态环境管理器**
- 将 Minecraft 复杂底层 API（如 `/effect`、`/attribute`）抽象为声明式 YAML 配置
- 支持注入复杂运行时动态事件（如熔岩波、方块生成/消失、实体生成）
- 开发者无需编写自定义服务器插件，降低技术门槛

**（2）可行性感知的自动化基准生成流程**
- **第一步**：用户提供任务元数据模板（定义参数空间和约束）
- **第二步**：LLM（GPT-5.1）在参数空间中起草多样化任务配置
- **第三步**：可行性验证器使用近似约束过滤无效配置
  - 设置安全裕度参数（α, β, γ）控制任务难度
  - 验证示例：
    - 必要工具检查：`max_n(τE_n) ≥ τ_req(b), ∀b ∈ B_target`
    - 足够生存方块数：`N_required ≥ (H + 1) · N_agent`
    - 足够准备时间：`T_crisis ≥ α · (T_gather + T_construct)`

**（3）综合评估引擎**
- 支持**同步模式**（固定时间步长，暂停模拟进行 LLM 推理）和**异步模式**（实时执行，推理延迟直接影响任务成功）
- 并行模拟执行与细粒度系统开销日志记录

#### 2.2 协作难度量化指标

| 指标 | 定义 | 公式/说明 |
|------|------|-----------|
| **智能体异构性 (H)** | 智能体间属性差异 | 平均成对归一化距离，包含连续值（HP、速度）和集合值（物品清单）的距离度量 |
| **协作必要性 (N)** | 任务工作量与单智能体最大容量的比值 | `N = min_{a∈A}(Σ_k Workload_k / Throughput_{a,k}) / T_max`，N > 1 保证协作必须 |
| **环境动态性 (D)** | 单位时间内环境状态变化次数 | `D = (总环境状态变化数) / T_max` |
| **失败时间 (τ)** | 不可逆失败发生前的窗口时间 | 危机到达时间/方块最小寿命/敌人击败智能体所需时间 |

#### 2.3 TickingCollabAgent 基线智能体

- **集中式协调**（TickingCollabAgent-centralized）：
  - 单一主智能体聚合所有同伴观察，执行联合规划并分发动作
  - 优势：最大化联合规划质量
  - 劣势：同步瓶颈，快智能体需等待慢智能体

- **分布式协调**（TickingCollabAgent-distributed）：
  - 智能体独立并行规划，通过 propose→wait→act 协商协议解决共享决策
  - 放宽了 MineCollab 的 1 对 1 消息限制，支持选择性多智能体广播

#### 2.4 三类核心任务

| 任务 | 描述 | 关键挑战 |
|------|------|----------|
| **任务#1: 为危机做准备** | 识别即将到来的危机（熔岩洪水等），收集合适材料建造生存避难所 | 角色分配（侦察者 vs 采集者）、材料选择（耐火vs可燃） |
| **任务#2: 挖掘消失的方块** | 在多种随机生成且有限寿命的方块消失前完成配额挖掘 | 动态能力感知分配，考虑旅行时间和挖掘效率 |
| **任务#3: 突袭Boss** | 击败动态生成不同属性小弟的 Boss 怪物 | 基于类型优势和生存能力的联合目标分配 |

## 6. 论文主要结论与发现

### 核心发现
1. **LLM 推理延迟是实时协作的关键瓶颈**：
   - 异步模式下平均 20 秒 API 延迟频繁超过基准测试的失败时间限制
   - 导致大多数任务在异步模式下失败

2. **集中式协调优于分布式协调**（同步模式下）：
   - 集中式避免了分布式中智能体间通信和推理开销导致的规划时间过长
   - 但集中式存在同步瓶颈（主智能体等待所有同伴报告后重新规划）

3. **即使在同步模式下，L LM 仍显著低于 Oracle**：
   - Oracle 拥有全局地面真值知识和人工设计调度规则
   - 表明部分可观测性下的多智能体规划具有固有难度

4. **团队规模影响因任务而异**：
   - "突袭Boss"和"挖掘方块"：增加智能体数量→更高吞吐量→更高成功率
   - "为危机做准备"：增加智能体→增加所需生存方块和避难所规模→成功率可能下降

5. **系统开销随团队规模快速增长**：
   - 消息量、推理次数、Token 使用量、壁钟时间均随智能体数量增加而飙升
   - 常逼近 40 分钟模拟超时限制

### 量化结果（GPT-5.1 为例）

| 任务 | 集中-同步 | 集中-异步 | 分布-同步 | 分布-异步 | Oracle |

## 7. 优点

   - LLM + 可行性验证器的组合实现了大规模多样化任务配置的自动生成
   - 避免了人工设计的巨大参数空间探索开销

3. **评估框架灵活性**：
   - 同步/异步双模式设计，分别隔离评估规划准确性和延迟影响

4. **数学定义严谨**：
   - 四个协作难度指标有清晰数学定义，便于量化比较
   - 可行性约束使用安全裕度（α, β, γ）平衡任务难度和可解性

5. **代码与框架可扩展性**：
   - 声明式 YAML 接口和模块化运行时设计便于开发者定制新任务和智能体

## 8. 不足与局限

1. **感知模态单一**：
   - 当前仅使用距离受限的结构化语义传感器
   - 未覆盖第一人称视角视频、音频等多模态输入
   - 限制了 VLM（视觉-语言模型）智能体的评估

2. **智能体-环境动态交互不足**：
   - 框架支持复杂的智能体条件交互（如基于物品的移动减速）
   - 但基准测试任务主要关注环境对智能体的影响，未充分探索智能体动作驱动的级联环境变化

3. **评估 LLM 数量有限**：
   - 仅测试 GPT-5.1 和 DeepSeek-R1 两种模型
   - 缺乏对其他主流 LLM（如 Claude、Llama 系列）的评估

4. **Oracle 定义可能存在优化空间**：
   - Oracle 使用人工设计启发式算法（非学习方法），可能并非最优调度
   - 不同任务类型的 Oracle 设计不一致（任务#1 依赖元数据，任务#2/3 依赖实时事件）

5. **部分可观测性边界未充分探索**：
   - 智能体的感知范围固定为预定义值
   - 未系统研究感知范围对协作难度的影响

6. **实验可复现性挑战**：
   - 依赖专有 LLM API（GPT-5.1），成本和可用性可能限制复现
   - 未开源完整代码库（框架部分可见）

7. **现实迁移性存疑**：
   - Minecraft 环境与真实机器人/物联网协作场景仍有差距
   - 简化的物品系统和交互机制可能无法完全捕捉现实复杂性
