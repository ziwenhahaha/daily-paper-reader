## Abstract
In large-scale enterprise settings, centralized multi-agent systems (MAS) are increasingly adopted, in which a coordinator delegates user requests to lightweight, domain-specialized sub-agents. While this architecture improves modularity, scalability, and cost efficiency, its reliability depends not only on accurate routing but also on sub-agents' ability to calibrate their responses to capability constraints. In particular, sub-agents built on smaller fine-tuned models often struggle with such calibration, leading them to over-answer ambiguous, underspecified, misrouted, or unsupported requests and produce hallucinated outputs instead of actionable feedback. To address this challenge, we present EARS (Explanatory Abstention for Reliable Sub-Agent Modeling), a production-oriented framework that reframes sub-agent abstention as an inter-agent communication protocol: a sub-agent does not merely abstain, but exposes an actionable failure state to the coordinator. EARS curates human-agent interaction data using an ensemble of calibrated LLM-as-a-Judge models, producing structured abstention labels and rationales under a taxonomy of sub-agent failure modes. These data are used to fine-tune sub-agents to detect failure conditions and return rationales for coordinator-level clarification, rerouting, or fallback. We evaluate EARS in a large-scale production e-commerce assistant supporting enterprise business intelligence workflows. EARS improves the overall response pass rate from 68.5% to 78.9%, demonstrating that sub-agent-side explanatory abstention improves MAS reliability.

### 1. 核心问题与整体含义（研究动机和背景）

- **背景**：大规模企业场景中，集中式多智能体系统（MAS）被广泛采用，其中协调器（Coordinator）将用户请求路由到轻量级、领域专用的子智能体（Sub-Agent）。
- **核心问题**：子智能体通常基于较小的微调模型构建，这些模型在能力约束校准方面存在困难，容易对以下类型的请求进行**过度回答**：
  - 歧义查询（Ambiguous Query）
  - 输入不足（Insufficient Input）
  - 缺失能力（Missing Capability）
  - 错误路由（Misrouting）
  - 结果产生**幻觉输出**而非可操作的反馈。
- **现有研究的不足**：大多数可靠性工作集中在协调器层（如 RIRS、MasRouter），主要解决路由错误，但**子智能体执行失败**和**子智能体与协调器之间的通信中断**问题未被充分探索。
- **研究动机**：将子智能体的弃权（Abstention）从用户-facing 的拒绝重新定义为**智能体间通信协议**，使子智能体能够向协调器暴露可操作失败状态，从而提升整体 MAS 可靠性。

### 2. 论文提出的方法论

#### 2.1 核心思想

EARS（Explanatory Abstention for Reliable Sub-Agent Modeling）将子智能体的弃权行为定位为**协调信号**，而非最终的用户拒绝。子智能体不仅弃权，还返回结构化的失败类别和理由，供协调器进行澄清、重新路由或回退处理。

#### 2.2 关键技术细节

- **弃权分类法（Abstention Taxonomy）**：定义了四个互斥的失败类别：
  - **Ambiguous Query（歧义查询）**：用户意图无法明确确定。
  - **Insufficient Input（输入不足）**：意图清晰但执行需要额外信息。
  - **Missing Capability（缺失能力）**：子智能体不支持请求的功能。
  - **Misrouting（错误路由）**：请求应由其他子智能体处理。

- **LLM-as-a-Judge 校准流程**：
  - 使用人工标注的种子数据集（600 个可解决查询 + 602 个弃权实例）进行迭代优化。
  - 初始使用通用指令提示，运行判断模型并测量准确率；若低于阈值（如 80%），则使用第二个 LLM 分析失败案例，生成修订建议，形成**人机协同的迭代修订过程**。
  - 该过程缓解了完全自动化提示优化的过拟合风险，并确保更新保持通用性、可解释性和与标注分类法的一致性。

- **数据筛选（Curation）策略**：
  - 使用三个模型家族的判断器：OpenAI（GPT-5）、Claude（Claude-sonnet-4.5）和 Gemini（gemini-2.5-flash）。
  - 采用**保守的集成策略**：只有当所有判断器**unanimously 一致同意**同一弃权类别时，才接受该标签，以避免将可回答的查询错误转换为弃权案例。
  - 该阈值可调，用于在精度-召回率之间进行权衡。

- **训练阶段（Fine-tuning）**：
  - 对子智能体（基于 Qwen3-32B-Instruct）进行全参数监督微调（SFT）。
  - 每个训练实例包含：用户查询、子智能体上下文、期望的子智能体响应（可解决查询的任务特定输出，或带弃权类别和理由的结构化弃权响应）。
  - 更新后的子智能体重新部署到 MAS 中，收集新交互数据，完成**在线数据飞轮（Data Flywheel）**的闭环。

#### 2.3 公式或算法流程（文字说明）

- **离线校准阶段**：
  1. 人工标注种子数据集 → 建立弃权分类法。
  2. 使用种子集校准 LLM-as-a-Judge。
  3. 迭代优化判断指令，直到准确率超过阈值且连续两次修订无改善。

- **在线数据飞轮阶段**：
  1. 应用校准后的判断器标注大规模人机交互数据（采用多模型一致性筛选）。
  2. 使用筛选后的数据对子智能体进行 SFT 微调。
  3. 将更新后的子智能体重新部署，收集新交互数据 → 返回步骤 1。

### 3. 实验设计

#### 3.1 数据集与场景

- **场景**：企业级电商 AI 助手，支持商业智能（BI）工作流，包括客户细分（Customer Segmentation）和业务分析（Business Analytics）两类查询。
- **架构**：协调器基于 Claude-sonnet-4-6，子智能体（BI 子智能体）基于 Qwen3-32B-Instruct，专门生成领域特定语言（DSL）的 SQL 类查询。
- **训练集**：13,853 个用户-智能体交互（7,000 个客户细分查询 + 6,853 个业务分析查询）。
- **评估集**：4,000 个平衡采样的查询。

#### 3.2 Benchmark 与对比方法

- **评估指标**：
  - **Pass Rate**：查询获得正确响应的百分比，包含两个验证步骤：

    - **语义正确性（Semantic）**：LLM-as-a-Judge 评估输出是否满足用户查询。
  - **弃权分类性能**：准确率、精确率、召回率。
  - **成本**：以单个 gemini-2.5-flash 判断器调用为基准的相对 token 使用量。

- **对比方法**：
  - **Baseline**：未使用弃权数据训练的 BI 子智能体。

### 6. 论文的主要结论与发现

- **整体性能提升**：EARS 将 BI 子智能体的**总体 Pass Rate 从 68.5% 提升至 78.9%**（相对提升 15.2%）。
- **细分指标差异**：
  - 客户细分查询收益最大：语法有效性提升 4.9%，**语义正确性提升 33.6%**。
  - 业务分析查询提升较小：语法有效性提升 0.6%，语义正确性几乎持平（-0.1%）。
  - **原因**：客户细分查询中 23.8% 被预测为弃权案例，而业务分析查询弃权案例不足 2%。
- **判断器性能**：
  - gemini-2.5-flash 提供最佳成本-性能比（90.4% 准确率，基准成本）。
  - 4 判断器集成显著提升精确率（+6.3%）和整体准确率（93.1%），但成本增加 20.4 倍。
- **消融实验结论**：
  - 弃权数据持续提升客户细分性能而不损害业务分析性能。
  - 人工标注数据使总体 Pass Rate 提升 10.2%。
  - 混合人工标注和原始交互数据时，EARS 筛选减少了标签冲突，进一步提升性能。
  - 4 判断器集成比任何单一判断器表现更好。
- **人类评估结果**：
  - 在 EARS 产生弃权信号的会话中，人类评估 Pass Rate 为 **67.1%**，显著高于基准系统的 2.4%。
  - 子智能体弃权精确率达 **94.0%**（549/584）。

### 8. 不足与局限

- **实验领域单一**：仅在电商商业智能助手场景中验证，未在其他领域（如金融、教育、医疗）进行验证。不同领域的弃权场景和评估成本可能差异显著。
- **自动化程度有限**：依赖人机协同的 LLM-as-a-Judge 校准流程，可扩展性受限。虽然设计了人工审核环节以减少过拟合风险，但增加了部署成本。
- **协调器优化未涉及**：EARS 仅关注子智能体层面的弃权能力，未探索协调器在接收到弃权信号后的**恢复策略优化**（如 RL 训练的协调器或回退策略）。系统级优化留待未来工作。
- **误判风险**：
  - 消融实验显示，即使是 4 判断器集成也可能引入标签噪声，导致部分指标略有下降（如 Analytics Semantic -0.1%）。
  - 弃权专用敏感度分析显示，Ambiguous Query 和 Insufficient Input 类别的判断准确率较低（部分低于 50%），存在混淆风险。
- **人工标注成本**：600 个可解决查询 + 602 个弃权实例的标注需要专业数据标注供应商，成本不可忽视。
- **模型依赖**：依赖于多个商业 LLM 判断器（GPT-5、Claude、Gemini），部署成本和 API 可用性可能受限。
