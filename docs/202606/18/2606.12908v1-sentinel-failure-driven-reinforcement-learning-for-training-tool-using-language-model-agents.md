## 摘要
语言模型智能体通过多轮工具使用在解决实际任务方面越来越有效。然而，在实际中训练可靠的使用工具的智能体仍然具有挑战性。尽管强化学习提供了一种通过自身环境交互改进智能体的在线策略范式，但其有效性在很大程度上取决于训练任务的分布。当任务在训练前固定时，任务分布可能与策略不断演化的能力之间出现越来越大的不匹配，导致大量 rollout 被花在信息量不足的任务上。我们提出了 SENTINEL，一个失败驱动的强化学习框架，将求解器的 rollout 失败转化为有针对性的训练任务。SENTINEL 采用 Controller--Proposer--Solver 循环：Controller 分析失败的轨迹并总结反复出现的错误模式，Proposer 生成针对这些弱点的可执行任务，而 Solver 在这些有针对性的任务上进行训练。在 Tau2-Bench Retail 数据集上使用 Qwen3-4B-Thinking-2507，SENTINEL 将 Pass^1 从 66.4 提升至 74.9，并在 Pass^k 指标上超越了在通用合成任务上的强化学习方法。这些结果表明，模型失败为改进使用工具的语言模型智能体提供了有效且可扩展的针对性训练信号来源。

## Abstract
Language model agents are increasingly effective in solving realistic tasks through multi-turn tool use. However, training reliable tool-using agents remains challenging in practice. While reinforcement learning provides an on-policy paradigm for improving agents from their own environment interactions, its effectiveness depends heavily on the training task distribution. When tasks are fixed before training, the task distribution can become increasingly mismatched with the policy's evolving capabilities, causing many rollouts to be spent on uninformative tasks. We propose SENTINEL, a failure-driven reinforcement learning framework that turns the Solver's rollout failures into targeted training tasks. SENTINEL follows a Controller--Proposer--Solver loop: the Controller analyzes failed trajectories and summarizes recurring error patterns, the Proposer generates executable tasks that stress these weaknesses, and the Solver is trained on the targeted tasks. On Tau2-Bench Retail with Qwen3-4B-Thinking-2507, SENTINEL improves Pass\^{}1 from 66.4 to 74.9 and outperforms RL on general synthetic tasks across Pass\^{}k metrics. These results demonstrate that model failures provide an effective and scalable source of targeted training signal for improving tool-using language model agents.

- **研究背景**：大语言模型智能体通过多轮工具使用（如调用 API、访问数据库、搜索引擎等）在实际任务中展现出越来越强的能力，涵盖客户服务、预订管理、工作流自动化和软件工程等应用场景。然而，智能体需要学习正确选择工具和参数、操作正确记录、遵循多步骤流程并遵守领域特定策略，这对训练方法提出了更高要求。

- **核心思想**：利用模型自身的失败轨迹作为训练信号，动态生成针对当前策略弱点的训练任务，而非依赖静态的、预先定义好的通用任务集。

## 2. 论文提出的方法论

### 2.1 核心思想

SENTINEL（**SENTINEL** = **S**elf-**E**volving **N**etwork for **T**ool-**I**ntegrated **N**atural Language Agents）提出了一种**失败驱动的强化学习框架**，将求解器的 rollout 失败转化为有针对性的训练任务。其核心理念是：**让智能体从自己实际犯的错误中学习**。

### 2.2 关键技术细节

#### 2.2.1 框架架构：Controller–Proposer–Solver 循环

| 组件 | 功能 |
|------|------|
| **Controller** | 分析 Solver 的失败轨迹，识别反复出现的错误模式，并生成生成指令（directive） |
| **Proposer** | 接收 Controller 的指令，基于失败轨迹生成针对弱点的可执行训练任务 |
| **Solver** | 在针对性任务上执行强化学习训练，产生更新的策略 |

#### 2.2.2 Controller 模块

1. **轨迹采样**：定期从 rollouts 中采样，并对同一任务和工具调用轨迹的重复失败进行去重，同时采样成功轨迹以避免生成已掌握的任务。
2. **失败模式分析**：调用 LLM 分析器检查每个失败轨迹（包含用户场景、金标准动作、智能体实际工具调用、奖励分解等），输出错误模式集合 Pt = {p1, p2, ..., pK}。
3. **训练历史维护**：维护跨轮次的训练历史 Ht = {(Pj, aj, rj)}，区分持续失败与已解决的问题。
4. **生成指令**：综合当前错误模式 Pt、成功轨迹 St 和训练历史 Ht，生成自然语言指令 dt 传给 Proposer。

#### 2.2.3 Proposer 模块

1. **基于轨迹的任务构建**：首先在环境中构建可行的工具调用轨迹 τ*（执行后状态可达），确保参考答案是自洽的。
2. **失败感知任务改写**：将轨迹 τ* 改写为自然语言用户请求 x*，条件化于 Controller 指令，强调要练习已识别的失败模式（如包含相似产品变体、需精确参数 grounding 等）。

#### 2.2.4 Solver 模块

1. **RL 训练**：使用 GRPO（Group Relative Policy Optimization）在任务缓冲区上进行训练。
2. **奖励设计**：
   - **任务成功奖励** Rtask：验证最终环境状态满足要求且通信正确
   - **动作级检查**：验证中间工具调用是否符合要求（如状态变更前是否完成认证）
   - **退化行为惩罚**：连续相同工具调用惩罚（-0.15）、token 长度惩罚（-0.0001/token）、格式错误惩罚（-0.1）
3. **迭代更新**：Solver 更新后重新 rollout，新失败传回 Controller 形成闭环。

## 6. 论文的主要结论与发现

1. **性能显著提升**：SENTINEL 在 Tau2-Bench Retail 上将 Pass^1 从 66.4 提升至 74.9（+8.5），Pass^2 从 51.6 提升至 60.5，Pass^3 从 43.2 提升至 51.2，全面超越 Base 和 General RL。

2. **通用 RL 的局限性**：即使在 SFT 强模型基础上，直接应用通用任务的 RL 反而导致性能下降（74.3→68.1），而 SENTINEL 继续提升（74.3→78.1），说明针对性任务比通用任务更有效。

3. **失败模式的演化**：训练初期 Controller 发现更多直接缺失动作错误（如未调用计算工具），随着 Solver 改进，后期暴露更多策略级和交互级错误（如未请求用户确认），证明框架能自适应调整训练分布。

4. **失败驱动训练的有效性**：错误类型集中在精确的工具使用行为（如正确的项目/参数选择、多记录推理、策略遵循约束），而非简单工具选择错误，说明针对性的训练任务提供了更有效的学习信号。

5. **核心洞察**：模型失败是改进 tool-using 语言模型智能体的有效且可扩展的训练信号来源。

## 7. 优点

1. **创新的失败驱动范式**：将 RL 中的失败从评估误差转化为训练信号，而非仅用于评估，开创了自适应训练数据生成的新方向。

2. **闭环可控的训练流程**：通过 Controller 分析→Proposer 生成→Solver 训练的循环，实现了模型进化的可控性，可检查模型失败内容、引导数据生成方向、约束 RL 优化。

3. **基于轨迹的任务自洽性**：Proposer 先构建可执行轨迹再生成任务，确保任务与参考答案一致，避免了同时生成任务和答案可能产生的不匹配问题。

4. **训练历史与优先级机制**：Controller 维护训练历史，区分持续失败与已解决错误，对高频失败模式提高优先级，提升了任务生成的效率。

5. **多层次奖励塑造**：结合任务成功奖励、动作级检查和退化行为惩罚，减少 reward hacking，鼓励正确的执行路径。

6. **实验设计合理**：通过 Base、General RL、SFT 对比，清晰展示了失败驱动方法的增量价值。

## 8. 不足与局限

1. **实验领域单一**：仅在 Tau2-Bench Retail（客户服务）单一领域验证，其他工具使用场景（如 API 调用、软件工程、工作流自动化）的泛化效果未知。

2. **任务覆盖依赖失败暴露**：生成的训练任务取决于当前 Solver 在训练中暴露的失败，可能遗漏罕见场景或未被充分采样的测试用例。

3. **Controller/Proposer 能力依赖**：失败模式分析的准确性和任务生成的质量依赖于 LLM 分析器的能力，可能存在误判或遗漏。

4. **迭代效果描述有限**：虽然框架是多轮迭代的，但论文对多轮训练后任务分布的具体变化、对不同失败模式的覆盖进度等描述较为简略。

7. **通用 RL 基线的任务质量未详细对比**：General RL 使用的预生成任务与 SENTINEL 生成任务的难度、规模等未做详细对比分析。
