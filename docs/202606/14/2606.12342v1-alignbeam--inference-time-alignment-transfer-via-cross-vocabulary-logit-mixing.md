## Abstract
Domain fine-tuning degrades the safety of large language models: fine-tuned specialists readily comply with harmful prompts framed in domain language. Existing inference-time defenses that mix logits from a safe anchor model require both models to share a vocabulary, which rules them out for the cross-family specialists where safety is most degraded. We present ALIGNBEAM, a training-free method that lifts this restriction by translating anchor logits into the target model's vocabulary token-by-token at each decoding step; a small LLM judge then selects the safest among K candidate continuations. No weights are changed, and the safety-utility trade-off can be tuned at deployment without retraining. Across both cross-vocabulary and same-vocabulary evaluation pairs, ALIGNBEAM substantially raises refusal on adversarial benchmarks while keeping task accuracy and inference overhead within practical bounds. The results show that safety alignment can be transferred between model families at inference time, without touching either model's weights.

## ALIGNBEAM 论文详细总结

### 1. 核心问题与整体含义

**研究动机与背景**：
- **领域微调危害安全**：医学、金融、数学等领域的专家模型在微调后，其安全性对齐被严重削弱。例如，医疗专家模型会对以临床语言包装的药物滥用问题给予帮助；金融模型会对伪装成投资组合问题的欺诈计划提供建议。
- **现有防御方法的局限**：Safe Decoding 等推理时防御方法要求锚模型和目标模型共享词汇表（shared vocabulary），这在跨家族模型（cross-family）中无法满足——而恰恰是这些跨家族专家模型的安全退化最为严重。
- **核心问题**：如何在**不修改任何模型权重**的前提下，将安全对齐从一个小型的、对齐良好的锚模型迁移到目标领域专家模型？

### 2. 方法论

**核心思想**：
ALIGNBEAM 是一个**无需训练（training-free）的推理时方法**，通过"文本桥接"（text bridge）在异构词汇表之间传递 logit 信息，并用 beam search 和 LLM judge 在 K 个候选续写中选择最安全的输出。

**关键技术细节**：

**（1）跨词汇表 Logit 桥接混合（Logit Bridge Mixing, LBM）**：
- 在每个解码步骤，将锚模型的 top-B（默认 B=50）预测 token 解码为文本字符串
- 用目标模型的 tokenizer 重新编码该字符串，投影到目标模型词汇表空间
- 混合分布公式（公式1）：`f[did] += α * ps + (1-α) * Pd(did)`
  - 其中 α ∈ [0,1] 是混合权重（部署时可调），ps 是锚模型概率，Pd 是目标模型概率
  - α=0 时锚模型不贡献 logit 信号；α=1 时完全使用锚模型分布
- 三种多 token fallback 变体：
  - **LBM-DROP**（默认）：丢弃多 token 映射结果并重新归一化
  - **LBM-FIRST**：只保留第一个子 token
  - **LBM-EXACT**：只保留 1:1 精确匹配

**（2）三阶段 Beam Pipeline**：
- **Phase 1 - Priming**：对查询 x 同时运行锚模型和目标模型，通过 LBM 获得混合分布，选取 top-K tokens 作为 beam 根节点
- **Phase 2 - Mixed Decoding**：每个 beam 贪婪扩展 N-1 步（默认 N=6），每步都经过 LBM 混合
- **Phase 3 - Draft Continuation**：用目标模型单独完成 K 个 beam 的生成（利用 KV-cache 复用），小型 LLM judge（J, 默认 Qwen2.5-3B-Instruct）在 1-5 危害量表上打分，选择最安全的 beam（score ≤ τ=2.5）

**（3）部署可调参数**：
- **N**（混合深度）：控制安全引导的 token 数量，默认 N=6（DeepSeek-Math 用 N=20）
- **α**（混合权重）：控制安全-效用权衡，默认 α=0.5
- **K**（beam 数量）：K=3（安全模式，默认），K=1（效用模式，最低开销）

### 6. 主要结论与发现

**（1）安全性提升效果**：
- ALIGNBEAM 将 AdvBench 拒绝率从 38.1% 提升至 91.5%（+53.4 pp 宏观平均）
- 将 HarmBench-Std 拒绝率从 25.9% 提升至 76.4%（+50.5 pp 宏观平均）
- 四对模型在 AdvBench 上达到 91-96%，HarmBench-Std 上达到 78-81%

**（2）安全-效用权衡**：
- 任务准确率仅下降 0.4 pp（GSM8K: 77.0%→76.6%；MedQA: 13.1%→12.7%）
- 良性过拒绝率上升（OR-Bench-Hard: 11.0%→22.3%；XSTest: 5.6%→40.0%），但低于 LlamaGuard 的 34.5%

**（3）早期 token 安全性集中**：
- 仅 3 个混合步骤（N=3）即可捕获 +61.9 pp 的安全增益
- N≥6 后收益急剧递减，确认安全对齐集中在前几个 token 位置

**（4）锚模型对齐的重要性**：
- Beam search 机制本身提供 ~25-52 pp 拒绝率提升
- 对齐良好的锚模型额外增加 ~25 pp（在免费情况下）

**（5）与基线方法比较**：
- 在有害基准上与 LlamaGuard 相当（78-94% vs 89-100%），但良性过拒绝率更低（22.3% vs 34.5%）
- 在同词汇表控制组上，优于 Proxy Tuning 和 Top-k CD（尤其在上下文敏感基准上）

**（6）规模扩展性**：
- 在 70B 目标模型上验证有效（Llama-3.1-70B: AdvBench +70.9 pp, HarmBench-Std +62.3 pp）

### 7. 优点

**（1）突破词汇表限制**：
- 首次实现跨词汇表（cross-family）的推理时安全对齐迁移
- 文本桥接方法优雅地解决了异构 tokenizer 的概率混合问题

**（2）零训练开销**：
- 不修改任何模型权重，无需重新训练或微调
- 部署时可灵活调整 N、α、K 等参数

**（3）部署灵活性**：
- 三个维度可独立调节：深度 N（安全-速度权衡）、权重 α（安全-效用权衡）、beam 数 K（安全-延迟权衡）
- 提供 K=1 效用模式和 K=3 安全模式两档配置

**（4）安全集中性利用**：
- 利用安全对齐集中在前几个 token 的特性，仅需浅层混合（N=6）
- 大幅降低推理开销，同时保持有效性

**（5）实验设计严谨**：
- 多样化的模型对覆盖（跨家族专家、同词汇表控制）
- 丰富的消融实验验证各组件贡献
- 使用官方基准评判器确保公平比较

**（6）鲁棒性验证**：
- 种子鲁棒性：±0.7 pp 波动
- Anchor 提示格式鲁棒性
- 多 LBM 变体验证

### 8. 不足与局限

**（1）校准上限受锚模型限制**：
- 锚模型的过拒绝倾向会被继承（OR-Bench-Hard ≈20%, XSTest ≈40%）
- 恶意过拒绝无法通过调整 α 改善，取决于锚模型校准质量

**（2）前缀长度瓶颈**：
- 混合仅覆盖前 N 个 token，多句攻击（如 WildJailbreak）效果有限（仅 +2-6 pp vs +51-77 pp on AdvBench）
- 需要根据目标模型调整 N（DeepSeek-Math 需要 N=20）

**（3）评估范围有限**：
- 仅单轮、≤200 tokens 提示
- 多轮 jailbreak 和自适应攻击未评估
- 仅英文评估

**（4）"Sorry-but-continues" 模式**：
- 模型产生拒绝短语后继续输出有害内容（DeepSeek-Math 最明显）
- 需要更长的混合窗口或更严格的 judge 阈值来缓解

**（5）输出截断问题**：
- max_new_tokens=150 可能导致拒绝响应被截断
- 截断响应被判定为非拒绝，低估真实安全增益

**（6）Anchor 依赖性**：
- 锚模型的拒绝风格需与评估指标匹配（替代锚模型 Llama 虽 judge 准确率高但字符串匹配低）
- Anchor 选择对结果有显著影响

**（7）推理开销**：
- 相比无防御基线仍有 4.6-6.6× 开销（K=3 模式）
- 在资源受限场景中部署可能面临挑战

**（8）LoRA 微调场景验证有限**：
- 仅一个 LoRA 权重组合（r=6, NF4, 5000 步）验证
- 更广泛的安全退化微调场景未系统评估
