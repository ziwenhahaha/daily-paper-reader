Abstract 

With AsgardBench we aim to evaluate visually grounded, high-level action sequence gener-ation and interactive planning, focusing specifically on plan adaptation during execution based on visual observations rather than navigation or low-level manipulation. In the landscape of embodied AI benchmarks, AsgardBench targets the capability category of interactive planning, which is more sophisticated than offline high-level planning as it requires agents to revise plans in response to environmental feedback, yet remains distinct from low-level execution. Unlike prior embodied AI benchmarks that conflate reasoning with navigation or provide rich corrective feedback that substitutes for perception, AsgardBench restricts agent input to images, action history, and lightweight success/failure signals, isolating interactive planning in a controlled simulator without low-level control noise. The benchmark contains 108 task instances spanning 12 task types, each systematically varied through object state, placement, and scene configuration. These controlled variations create conditional branches in which a single instruction can require different action sequences depending on what the agent observes, emphasizing conditional branching and plan repair during execution. Our evaluations of leading vision language models show that performance drops sharply without visual input, revealing weaknesses in visual grounding and state tracking that ultimately undermine interactive planning. Our benchmark zeroes in on a narrower question: can a model actually use what it sees to adapt a plan when things do not go as expected?

## 1 Introduction 

Vision language models have enabled agents to generate plausible action sequences for embodied tasks, prompting renewed interest in benchmarks that evaluate multimodal reasoning in interac-tive environments. Many existing benchmarks mix high-level reasoning with navigation and other environment-specific mechanics and agents may succeed by exploiting textual priors, memorized procedures, or detailed corrective feedback that substitutes for perception, even when visual ground-ing is weak. We found that producing a plan upfront is rarely enough for embodied tasks, as agents must continually reassess what they see and adjust accordingly. They must repeatedly interpret visual observations, detect when assumptions no longer hold, and revise action sequences accord-ingly. Despite its importance, visually grounded interactive planning is rarely evaluated on its own. In several widely used benchmarks, most notably in ALFWorld [1] and EmbodiedBench [2], text only agents perform competitively or even better, and detailed failure messages often reveal why an action failed, reducing the need for genuine multimodal reasoning. We designed AsgardBench to isolate this gap, deliberately abstracting away navigation and low-level manipulation to avoid conflating reasoning failures with control errors. Recent embodied AI evaluation surveys [3, 4] categorize benchmarks along two axes: the capability being measured (e.g., navigation, manipulation, planning) and the evaluation setting (e.g., simulators, datasets, real world environments). Within this taxonomy, AsgardBench targets interactive planning as the capability being evaluated and uses a controlled simulator as the evaluation setting. Agents inter-act with the environment through a simplified action space and receive only lightweight success or failure signals, requiring them to infer task relevant state, such as object cleanliness, container acces-sibility, or scene configuration, directly from images. Tasks in AsgardBench are constructed so that a single instruction can require different action sequences depending on what the agent observes during execution. This setup requires conditional branching and plan repair based on visual obser-vations, exposing whether agents can maintain coherent state across steps and adapt plans when observations contradict expectations. Through systematic ablations, we show that visual grounding is essential for success in AsgardBench, clarifying where current multimodal agents struggle and distinguishing visually grounded plan adaptation from broader embodied AI benchmarks where multimodal perception can often be bypassed. 

1.1 Contributions 

Our work makes the following contributions: 

• A benchmark that isolates visually grounded interactive planning. It removes naviga-tion, low-level manipulation, and rich corrective feedback so the evaluation centers on whether agents can ground action sequencing in visual observations. Because tasks require multiple steps with conditional dependencies, agents must not only react to what they see but also main-tain coherent progress toward their goal—tracking intent across steps rather than treating each observation in isolation. 

• Controlled task variations that require conditional branching during execution. Each instruction admits multiple valid action sequences depending on object cleanliness, accessibil-ity, or scene configuration, enabling precise evaluation of visual grounding, state tracking, and adaptive plan revision. 

• A targeted empirical analysis of state-of-the-art multimodal models within this ca-pability category. Through image based, text only, and feedback ablated conditions, we reveal consistent failures in visual discrimination, internal state maintenance, and interactive planning, highlighting gaps that are not exposed by broader embodied AI benchmarks. 22 Related Work 

Embodied AI benchmarks have traditionally evaluated agents on end-to-end task execution that combines navigation, manipulation, and language grounding. Early benchmarks like ALFRED [5] and its photo-realistic variant, ReALFRED [6], frame household activities as long horizon instruc-tion following problems in simulated 3D environments. These benchmarks have advanced multi-modal agents, but their integrated design makes it difficult to isolate high-level reasoning failures from errors in navigation, viewpoint selection, or low-level interaction. Empirical results, particu-larly in ALFWorld [1] and as reflected in EmbodiedBench [2], show that agents can often perform competitively using textual priors or memorized action templates, even when visual grounding is weak or absent. Large scale activity benchmarks, for example BEHAVIOR-1K [7], extend this paradigm by in-creasing task diversity and physical realism, emphasizing general manipulation competence across 1,000 everyday activities. While these environments stress robustness and scalability, performance is often strongly influenced by challenges in low-level control and physics, which can obscure de-ficiencies in interactive decision making and plan adaptation. Similarly, procedurally generated extensions of AI2-THOR [8] like ProcTHOR [9] increase visual diversity but retain a focus on navigation and interaction mechanics. Meta-benchmarks (i.e., EmbodiedBench [2]) attempt to broaden evaluation coverage by ag-gregating tasks across multiple environments and organize them by capability categories. This approach improves comparability across models but inherits many of the design assumptions of their source benchmarks, including varying degrees of navigation complexity and simulator specific affordances. As a result, while these suites provide comprehensive evaluation across multiple di-mensions, the varying feedback mechanisms across component benchmarks can make it challenging to isolate interactive planning as a distinct capability. Other work examines planning in offline settings, where agents generate full action sequences without interacting with the environment. ET-PLAN-BENCH [10] evaluates whether agents can generate valid plans with spatiotemporal reasoning, though they often provide structured task representations or environmental feedback that may reduce the demands on continuous real time visual state estimation during execution. LoTa-Bench [11] and related frameworks examine long horizon planning with language oriented task decomposition, though the emphasis on symbolic representations may reduce the need for visual grounding. ActPlan-1K [12] emphasize procedural plan generation from video observations but evaluate plans in offline settings without requiring agents to revise them during execution. A related line of work evaluates agents in dynamic, stepwise environments. WebArena [13], WebShop [14] and OSWorld [15]. These settings share with AsgardBench the requirement that an agent operate interactively in a simulator, making them conceptually aligned with the broader goal of evaluating interactive planning. However, web based and desktop control environments differ in several important ways. In particular their action spaces involve symbolic clicking, typing, or selecting UI elements. AsgardBench employs a fully controlled 3D simulator with unstructured RGB observations. Other diagnostic frameworks, including CanDo! [16] and the Embodied Agent Interface (EAI) [17], aim to decompose embodied decision making into modular components and provide fine grained error analysis. While valuable for system level diagnosis, these approaches often rely on structured state, explicit affordances, or detailed error signals, which can mask the extent to which agents genuinely rely on visual input during execution. Unlike settings where text only agents succeed by exploiting structured feedback, AsgardBench requires models to interpret the scene itself, as confirmed by systematic ablations that remove or 3degrade visual input. By varying object states and placements, providing only minimal feedback, and requiring interactive execution, AsgardBench exposes limitations in models’ ability to maintain coherent state, detect mismatches between expectation and observation, and adapt plans mid-execution. This positions AsgardBench as the first benchmark to isolate interactive planning in embodied environments, filling a gap between physically rich embodied simulations and abstract offline planning evaluations. 

## 3 Benchmark Design 

In order to focus on the ability of models to adapt plans as new visual information becomes available during execution, skills such as navigation and grasping are deliberately excluded, as benchmarks that emphasize navigation or manipulation can make it difficult to disentangle high-level reasoning failures from low-level control errors, as seen in embodied evaluations derived from ALFRED [5], BEHAVIOR-1K [7], and AI2-THOR-based tasks [8]. Consistent with the capability-focused eval-uation principles discussed in section 1 [3, 4], AsgardBench adopts a controlled simulation based design that targets interactive planning while minimizing confounding factors from navigation and low-level control. AsgardBench was built on AI2-THOR, an open source, mature simulation environment widely used in embodied AI research [8]. AsgardBench uses three of AI2-THOR’s scene types: Kitchens 

(for cooking and organization tasks), Living Rooms (for simple interaction tasks), and Bath-rooms (for cleaning tasks). For the purposes of the AsgardBench benchmark, a higher-level action abstraction layer was added, simplifying several elements for the agent so that the focus remains on visually grounded interactive planning and not motor control. In AI2-THOR, to interact with an object (e.g., PUT or OPEN), an agent must navigate to the object and identify a position relative to it from which the action can be executed. For example, when opening a fridge, the agent can be neither too close nor too far from the door. AsgardBench simplifies this by automatically placing the agent in a position from which it can interact with the object, including setting whether the agent is crouched or upright. In this way, we remove navigation entirely by placing the agent directly in front of the relevant object, ensuring that errors reflect planning difficulties. For all object interactions, however, we still require that the object be within the agent’s field of view before any interaction can occur. A "FIND object" action was also added, which positions the object within the agent’s field of view, eliminating the need for visual search and navigation while still requiring the agent to interpret what it sees and update its plan accordingly. The sizes of containers and objects in the AI2-THOR environment vary, meaning that not every container will fit an object of a particular type. For example, a Bowl may fit in only a subset of the available Cabinets. This has been abstracted in the simulation environment so that the agent does not need to determine which container of a particular type to employ. When the agent requests an "OPEN Drawer" or "PUT Drawer" action, AsgardBench automatically selects a drawer that fits the object. Likewise, when the agent performs a "PUT CounterTop" whilst holding an item, AsgardBench selects a CounterTop that has sufficient space for the held item, removing low-level affordance reasoning that is not central to the benchmark’s goals and keeping the evaluation focused on interactive planning.

To create meaningful variations, we designed each Task and its variations so that the required action sequence diverges based on visual observations made during execution, ensuring that interac-tive plan adaptation is necessary for success. For example, in Figure 1 (left panel), the agent forms an initial plan for " Consume coffee from a mug, then wash and store the mug " that includes finding the mug and then putting it in the CoffeeMaker. However, upon observing that the Mug is dirty, it must alter the plan to include washing the Mug first (center panel). Subsequently, the agent observes that the SinkBasin contains other objects, and it must further alter its plan to remove items from the SinkBasin before washing the Mug (right panel). 5Concretely, at each turn the model produces a complete action sequence covering all remaining steps, but only the first action in that sequence is executed. After execution, the agent receives updated images and minimal success/failure feedback, and the previous plan is carried forward for re-evaluation. This loop ensures that the model must continuously reconcile its plan with what it actually observes, rather than committing to a fixed sequence up front. Full details of the observation space and prompt structure are provided in subsection A.2. By requiring the agent to revise its plan dynamically during execution, the benchmark ensures that the agent must ground its planning in the visual environment. Figure 4 shows that even with detailed feedback on why actions fail, the agent’s performance drops dramatically if it does not have access to visual observations. Prior work on derived text only abstractions of embodied benchmarks, most notably ALFWorld [1] for ALFRED [5], shows that agents achieve high success rates in the text only environment but experience severe degradation when transferred to the embodied setting. More broadly, reducing control complexity consistently improves performance across embodied settings: hierarchical skill decompositions substantially outperform flat policies in physics-based manipulation [18], and oracle stopping conditions boost success in vision-and-language navigation [19], further underscoring how dependent current agents remain on simplified action abstractions. 

3.2 Evaluation Metrics 

A Task is considered to have been completed by the Agent when all task specific goals have been met. For example, the Coffee has been drunk, the Mug is clean, and the Mug has been placed in a cabinet . See Table 4 for all task specific goals. Additionally, for the kitchen scene, the benchmark requires that the world be left in a reasonable state (i.e., the Refrigerator is not open and the Faucet is not running), mirroring cleanup constraints used in prior embodied benchmarks [5, 8]. These metrics are designed to evaluate visually grounded interactive planning: the agent must not only achieve the final goals but must do so by adapting its plan based on visual observations made during execution. Because different scene variations require different action sequences, success reflects the agent’s ability to detect visually observable state, maintain internal task progress, and revise its plan accordingly. 

3.3 Termination Conditions 

Determining when to terminate a test is not straightforward. Agents should be given adequate opportunity to solve a Task, yet time and compute cycles should not be wasted when the Agent is floundering. The easiest Tasks can be completed in only four steps, but in some cases the Agent may take over 100 steps to reach the goal state, a pattern also observed in long-horizon embodied evaluations such as ALFRED and BEHAVIOR-1K [5, 7]. For AsgardBench, testing for a Task is terminated under the following conditions, balancing exploration with practical compute limits: 1. All goals are met. 2. The Agent attempts ten undoable actions in a row. 3. The Agent repeats the same action or sequence of actions more than eight times. 4. The Agent exceeds the maximum allowed number of steps (soft or hard limits). On the last point, the benchmark imposes soft and hard maxima on the total number of steps that an Agent is allowed to use. The soft limit is 1.5x the number of steps in the known solution created by the plan generator (see the Benchmark Generation section in A.3), with a minimum of 15 steps. The hard limit is 2x the number of steps, with a minimum of 20 steps. When the Agent reaches the soft limit, it is allowed to continue as long as it has interacted with an object that was not employed by it in the previous ten steps or has completed a new subgoal, until it reaches the 6hard limit. This novelty check allows an Agent to continue if it is trying new actions but helps terminate agents that have become stuck. We chose these termination thresholds empirically by examining Agent trajectories. It remains possible that an Agent might eventually reach its goals given additional steps, but we believe the current approach keeps a reasonable balance between allowing recovery from errors and preventing unbounded exploration. These conditions also ensure that evaluation remains focused on interactive planning: agents are given enough room to revise their plans in response to new visual informa-tion, but not so much freedom that uncontrolled exploration obscures whether plan adaptation is occurring. 

## 4 Results 

We tested leading frontier and near frontier proprietary and open source models with images and in Text Only mode. For cost control, some of the more expensive models were not included in all experimental conditions. Following the " what to evaluate / where to evaluate " framework [3, 4], these results speak directly to the interactive planning capability category. Success depends on adapting plans based on what the agent sees, not on producing a fixed plan or relying on detailed feedback. As shown in Figure 2, all but the lowest performing models benefit from visual input, more than doubling performance over the Text-Only condition. This indicates that success on AsgardBench requires visual grounding and cannot be achieved through memorized action templates or static plans. This stands in contrast to derived text only abstractions of embodied benchmarks, like ALFWorld [1] for ALFRED [5], where agents can perform competitively by exploiting textual priors or structured feedback. We also examined the range of steps required to successfully complete each Task. Figure 3 shows the range of steps plotted against the average success rate for each of the 108 tasks in the benchmark. Tasks that require longer action sequences tend to be more difficult, a pattern also observed in long horizon embodied evaluations derived from ALFRED [5] and in large scale activity benchmarks like BEHAVIOR-1K [7]. Longer horizon tasks in AsgardBench typically involve more assumption changing variations, increasing the need for interactive plan adaptation. In the Baseline configuration, the prompt included a list of the Agent’s previous actions and an indication of whether each attempted action was successful ( Success/Failure ). Two additional variations were examined: removing feedback about the success or failure of each action entirely ( No Feedback ) and providing detailed feedback that explains why an action failed ( Detailed Feedback ). See Table 2 for a demonstration of how the prompt varies by feedback type. In the No Feedback condition, performance degrades because the agent must rely solely on visual feedback to determine whether an action has succeeded. This reliance is challenging in AI2-THOR [8], as visual differences between states can be extremely subtle (for example, the on/off state of a Toaster or Microwave). 

Detailed Feedback substantially reduces the need for visual grounding, as the feedback provides explicit corrective information that could, in principle, guide the agent toward the correct action sequence. Detailed Feedback does raise performance across all models. For some models—notably Qwen3-VL, Mistral-Large-3, and Maverick— Text-Only with Detailed Feedback can match or exceed the image-based Baseline (see Figure 4), mirroring findings from ET-PLAN-BENCH [10] and LoTa-Bench [11], where structured or explicit feedback reduces the need for perception or interactive plan repair. However, for the strongest vision-capable models (e.g., Kimi-K5.2, GPT-4o), image-based performance remains substantially higher even compared to Text-Only with Detailed Feedback ,suggesting that visual grounding provides information beyond what corrective feedback alone can supply. 7Figure 2. ## 5 Ablations 

In addition to variation in the amount of feedback and Text-Only variations, we experimented with the following ablations: (1) removing the hand overlay, (2) removing the Things to Remember 

section, and (3) showing only the current-state image. These ablations probe different components of visually grounded interactive planning, including state estimation, memory maintenance, and the use of visual history for plan repair. 11 5.1 No Hand 

In AI2-THOR, when items are picked up by the Agent, they are displayed in the bottom center of the image without any indication of a hand. In some cases, even for a human, it can be difficult to tell if the object is resting on a counter in front of the agent or being held by it (see Figure 8). For this reason, we superimposed a translucent hand over the image whenever an object was being held, to make the held-object state visually unambiguous. The position of the hand was shifted to roughly match the position of the held object on the screen.    

> Figure 8. Ambiguity in interpreting held objects without the hand overlay. In both images, the saltshaker is being held by the agent, but without the superimposed hand overlay it can easily be misinterpreted as resting on the table. This illustrates how subtle spatial cues in AI2-THOR can lead to incorrect state inferences, motivating the inclusion of the hand overlay in AsgardBench.

Our expectation was that this would help the models disambiguate as the presence of the hand better aligns with the models’ training data. We chose translucent rather than opaque because the object visibility check in AI2-THOR cannot account for the superimposed hand image, and we needed to ensure that items are indeed visible to the Agent. To test our expectations, we evaluated models without the superimposed hand. For these evaluations, we augmented the prompt with additional instructions that explained that held objects would be seen as floating. All models were found to perform worse without the superimposed hand (see Figure 10). We also examined the logs and confirmed that models were having difficulty interpreting images without the hand. Figure 11 shows an example where without the hand overlay the model interpreted a held coffee mug as being on the DiningTable. This ablation highlights that accurate state estimation, specifically determining whether an object is held or placed, is a prerequisite for correct interactive plan adaptation.

5.2 Things to Remember 

During evaluation, Agents are only provided images from the current and previous turn and past actions. We expected that they might struggle without additional context. For example, if the Agent notes that a plate is dirty but then turns away to pick up a sponge, the knowledge of the dirty plate is lost. To mitigate this, we prompted the model to generate a list of Things to Remember 

that is shared with the Agent on the subsequent turn (see the Prompt Example section in A.9). Observationally, the content generated by the models for Things to Remember was of high quality (see Figure 12). In the No Things to Remember condition, we removed the auxiliary memory summary generated by the model. This ablation tests whether models can maintain task-relevant state purely from the action history and visual observations, without relying on an explicit textual recap. Removing 

Things to Remember had mixed results. On the surface, the more performant models appeared to benefit from the additional information, while the less performant models suffered (see Figure 13). In our default setup, we provided the model with two images: the state of the world before the previous action was applied, and the state after. We hypothesized that this temporal context would make it easier to determine the success or failure of each action. As shown in Figure 15, model performance degrades when only the current state is provided, supporting this hypothesis. We speculate that the previous image helps because (1) the model can compare states to determine the outcome of its actions, and (2) it provides additional spatial context when the viewing direction changes. Short-horizon visual history appears to support interactive planning by helping models detect state changes and repair their plans accordingly. 

## 6 Discussion 

The AsgardBench results highlight several consistent challenges for current multimodal agents. Even the strongest models show substantial performance drops when visual input is removed, in-dicating that the benchmark successfully enforces reliance on multimodal perception rather than letting models fall back on memorized semantic patterns or simulator feedback. This stands in contrast to many existing embodied benchmarks, where text only agents can perform surprisingly well because they receive informative error messages or can rely on stereotyped action templates [5, 1]. Across Task types, agents struggle when execution depends on fine grained visual cues. Small differences, like whether a dish is clean or whether a container is open, regularly lead to incorrect state inferences. Visual misinterpretations and clutter mistaken for task relevant objects further show that current vision pipelines remain brittle in cluttered 3D scenes. Once a model adopts an incorrect assumption, it often repeats undoable actions or becomes stuck in narrow behavior loops, suggesting state tracking, mismatch detection, and plan adjustment remain significant bottlenecks for interactive planning. Maintaining an accurate internal model of the environment is another recurring difficulty. Be-cause AsgardBench does not provide structured symbolic state, agents must infer and remember object states from images and action histories. Adding the Things to Remember mechanism improved consistency for some models but did not provide uniform benefits, highlighting that such ad-hoc memory scaffolding is insufficient for reliable state maintenance. Maintaining task relevant state across interactive execution remains difficult, and prompt-based memory scaffolding provides only inconsistent support. The guidance we provided for generating the memory summary was deliberately simple; a more structured or task-aware memory mechanism might yield more reliable state tracking. The ablation studies reveal how dependent current systems are on auxiliary cues. Removing the hand overlay or the previous-state image consistently reduced performance, and removing even minimal action feedback made tasks considerably harder. Conversely, providing detailed failure messages sharply improved accuracy, even for Text-Only agents. Together, these results show that models rely heavily on cues that simplify state estimation or plan repair, and struggle when required to infer state changes directly from visual observations. This is consistent with the benchmark’s goal of evaluating interactive planning under minimal feedback, where progress must come from reasoning grounded in multimodal perception. 

6.1 Benchmark Limitations 

While AsgardBench is effective as a diagnostic tool, it has limitations. First, the benchmark omits navigation, grasp physics, and other low-level control elements. This abstraction isolates high-level reasoning but reduces ecological validity relative to full embodied settings. Second, the tasks and scenes are confined to AI2-THOR and a curated object set, which limits visual and physical diversity. Third, images are captured under relatively controlled conditions; performance in more varied or photorealistic environments may differ. Fourth, requiring models to produce a full action sequence each turn (even though only the first action is executed) may favor planning-oriented language models over agents designed for reactive, single-step policies. Finally, while minimal feedback is essential for the benchmark’s goals, real systems may legitimately incorporate richer sensory inputs, which AsgardBench does not capture. It does not aim to represent the full space of embodied intelligence. Instead, it provides a controlled way to expose where current multimodal models break down and where future work is needed, including more robust visual understanding, better internal state representations, and methods for integrating perception and planning more tightly during execution. This capability focused design is consis-tent with recent evaluation surveys, which emphasize the need for benchmarks that disentangle high-level reasoning from navigation and low-level control. ## 7 Conclusions 

AsgardBench was developed to evaluate a capability that is frequently overlooked in existing em-bodied AI benchmarks: visually grounded interactive planning, the ability of an agent to ground its decisions in visual observations and revise its plans as the environment changes during execution. By removing navigation and low-level manipulation, and by limiting feedback to simple success or failure signals, the benchmark isolates the role of perception in high-level action sequencing. The results across a range of state-of-the-art multimodal models show that this setting remains challenging. Agents struggle with subtle visual distinctions, maintaining coherent internal state, and adjusting plans when initial assumptions prove incorrect. The results show that the current generation of vision-language models struggle in scenarios where perception must drive ongoing decision making and plan repair. In the terminology of the Embodied AI Evaluation Survey and the Authorea survey, AsgardBench directly targets the interactive planning capability category ("what") and evaluates it in a controlled simulator setting ("where"), addressing a gap both surveys identify. Although AsgardBench is intentionally constrained, the framework provides several practical insights. The consistent performance drop in Text-Only conditions demonstrates that visual ground-ing is necessary for success. The ablation studies further show how existing systems may rely on 17 explicit error messages or auxiliary cues. These findings suggest that progress on interactive plan-ning will require models to integrate perception, memory, and planning more tightly and can’t treat these components as loosely connected modules. This echoes the surveys’ observation that interac-tive planning remains under-evaluated and that disentangling high-level reasoning from navigation and low-level control is essential for meaningful capability assessment. There are also areas where the benchmark can be expanded. Introducing more varied scenes, lighting conditions, and object sets would broaden coverage. Incorporating controlled navigation or limited manipulation elements could help bridge the gap between abstract plan evaluation and full embodied tasks. Additional ways of representing or probing memory, beyond the Things to Remem-ber mechanism, may also help clarify how different architectures handle long horizon dependencies. Finally, evaluating models that combine explicit state tracking modules with large scale perceptual front ends could shed light on hybrid approaches that balance learned and structured reasoning. Such extensions would further align AsgardBench with the capability oriented evaluation principles emphasized in recent surveys. AsgardBench offers a targeted, reproducible testbed for studying visually grounded interactive planning. While it does not aim to model all aspects of embodied intelligence, it exposes specific limitations that are likely to matter in more complex environments as well. As multimodal agents continue to advance, benchmarks of this kind will be essential for distinguishing genuine progress in interactive planning from improvements driven by feedback shortcuts or textual priors. By situating AsgardBench within a principled evaluation taxonomy, we aim to support more systematic progress on perception-conditioned reasoning and clarify where current models fall short.

• OPEN {container_name} – Open {container_name} .

• PICKUP {object_name} – Pick up {object_name} and hold it.

• SLICE {object_name} – Slice {object_name} or crack an egg. 

– To "SLICE", the agent must be holding a knife. This action is also used for cracking an egg, in which case no knife needs to be held. 

• SPRAY {object_name} – Spray {object_name} using the held object. 

• TOGGLE_ON {object_name} – Turn on {object_name} .

FIND – abstracts away navigation entirely: the benchmark handles the low-level actions re-quired to locate the target object and positions the agent in front of it. If the object is inside a container, the agent is placed in front of that closed container. 

PUT – handles placement logistics. The simulator verifies that space exists for the held object at the target location. If not, it automatically tries other containers of the same type. For example, if ‘PUT Cabinet‘ fails because ‘Cabinet_1‘ is full, it will try ‘Cabinet_2‘, ‘Cabinet_3‘, and so on, navigating to each and opening as needed until it finds available space or exhausts all candidates. Note that the initial ‘PUT‘ requires the agent to already be facing a valid, visible container (the automatic navigation occurs only when retrying alternative containers). 

A.2 Observation Space 

At each turn, the agent receives the following inputs: 

Action history: A numbered list of previously attempted actions with simple feedback indi-cating success or failure (e.g., ‘PICKUP Mug Success‘ or ‘CLEAN Mug Failure‘). 

Suggested plan: Rather than outputting only a single next action, the model outputs a complete action sequence covering all remaining steps. However, only the first action in the sequence is executed at each turn. In subsequent turns, the previous plan is provided back to the model alongside the newly observed state, allowing it to re-evaluate and revise the plan before selecting its next action. 

Images: Two 1024 ×1024 RGB images are provided: the current scene and the state before the previous action. Comparing these images enables the agent to infer whether its last action succeeded. 

Hand overlay: When the agent is holding an object, a translucent hand (60% opacity) is superimposed on the image at a predefined location aligned with the held item. This reduces ambiguity between held items and objects resting on nearby surfaces. 21 A.2.1 Prompt Structure 

Each prompt comprises the following components: 

• High-level goal – e.g., "Make a slice of toast and serve it on a plate." 

• Available objects – the list of objects present in the current scene. 

• Available actions – the action space with syntax and descriptions. 

• World rules – constraints governing valid action sequences (e.g., "Items must be placed in the sink before cleaning," "The microwave must be open before placing items inside"). See A.9 for an example. 

• Agent notes: – Action history with success/failure feedback. 

– The suggested plan from the previous response. 

• Task instructions – detailed guidance on how to structure reasoning and output. 

• Examples – three in-context examples demonstrating correct reasoning and action format. For a full prompt example, see A.9. 

A.3 Benchmark Generation 

AsgardBench uses a pre-generated, fixed test set to ensure reproducibility and enable fair compari-son across models. Each task instance is referred to as a plan (a specific configuration of a task type, scene, object placements, and variation conditions). All plans are constructed before any model evaluation occurs, ensuring that every model is tested on the same curated collection of scenarios. A plan captures everything required to define a solvable task: which scene to use, where objects are placed, which preconditions apply (e.g., whether the plate starts dirty), and the goals that must be fulfilled for the given task. Because object placement is randomized and tasks involve conditional logic (e.g., dirty items must be cleaned before use), the same task type can yield hundreds of distinct plans, each requiring a different sequence of actions to complete. AsgardBench scenes vary in which objects are present, creating natural variation across plans. For each task, the positions of objects are additionally randomized while ensuring that all task-relevant items (e.g., dishes, food items) remain visible and are not spawned inside closed containers such as drawers or refrigerators.

A.3.2 Plan Sanitization 

To maintain consistency and avoid ambiguous or unsolvable states, several modifications are applied during scene preparation, before a given plan is finally included in the final test set: 

• Remove duplicate items to avoid ambiguous commands (e.g., multiple vases making "PICKUP Vase" unclear). 

• Avoid placing silverware in the sink, as deep basins can occlude them from the agent’s viewpoint. 

• Remove visually confusable objects, such as the PepperShaker, which is nearly indistinguishable from the SaltShaker in most views. 

• Remove objects that cannot be stably placed, either because they fall through surfaces or cannot contain other items. 

• Exclude malformed AI2-THOR scenes, such as kitchen floor plans where objects spawn below the floor or where an invisible mesh blocks placement (e.g., FloorPlans 8, 9, and 17). Not every random configuration yields a solvable task (for example, a required object might spawn inside an unreachable container). To ensure that all plans in the benchmark are completable, each configuration is validated with a deterministic state machine before inclusion in the test set. The following pseudocode describes the plan generation process: 

FOR each task_type in [cook_egg, make_toast, put_away, ...]: FOR each scene in compatible_scenes(task_type): FOR each variation in [clean, dirty_plate, dirty_pan, ...]: 1. Initialize scene with random seed 2. Remove duplicate objects (keep one per type) 3. Remove known buggy objects 

23 4. Place task-relevant objects on visible surfaces 5. Apply variation conditions (e.g., make Plate dirty) 6. Attempt to complete task using deterministic state machine: - Has perfect knowledge of the scene - Determines valid interaction positions - Checks object compatibility (e.g., slice fits in bowl) 7. IF state machine succeeds: Save configuration to benchmark ELSE: Discard configuration 

This process guarantees that every accepted plan has at least one valid solution path, while the randomized placements introduce meaningful diversity in the required action sequences. The final benchmark test set consists of 108 plans spanning all 12 task types.
