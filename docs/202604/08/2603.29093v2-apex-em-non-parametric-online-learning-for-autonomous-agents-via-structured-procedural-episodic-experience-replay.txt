# APEX-EM: Non-Parametric Online Learning for Autonomous Agents via Structured Procedural-Episodic Experience Replay 

Pratyay Banerjee, Masud Moshtaghi and Ankit Chadha 

Amazon, AGI / Sunnyvale, USA 

pratyay, mmasud, ankitrc@amazon.com 

Abstract 

LLM-based autonomous agents are fundamen-tally stateless at the procedural level: they re-derive solutions from scratch for every task, even when structurally identical tasks have been solved before. We present APEX-EM ,a non-parametric online learning framework that enables agents to iteratively accumulate, retrieve, and reuse structured procedural plans without modifying model weights. APEX-EM introduces three key innovations: (1) a struc-tured experience representation that encodes the complete procedural-episodic trace of each task execution—including planning steps, gen-erated artifacts, iteration history with error anal-ysis, and quality scores; (2) a Plan-Retrieve-Generate-Iterate-Ingest (PRGII) workflow that decomposes task execution into deterministic phases with Task Verifiers providing multi-dimensional reward signals; and (3) a dual-outcome Experience Memory store with hy-brid retrieval combining semantic search, struc-tural signature matching over previous exper-iments, and plan graph traversal—enabling cross-domain transfer of procedural knowledge between tasks that share no lexical overlap but have analogous operational structure. Success-ful experiences serve as positive in-context ex-amples while failed experiences serve as nega-tive examples with structured error annotations. We evaluate on three diverse benchmarks— BigCodeBench (Zhuo et al., 2025) (code gen-eration), KGQAGen-10k (Zhang et al., 2025a) (structured query generation), and Humanity’s Last Exam (Phan et al., 2025) (multi-domain knowledge reasoning)—using Claude Sonnet 4.5 and Claude Opus 4.5 as agent backbones, and compare against MemRL (Wang et al., 2025) and its baselines under identical frozen-backbone conditions. On KGQAGen-10k, APEX-EM achieves 89.6% accuracy (95.3% CSR) compared to 41.3% without memory which is a 48.3 percentage point improvement attributable solely to online learning through accumulated procedural experience, surpass-ing the oracle-retrieval upper bound (GPT-4o w/ SP at 84.9%). On BigCodeBench, APEX-EM reaches 83.3% SR / 84.0% CSR from a 53.9% baseline (+29.4pp), exceeding MemRL’s +11.0pp gain with GPT-4o. On HLE, entity graph retrieval reaches 48.0% from a 25.2% baseline (+22.8pp). Ablation analysis reveals that the value of each component is task-dependent: rich judge feedback provides negli-gible benefit for code generation (A1 ≈A2 on BCB) but is critical for structured query gen-eration (+10.3pp on KGQA), while iteration with binary signal can partially compensate for weaker feedback quality. 

1 Introduction 

Large language model (LLM)-based autonomous agents have demonstrated remarkable capabilities in multi-step reasoning, code generation, and tool use (Yao et al., 2023; Wei et al., 2022). Yet un-like reinforcement learning (RL) agents—which maintain experience replay buffers to store past transitions and replay them during training (Lin, 1992)—LLM-based agents operate without a com-parable mechanism for accumulating and reusing task execution experience. RL agents store struc-tured tuples of (state, action, reward, next_state) and sample from them to stabilize learning. LLM agents, by contrast, are fundamentally stateless at the procedural level : each task invocation triggers the full reasoning pipeline from scratch, even when a near-identical task was successfully completed moments earlier. Recent work has begun to ad-dress this gap through memory mechanisms for LLM agents. Early approaches introduced episodic narrative logs (Shinn et al., 2023), code snippet li-braries (Wang et al., 2023), retrieval-augmented generation over static documents (Lewis et al., 2020), and memory stream architectures (Park et al., 2023). These systems store either unstruc-tured natural language reflections, domain-specific code fragments, or narrative observations—and re-

trieve them via passive semantic similarity, treating all retrieved content as equally useful regardless of its actual effectiveness in prior executions. The most recent advance, MemRL (Wang et al., 2025), directly addresses this passive retrieval limitation by framing memory selection as a value-based de-cision process. MemRL organizes memory into Intent-Experience-Utility triplets, where each ex-perience carries a learned Q-value reflecting its ex-pected utility. A Two-Phase Retrieval mechanism first recalls candidates by embedding similarity, then re-ranks them by a composite score blending similarity with the Q-value. Environmental feed-back updates these Q-values via a Monte Carlo rule, enabling the agent to distinguish high-utility mem-ories from semantically similar noise—all without modifying model weights. MemRL achieves state-of-the-art results on HLE (Phan et al., 2025), Big-CodeBench (Zhuo et al., 2025), ALFWorld, and Lifelong Agent Bench, establishing the strongest frozen-backbone memory baseline to date. De-spite this progress, three fundamental limitations persist in MemRL and other existing memory ap-proaches. They store flat experience representa-tions : either unstructured reflections (Shinn et al., 2023) or LLM-generated text summaries of past trajectories (Wang et al., 2025). These summaries discard the procedural structure of task execution: planning steps, entity and schema discovery, gener-ated artifacts, iteration history, error traces, and the causal chain linking decisions to outcomes. Without this structure, retrieved memories provide narrative guidance rather than replayable proce-dures that an agent can directly adapt. Second, no existing system benefits directly from nega-tive experiences by implementing dual-outcome learning . MemRL assigns each memory a single scalar Q-value that captures aggregate utility but encodes nothing about what specifically succeeded or failed within an execution. Systems like Voy-ager (Wang et al., 2023) store only successes; Re-flexion (Shinn et al., 2023) stores unstructured re-flections without systematic quality evaluation. No system maintains a quality-gated memory where successes serve as positive in-context examples and failures serve as negative examples with structured error annotations—the procedural equivalent of hindsight experience replay (Andrychowicz et al., 2018). Third, all existing systems lack structural transfer across domains . MemRL retrieves by embedding similarity re-ranked by Q-value, but both signals are tied to surface-level task descrip-tions. MemRL’s own evaluation confirms this limi-tation: on HLE, where intra-dataset semantic simi-larity is low, cross-task retrieval provides no gain over single-task reflection (Wang et al., 2025). Pro-cedures learned in one domain (e.g., querying a sports database) cannot transfer to a structurally analogous task in another domain (e.g., querying a financial database) when surface-level descrip-tions diverge. Drawing on the cognitive science distinction between procedural memory (encoding how to perform tasks) and episodic memory (stor-ing records of specific experiences)(Tulving, 1972; Squire, 2004), we observe that what LLM agents lack is precisely a procedural-episodic memory: a store where each entry is simultaneously procedu-ral (encoding a reusable procedure) and episodic (encoding a specific experience with its context and outcome). The successful/failed distinction in such a memory is not a distinction between mem-ory types but a distinction in reward outcome —analogous to how humans refine skills through both positive reinforcement and error avoidance. This framing motivates a system that functions as an experience replay buffer(Lin, 1992) for LLM agents, implementing a form of non-parametric online learning (Shalev-Shwartz, 2012) where the agent’s effective knowledge grows through an exter-nal memory store rather than parameter updates. To address these limitations, we present APEX-EM ,a non-parametric online learning framework for au-tonomous agents that advances beyond MemRL’s scalar-utility retrieval by storing complete struc-tured procedural traces with multi-dimensional quality signals. Our contributions are threefold: 1. We introduce a Procedural Knowledge Graph (PKG) which provides a structured experi-ence representation that encodes the com-plete procedural-episodic trace of a task exe-cution as a replayable unit of experience. Each experience captures planning steps, entity and schema discovery, generated artifacts, itera-tion history with error analysis, quality scores, and a structural signature —an abstract op-eration sequence (Eg. [entity_resolution →

temporal_filter → aggregation]) that enable re-trieval of procedurally analogous plans across surface-dissimilar domains. Where MemRL stores flat text summaries scored by a single Q-value, APEX-EM stores the full causal chain of an execution, enabling agents to replay and adapt complete procedures rather than receive vague experiential hints. 2. We define the PRGII workflow (Plan-Retrieve-Generate-Iterate-Ingest), a struc-tured multi-phase algorithm that governs task execution with Task Verifiers provid-ing multi-dimensional reward signals. Un-like MemRL’s scalar environmental reward (binary success/fail), our verifiers produce structured quality assessments across multiple dimensions—correctness, completeness, effi-ciency, and error categorization—that are pre-served in the experience representation. The workflow implements a non-parametric online learning loop: each execution upon passing a quality gate, is committed to the Experience Memory, directly improving subsequent task performance without model weight updates. 3. We develop a dual-outcome Experience Memory with hybrid retrieval combining se-mantic search over task description embed-dings, structural signature matching over ab-stract operation sequences, and plan graph traversal. Successful experiences (quality 

≥ θ) serve as positive in-context examples; failed experiences (quality < θ ) serve as neg-ative examples with structured error annota-tions. This addresses both MemRL’s lack of explicit negative examples and its embedding-dependent transfer limitation: structural sig-natures enable retrieval of procedurally analo-gous plans across surface-dissimilar domains where embedding-based methods fail. We evaluate on the same benchmarks as MemRL (Wang et al., 2025)— BigCodeBench (Zhuo et al., 2025) and Humanity’s Last Exam (HLE)(Phan et al., 2025)—plus KGQAGen-10k(Zhang et al., 2025a), using Claude Sonnet 4.5 and Claude Opus 4.5 as agent backbones. We compare against MemRL and its full baseline suite (RAG, Self-RAG, Mem0, MemP, Reflexion) under identical frozen-backbone conditions. On HLE—where MemRL’s cross-task retrieval yields no improvement over single-task reflection due to low semantic similarity— Expe-rience Memory’s structural signature matching enables effective cross-task transfer, demonstrating the added benefit of procedural structure compared to surface similarity. 

2 Related Work 

Memory Systems for LLM Agents. The land-scape of LLM agent memory has evolved rapidly. MemGPT (Packer et al., 2023) pioneered OS-inspired memory hierarchies for managing context limitations. Reflexion (Shinn et al., 2023) intro-duced verbal reinforcement learning through natu-ral language reflections stored in episodic buffers. Voyager (Wang et al., 2023) built a skill library of verified code snippets for embodied agents. Gener-ative Agents (Park et al., 2023) demonstrated mem-ory streams with periodic reflection and synthe-sis. More recently, A-MEM (Xu et al., 2025) pro-posed Zettelkasten-inspired agentic memory with autonomous linking, Mem0 (Chhikara et al., 2025) introduced entity-centric relational graphs, and MIRIX (Wang and Chen, 2025) achieved strong performance through six specialized memory stores with multi-agent routing. Zep (Rasmussen et al., 2025) builds temporally-aware knowledge graphs for conversational memory. A comprehensive sur-vey of memory mechanisms is provided by Zhang et al. (2025b). These systems store either unstruc-tured reflections, code fragments, entity relation-ships, or narrative observations—none provides structured procedural traces with dual-outcome in-dexing and cross-domain structural retrieval. 

Experience Replay and Online Learning. Ex-perience replay (Lin, 1992) is foundational in deep RL, storing past transitions for training stability. The APEX-EM is conceptually an experience re-play memory/buffer for LLM agents, but with key differences: (a) experience replay updates weights via gradient descent; APEX-EM operates through in-context learning; (b) replay stores low-level (state, action, reward, next_state) transitions; APEX-EM stores high-level procedural plans; (c) replay samples randomly or by priority; APEX-EM retrieves semantically and structurally; (d) re-play does not distinguish positive from negative examples for different retrieval roles. This posi-tions APEX-EM as a form of non-parametric on-line learning (Shalev-Shwartz, 2012): the agent im-proves during deployment without weight updates, analogous to how a nearest-neighbor classifier im-proves as its reference set grows. The approach is model-agnostic, avoids catastrophic forgetting, and provides full auditability. MemRL (Wang et al., 2025) is the closest prior work—a non-parametric, frozen-backbone system that accumulates expe-rience and improves at runtime. MemRL stores Intent-Experience-Utility triplets with learned Q-values and employs a Two-Phase Retrieval mecha-nism that first recalls candidates by semantic simi-larity, then re-ranks by Q-value. However, MemRL stores LLM-summarized reflections rather than full structured execution traces, uses Q-value-weighted ranking rather than structural signature matching, and implicitly encodes success/failure through Q-values rather than explicit dual-outcome index-ing. Critically, MemRL’s cross-domain transfer de-grades when intra-dataset semantic similarity is low (as demonstrated in their ablation on HLE, where similarity of 0.186 reduced gains to memorization-like behavior), whereas APEX-EM’s structural sig-natures are specifically designed for cross-domain transfer with zero lexical overlap. We evaluate on the same benchmarks as MemRL (BigCodeBench, HLE) plus KGQAGen-10k, enabling direct com-parison. 

Case-Based Reasoning. The CBR cy-cle (Aamodt and Plaza, 1994; Kolodner, 1993)— Retrieve, Reuse, Revise, Retain—bears struc-tural resemblance to PRGII. However, APEX-EM differs in maintaining dual-outcome indexing (CBR typically stores only successes), leveraging LLM-based in-context adaptation rather than hand-crafted transformation rules, using dense semantic embeddings and structural signatures rather than feature-based similarity, and employing quality-gated commitment with multi-criteria evaluation. 

Other Approaches. ALMA (ALMA Authors, 2025) uses meta-learning to discover memory de-signs but does not provide a concrete deployable architecture. SkillRL (SkillRL Authors, 2025) ex-tracts reusable skills via RL weight updates rather than in-context learning. MemSkill (MemSkill Au-thors, 2025) learns memory operations themselves rather than defining structured plan representations. LATS (Zhou et al., 2023) combines LLM agents with Monte Carlo Tree Search but maintains experi-ence only within single episodes without cross-task persistence. 

3 Experience Replay Framework 

3.1 Problem Formulation 

We consider an autonomous agent A operating in an environment E that receives a sequence of tasks 

{t1, t 2, . . . , t n} expressed in natural language. For each task ti, the agent must produce an executable artifact ai (e.g., a database query, code, action se-quence) and obtain a result ri = E(ai). The agent has access to a Experience Memory M that grows over time. The Experience Memory is organized as a Procedural Knowledge Graph (PKG)—a typed graph where nodes are experiences, entities, sub-tasks, and operations, and edges encode their rela-tionships (Section 3.2). The goal is to maximize task success rate as M accumulates experience, without modifying the agent’s underlying model weights θ. Formally, the agent’s policy at task ti is: 

ai = πθ(ti | M (i−1) ) (1) where M(i−1) is the Experience Memory state after processing tasks t1, . . . , t i−1. After executing ti,the memory is updated: 

M(i) = M(i−1) ∪ Plan (ti) (2) where Plan (ti) is the structured procedural-episodic record of the execution. The system exhibits non-parametric online learning : perfor-mance generally improves as |M| grows, with θ

held fixed. 

3.2 Procedural Knowledge Graph 

The Experience Memory M is organized as a typed graph that stores everything the agent has learned and world knowledge together. We call this graph Procedural Knowledge Graph (PKG). In this setup, a change to any entity (e.g., a team win-ning a championship, an API endpoint migrating) instantly propagates to all connected experiences. Every node is self-contained: it carries its own ti-tle, description, and unique identifier, so it can be understood without chasing context elsewhere. 

3.2.1 Node Types 

The graph contains five node types: • Entity : Real-world objects the agent reasons about (a company, a sports team, an API end-point), each with measurable properties such as season wins, three-point percentage, or rev-enue. When a fact changes, only the Entity node is updated; every experience that refer-ences it automatically reflects the change via typed uses_entity edges. • Experience : The core node type. Each Ex-perience contains a complete record of how the agent solved a task, organized into five re-flection layers (Section 3.3): Goal Reflections, Procedure Reflections, Evidence Reflections, Error Registry, and Patch Reflections. Ex-periences are indexed as either successful or failed. • Sub-Task : Captures how complex tasks de-compose into smaller problems. Each sub-problem gets its own Experience node linked to the parent via a subtask_of edge, preserv-ing the full problem-solving hierarchy. • Operation : Abstract procedural steps that form structural signatures—domain-agnostic reasoning primitives (e.g., 

entity_resolution , aggregation ,

temporal_filter ). As the agent solves tasks, the underlying reasoning pattern is extracted as an ordered sequence of Operation nodes. A structural signature is the ordered sequence of Operations for a given experience. Think of Operations as individual moves, and a Procedure Reflection as the full playbook composed of those moves. • TaskTopic : Hub nodes representing task cate-gories, connecting related entities and opera-tions via MEMBER_OF edges.

3.2.3 Entity Resolution 

When the Planning phase extracts potential entity mentions from a new task, each mention is resolved against the PKG via an EntityResolver :1. Embed the mention text m to obtain em.2. Vector search the PKG for candidates of the same node type and domain. 3. If the top candidate exceeds a similarity threshold τr, map to the existing node (dedu-plication). 4. If multiple candidates are close, an LLM dis-ambiguates. 5. Otherwise, create a new node and add it to the PKG. This ensures that structurally equivalent operations across tasks (e.g., group by category” and aggre-gate by type”) resolve to the same Operation node, producing comparable structural signatures even when surface descriptions differ. The resolver uses an LRU cache and batched writes for efficiency. 

3.2.4 Signature Extraction 

Given the resolved entities, properties, and operations, the StructuralSignatureExtractor 

constructs ˆσ as a type-prefixed, ordered se-quence and simultaneously builds the graph edges (FOLLOWED_BY , RELATES_TO , USES , MEMBER_OF ). This makes signature extraction a side effect of graph construction: each task processed enriches the PKG, improving resolution quality for future tasks. 

3.2.5 World-Change Propagation 

Entities are first-class nodes linked to Experiences via typed uses_entity edges, while procedural logic lives in separate Procedure Reflection layers. When a world entity changes (e.g., a team wins a championship, an API endpoint migrates), only the Entity node is versioned—a supersedes edge points to the replacement, and every Experience referencing that entity automatically resolves to the latest version. The procedure for "look up the president’s policy positions" is unchanged; only the entity binding shifts. In flat-text memory systems, every record mentioning the old entity must be found and rewritten, or stale plans silently degrade retrieval.

The six layers are: • Goal Reflection (RG): Captures what the task aimed to achieve . Each piece of evidence is content-addressed (SHA-256 hash) so identical outputs are never du-plicated and provenance is always traceable. Contains the evidence kind (web, tool output, database result, file snapshot), locator (URL, tool name, query, timestamp), content digest, span (DOM range, table cells, code region), and a trust score (source type, authority). • Execution Trace (RT ): The ephemeral record of agent progress, archived on completion. This structured decomposi-tion turns documented failures into queryable guardrails, preventing re-exploration of known dead ends. • Patch Reflection (Rpatch ): Captures specific fixes that modify procedure steps to prevent the same failure from recurring . Each patch contains a trigger signature (symptom pat-terns), a program delta (insert step or replace logic, with target location and new logic), ra-tionale explaining why the patch prevents the failure, and a utility delta measuring impact on reliability and tool cost. The Evaluation layer scores each execution via a Teacher model on correctness c, efficiency η,and completeness κ, producing an overall qual-ity score q = wc · c + wη · η + wκ · κ. The Teacher additionally generates qualitative feedback FT —reasoning about why the approach worked or failed—bridging the structured and unstruc-tured sides of the ontology. For failed executions, Teacher feedback is decomposed into the Error Registry and Patch Reflections, turning unstruc-tured feedback into typed fields that support pro-grammatic querying and pattern detection. Both successful and failed experiences share the same schema. The distinction is the reward outcome 

from the quality gate, not a difference in record type—analogous to how RL replay buffers store transitions regardless of reward sign. Failed experi-ences additionally populate the Error Registry and Patch Reflections with structured error decompo-sition and reusable fixes, providing guardrails for future tasks sharing the same structural pattern. 

3.4 The PRGII Workflow 

The PRGII (Plan-Retrieve-Generate-Iterate-Ingest) workflow is a structured, multi-phase algorithm governing task execution. The phases exe-cute in a fixed sequence, implementing a non-parametric online learning loop. The work-flow is a generic orchestrator : domain spe-cialization is injected through pluggable Do-mainAdapters (entity/property/procedural-step ex-traction), Validators (domain-specific verification), and PromptTemplates (phase-specific prompts per task type). A configuration object toggles individ-ual phases on/off for ablation studies. Domain-specific instantiations are detailed in Appendix A.3. 

3.4.1 Phase 1: Plan 

Given task t, the Planning phase produces a struc-tured decomposition P(t) = ( TU , E, S, ˆσ):• Task Understanding (TU): Intent, con-straints, expected output format, and com-plexity classification, extracted via LLM struc-tured output. • Entity Discovery (E): A set of domain enti-ties referenced in t, resolved against the PKG through the DomainAdapter and EntityRe-solver (Section 3.2). • Schema Discovery (S): Relevant schemas, properties, or API signatures discovered from the environment. • Structural Signature (ˆσ): An abstract oper-ation sequence ˆσ = [ op 1, . . . , op K ] extracted from t, enabling structure-based retrieval (see example in 2).

The Teacher model scores the result on correctness, efficiency, and completeness, and additionally generates qualitative feedback FT —reasoning about approach strengths, weaknesses, and improvement suggestions. For failed execu-tions, the Teacher identifies where reasoning went wrong without providing the answer, preventing answer leakage while teaching process improve-ment. This Teacher signal is then decomposed: the qualitative feedback populates the text fields on the Experience node for in-context learning, while structured error information is extracted into the Error Registry (error class, root cause, recovery steps) and Patch Reflections (program deltas en-coding reusable fixes). This is where unstructured feedback becomes structured: the system extracts what went wrong, why, and how to fix it from the Teacher’s natural language into typed fields that support programmatic querying and pattern detec-tion. Failed experiences additionally populate the Error Registry and Patch Reflections with structured error decomposition and reusable fixes. 

3.5 Experience Memory Store Architecture 

The Experience Memory store M is a persistent, multi-index repository over the PKG, comprising four components: Vector Index. Stores Goal Re-flection embeddings et for approximate nearest neighbor search, enabling semantic similarity re-trieval across Experience nodes. Graph Store. The PKG itself (Section 3.2), a directed typed graph where Experience, Entity, Sub-Task, Operation, and TaskTopic nodes are connected by the ten edge types defined in Section 3.2. Supports entity-graph traversal, structural signature matching, and multi-hop discovery of related experiences. Metadata Index. Supports structured filtering by domain, status, error class, entities, schema elements, qual-ity score, and timestamp. Experience Store. Full Experience records with all reflection layers, exe-cution traces, and compressed representations for prompt inclusion. 

3.6 Cold Start and Bootstrapping 

The Experience Memory begins empty. We ad-dress cold start through: (1) seed plans —a small set (5–20 per domain) of human-authored experi-ences as initial few-shot examples; (2) seed task execution —running PRGII against curated tasks to generate initial experiences; and (3) progressive enrichment —organic growth as the system pro-cesses tasks, creating a virtuous cycle where each committed experience improves future retrieval. The system also supports an offline-to-online learn-ing pipeline (Nair et al., 2020): an offline phase seeds M with high-quality experiences generated by powerful models, followed by an online phase where a smaller model retrieves from the seeded memory and commits new experiences during live operation. 

3.7 Memory Maintenance 

As M grows, maintenance mechanisms ensure re-trieval quality. The typed graph structure enables three categories of maintenance that flat memory systems cannot support: 

Entity Swaps. world knowledge change prop-agate to all connected Experiences automatically resolve to the latest version without modifying pro-cedure templates. 

Continuous Fresh Updates. Every quality-gated execution trace commits to the graph automatically—new Experiences link to existing Entity and Operation nodes, extending coverage without duplicating structure. The EntityResolver ensures structurally equivalent operations resolve to the same node, so a new “compare quarterly earnings” experience reuses existing aggregation and comparison Operation nodes. Failed experi-ences are retained with Error Registry annotations as negative examples. Every quality-gated exe-cution trace commits to the graph automatically extending coverage without duplicating structure. 

Compaction and Cleanup. Each Experience carries a quality score q = c, η, κ . Periodic con-solidation: (a) archives dominated experiences— same structural signature, lower quality—via 

supersedes edges, (b) consolidates similar expe-riences into parameterized procedure templates, and (c) flags experiences referencing stale enti-ties for re-validation. The typed structure makes these judgments mechanical: if Experience p2 has the same signature as p1 with higher quality, p1 is safely archived. Additionally: (4) context window management —configurable top-k limits (default: 3 successful, 2 failed), experience compression for prompt inclusion, and dynamic k adjustment based on task complexity and context budget. 

4 Task Domains and Datasets 

We evaluate APEX-EM on five benchmarks span-ning code generation, structured query generation, multi-domain knowledge reasoning, OS interac-tion, and embodied navigation. Three of these (Big-CodeBench, HLE, and the two additional MemRL benchmarks) are shared with MemRL (Wang et al., 2025), enabling direct comparison; KGQAGen-10k extends evaluation to structured query generation. Table 2 summarizes the benchmarks. 

4.1 Code Generation (BigCodeBench) 

BigCodeBench (Zhuo et al., 2025) is a challeng-ing code generation benchmark comprising 1,140 tasks that require composing multiple library APIs to solve practical programming problems. Un-like simpler function-completion benchmarks, Big-CodeBench tasks demand understanding of library documentation, correct API composition across packages (e.g., pandas , numpy , matplotlib ), and generation of complete, executable Python pro-grams. Each task includes a natural language prompt and a comprehensive test suite; evaluation runs generated code through a sandboxed execution environment ( untrusted_check ). MemRL (Wang et al., 2025) uses BigCodeBench with GPT-4o as the backbone, reporting 48.5% baseline accuracy and 59.5% with MemRL after 10 epochs. APEX-EM’s PRGII workflow instantiates BigCodeBench with code-specific schema discovery (function sig-nature extraction, import analysis, test structure parsing) and sandbox-based execution verifiers.

Questions require multi-hop reasoning (2– 5 hops), temporal filtering, comparative aggrega-tion, and entity resolution across diverse topics including Arts, STEM, Sports, and Geography. 4.3 Multi-Domain Knowledge Reasoning (HLE) 

Humanity’s Last Exam (HLE) (Phan et al., 2025) is a frontier benchmark of expert-level questions spanning mathematics, physics, chemistry, biol-ogy, computer science, engineering, humanities, and social sciences. Questions are designed to be at the boundary of current AI capabilities, re-quiring deep domain expertise, multi-step rea-soning, and often creative problem-solving ap-proaches. MemRL (Wang et al., 2025) uses HLE with Gemini-3-pro as the backbone, reporting 35.7% baseline accuracy and 57.3% with MemRL after 10 epochs—their largest absolute gain, at-tributed to runtime memorization of specific so-lutions through repeated exposure. APEX-EM’s PRGII workflow instantiates HLE with a 7-action iteration space including tool-augmented reason-ing (web search, calculation, decomposition), self-verification loops, and multimodal generation sup-port for image-based questions. 

4.4 OS and Database Interaction (Lifelong Agent Bench) 

Lifelong Agent Bench (Zheng et al., 2025) eval-uates agents in a continuous learning setting in-volving Operating System (OS) and Database (DB) interactions. It tests the agent’s capacity to adapt to new tools and commands over a long horizon without forgetting previous skills. MemRL (Wang et al., 2025) uses Lifelong Agent Bench with GPT-4o-mini as the backbone, reporting 67.4% (OS) and 86.0% (DB) baseline accuracy, reaching 78.8% / 96.0% with MemRL after 10 epochs. The OS task exhibits high intra-dataset similarity, making it particularly amenable to cross-task memory trans-fer. APEX-EM experiments on this benchmark are planned. 

4.5 Embodied Navigation (ALFWorld) 

ALFWorld (Shridhar et al., 2021) is an embod-ied navigation and manipulation benchmark requir-ing agents to solve textual logic puzzles within a simulated household environment (e.g., “put a clean apple in the fridge”). It tests the agent’s abil-ity to learn and retrieve multi-step action plans. MemRL (Wang et al., 2025) uses ALFWorld with GPT-5-mini as the backbone, reporting 77.7% base-line accuracy and 94.9% with MemRL after 10 epochs. The benchmark’s high intra-dataset simi-larity enables strong cross-task transfer. APEX-EM experiments on this benchmark are planned.

5.1 Experimental Setup Models. We evaluate with Claude Sonnet 4.5 as the agent backbone for BigCodeBench and KGQAGen-10k, and Claude Opus 4.5 for HLE. Entity extrac-tion uses Claude Sonnet 4.6. Baselines. Following Wang et al. (2025), we compare against: (1) No Memory —the agent op-erates without any persistent memory; (2) RAG —standard semantic retrieval of past experiences; (3)

Metrics. We report Last Epoch Success Rate (SR) and Cumulative Success Rate (CSR)—the proportion of tasks solved at least once across all epochs—following Wang et al. (2025). 5.3 Runtime Learning Dynamics 

Figure 3 illustrates the online learning effect, anal-ogous to the epoch-over-epoch improvement re-ported by MemRL (Wang et al., 2025). With an empty APEX-EM, the agent achieves baseline ac-curacy. As experiences accumulate, accuracy im-proves monotonically, with the steepest gains in the first 50–100 experiences per domain. Performance plateaus around 200–300 experiences, suggesting diminishing returns as the memory saturates com-mon task patterns. This mirrors MemRL’s observa-tion that CSR grows fastest in early epochs before plateauing, though APEX-EM’s structural signa-tures may enable continued gains in cross-domain settings where MemRL’s semantic similarity de-grades.

No new learning occurs during transfer—the agent can only retrieve from frozen memory. 6 Conclusion 

We have presented APEX-EM, a non-parametric online learning framework that enables LLM-based agents to accumulate, retrieve, and reuse structured procedural experience without model weight modification. Through the PRGII workflow, dual-outcome indexing, and hybrid retrieval with structural signature matching, APEX-EM achieves 89.6% accuracy / 95.3% CSR on KGQAGen-10k at epoch 5—surpassing both the no-memory base-line (41.3%) and the oracle-retrieval upper bound (GPT-4o w/ SP at 84.9%)—83.3% SR / 84.0% CSR on BigCodeBench at epoch 10 (+29.4pp over base-line), and 48.0% on HLE at epoch 10 (+22.8pp over baseline). A key finding is that component contri-butions are task-dependent: rich judge feedback is negligible for code generation but critical for struc-tured query generation (+10.3pp on KGQA), while iteration with binary signal can partially compen-sate for weaker feedback. We evaluate on three benchmarks—BigCodeBench (code generation), KGQAGen-10k (structured query generation), and Humanity’s Last Exam (multi-domain knowledge reasoning)—using Claude Sonnet 4.5 and Claude Opus 4.5 as agent backbones, with direct compar-ison against MemRL (Wang et al., 2025) and its baselines on the two shared benchmarks. Key directions for future work include: (1) com-pleting evaluation with full results across all three benchmarks; (2) investigating learned retrieval poli-cies that optimize the composite ranking function, potentially combining APEX-EM’s structural sig-natures with MemRL’s Q-value-based utility esti-mation; (3) exploring plan compression techniques that preserve procedural fidelity while reducing context window consumption; and (4) evaluating on MemRL’s remaining benchmarks (ALFWorld, Lifelong Agent Bench) to assess APEX-EM’s ef-fectiveness in embodied and multi-step interactive environments. 

Limitations 

The current evaluation reports results at 10 epochs for BCB and HLE (all ablations complete), and epochs 4–7 for KGQA (experiments in progress). While we compare against MemRL’s published re-sults on BigCodeBench and HLE, the backbone models differ (we use Claude Sonnet 4.5 / Opus 4.5 vs. MemRL’s GPT-4o / Gemini-3-pro), which complicates direct numerical comparison; the com-parison is most meaningful at the level of rela-tive improvement over each system’s respective no-memory baseline. We do not yet evaluate on MemRL’s ALFWorld and Lifelong Agent Bench benchmarks, where MemRL’s trajectory verifica-tion mechanism shows its largest gains. The qual-ity gate relies on either a Judge LLM (which may introduce evaluation noise) or human evaluators (which limits scalability). The structural signature taxonomy is currently hand-designed; automated signature discovery from execution traces could improve coverage. Finally, the system does not pro-vide formal convergence guarantees—the “online learning” characterization reflects the operational property of improvement through experience ac-cumulation rather than theoretical regret bounds, unlike MemRL which provides Bellman contrac-tion guarantees for Q-value convergence. 

Ethical Considerations 

The APEX-EM architecture stores complete exe-cution traces, which may contain sensitive infor-mation from the task environment. Deployments must implement appropriate access controls and data retention policies. The dual-outcome indexing of failed plans requires careful handling to prevent storage of plans that failed due to privacy violations or harmful outputs. The offline-to-online pipeline, where plans generated by powerful models guide smaller models, raises questions about knowledge provenance and accountability that should be ad-dressed in deployment settings. Throughout the development of this work, gen-erative AI systems were employed for language refinement and code test case generation. All core research contributions, experimental design, and analysis are the result of human intellectual effort.

A.1 Generic Orchestrator 

The PRGIIWorkflow class orchestrates the five phases via dependency injection. It accepts a 

PRGIIConfig (phase toggles and parameters) and five phase objects: 

class PRGIIWorkflow: config: PRGIIConfig {toggles, max_iter, θ}planning: PlanningPhase retrieval: RetrievalPhase generation: GenerationPhase iteration: IterationPhase ingest: IngestPhase embedding: EmbeddingService The execute() method runs the phases in se-quence: Plan → Retrieve → [Generate ↔ Iterate] J

A.7 Two Mechanisms of Self-Improvement 

PRGII improves over single-shot generation through two independent mechanisms, illustrated by BCB tasks from early experiments: 

Mechanism 1: Cross-Epoch Memory Retrieval. 

BCB task 15 (execute shell commands from CSV, save outputs to files): the single-shot LLM gen-erates reasonable code but names output files 

command_{index}_output.txt . The oracle test suite expects a different naming convention that cannot be inferred from the task description alone. In epoch 2, the failed experience from epoch 1 is re-trieved as a negative example with the oracle’s error message, and the agent reverse-engineers the cor-rect naming convention from the test output. This is the classic memory-augmented learning pattern— the agent learns from past failures on the same task.

A.8 Iteration Reduction Over Epochs 

As memory accumulates, the agent requires fewer self-correction iterations to solve tasks. This sec-tion provides detailed analysis of the iteration dis-tribution shift underlying the aggregate statistics in Section 5.7. 

FAIL →PASS Flips with Iteration Savings. The most striking evidence of memory-driven improve-ment comes from tasks that were unsolvable in E1 but solved by E10, often on the first attempt. On BCB (EG2), 209 tasks flipped from FAIL to PASS across epochs. Many of these exhausted all 3 iter-ations in E1 (quality ∼0.05) but were solved in a single iteration by E10 (quality ∼0.99)—a pattern indicating that memory provides sufficient context for correct first-attempt generation. On KGQA (A3), 343 tasks flipped FAIL →PASS, with even more dramatic iteration savings: tasks that burned through all 10 iterations in E1 (score ∼0.16) were solved in 1 iteration by E10 (score ∼0.95), repre-senting 9 fewer LLM inference calls per task. 

Iteration Efficiency for Already-Solvable Tasks. 

Even for tasks the agent could solve in E1, mem-ory makes it solve them more efficiently. On BCB (EG2), 104 tasks that passed in both E1 and E10 showed reduced iteration counts—many dropping from 3 iterations (the maximum) to 1. These tasks required two rounds of self-correction in E1 but were solved on the first attempt by E10, represent-ing a 2/3 reduction in LLM inference cost per task. This demonstrates that memory benefits extend be-yond expanding the set of solvable tasks to reduc-ing the cost of solving already-solvable ones.

Cross-Task Transfer Examples. The transfer experiments (Section 5.6) reveal that memory gen-eralizes beyond the specific tasks it was trained on. On KGQA, iteration-enabled configs transfer at +29–32pp over baseline to 1,079 unseen questions, while non-iteration configs show near-zero transfer. This binary separation occurs because SPARQL patterns learned from training questions (e.g., how to traverse Wikidata property chains, which prop-erties to use for specific relationship types) are directly applicable to new questions about different entities but similar relationship structures. How-ever, without the self-correction loop to adapt re-trieved patterns to new entity structures, the re-trieved patterns alone are insufficient—the agent needs iteration to bridge the gap between a struc-turally similar retrieved example and the specific entities in the new question.
