## 摘要
手动优化 GPU 算子是一项具有挑战性且耗时的任务。随着大语言模型（LLMs）的快速发展，自动化的 GPU 算子优化正逐渐成为现实。然而，目前的 LLM 驱动的自动化优化方法主要局限于机器学习应用（如 PyTorch 算子优化），而忽视了更广泛的领域，如科学计算中的稀疏矩阵运算。扩展到这些更广泛的应用为基准测试和算法带来了新的挑战。因此，开发一种通用的自动化算子优化方法成为我们的研究重点。在本文中，我们通过引入 MSKernelBench 解决了多场景设置下缺乏系统评估的问题。该基准测试涵盖了多个场景，包括基础代数运算、常见的 LLM 算子、稀疏矩阵算子和科学计算例程，且均支持 FP32 和 BF16 精度。基于该基准测试，我们提出了 CUDAMaster，这是一个多智能体、硬件感知的算子优化系统，它利用性能分析信息并自动构建完整的编译与执行工具链。实验结果表明，CUDAMaster 在大多数算子上实现了显著的加速，性能优于 Astra 约 35%。在某些情况下，其性能甚至达到或超过了高度优化的闭源库（如 cuBLAS）。展示每个算子原始代码和优化代码的 Demo 可在 https://hanyx2021.github.io/MSKernelBenchDemo/ 获取。

## Abstract
Optimizing GPU kernels manually is a challenging and time-consuming task. With the rapid development of LLMs, automated GPU kernel optimization is gradually becoming a tangible reality. However, current LLM-driven automated optimization methods narrowly focus on machine learning applications, such as PyTorch operator optimization, while overlooking broader domains like sparse matrix operations in scientific computing. Extending to these broader applications brings new challenges for the benchmark and algorithm. Therefore, developing a general-purpose automated kernel optimization method becomes our primary focus. In this paper, we address the absence of systematic evaluation for multi-scenario settings by introducing MSKernelBench, which spans multiple scenarios, including fundamental algebraic operations, common LLM kernels, sparse matrix operators, and scientific computing routines, each supporting both FP32 and BF16 precision. Building on this benchmark, we introduce CUDAMaster, a multi-agent, hardware-aware system for kernel optimization that leverages profiling information and automatically constructs the full compilation and execution toolchain. Experimental results demonstrate that CUDAMaster achieves significant speedups across most operators, outperforming Astra by about 35%. In several cases, its performance matches or surpasses that of highly optimized, closed-source libraries such as cuBLAS. A demo showcasing the original and optimized code for each operator is available at https://hanyx2021.github.io/MSKernelBenchDemo/.

### 1. 核心问题与整体含义
*   **研究动机**：手动优化 GPU 内核（CUDA Kernel）极度依赖专家经验且耗时费力。虽然 LLM 在代码生成领域表现出色，但现有研究（如 KernelBench）主要集中在深度学习算子（如 PyTorch 算子），忽视了科学计算、稀疏矩阵运算等更广泛的高性能计算（HPC）领域。
*   **核心挑战**：非 AI 领域的算子通常具有不规则的内存访问模式和复杂的并行逻辑，且缺乏像深度学习那样公开透明的优化路径，这对 LLM 的泛化能力和硬件感知能力提出了更高要求。
*   **整体含义**：本文旨在打破 LLM 仅能优化简单 AI 算子的局限，通过构建多场景基准测试集和闭环优化系统，探索 LLM 在通用 GPU 编程任务中达到专家级水平的可能性。

### 2. 方法论
论文提出了 **CUDAMaster** 系统，这是一个多智能体、硬件感知的端到端优化框架，其核心思想是“性能分析驱动的迭代优化”。
*   **硬件分析过滤器（Hardware Analysis Filter）**：
    *   利用 NVIDIA Nsight Compute 收集细粒度的硬件指标。
    *   **瓶颈分类**：使用 **Otsu 算法**（大津法）对吞吐量分布进行自动阈值计算，将算子客观地分类为：计算受限（Compute Bound）、内存延迟受限（Memory Latency Bound）和内存带宽受限（Memory Bandwidth Bound）。
    *   **指标过滤**：根据瓶颈类型，仅提取最相关的指标（如 L2 命中率、SM 利用率等）反馈给 LLM，减少冗余信息干扰。
*   **多智能体协作系统**：
    *   **Planner Agent**：分析过滤后的硬件报告，制定高层优化策略（如循环展开、共享内存分块）。
    *   **Coder Agent**：根据策略编写 CUDA 代码，并自动生成配套的 C 语言包装器。
    *   **Compiler Agent**：管理编译命令和执行脚本，处理复杂的库链接和架构参数。
    *   **Debug Agent**：当代码运行出错或结果不一致时，结合错误日志进行针对性修复。
*   **算法流程**：系统运行 $R$ 轮迭代，每轮包含“规划-编码-编译-执行-反馈”闭环，若结果错误则触发最多 $D$ 轮调试子循环。

### 3. 实验设计
*   **基准测试集 MSKernelBench**：
    *   包含 **50 个任务**，涵盖 9 大类别：稠密线性代数、稀疏矩阵（COO/CSR/CSC/ELL 格式）、归一化、激活函数、损失函数、LLM 算子序列、张量处理、模板计算（Stencil）、数学算法（排序、FFT、前缀和）。
    *   **双精度支持**：每个算子均提供 FP32 和 BF16 版本。
    *   **多尺度评估**：在不同数据规模下测试，确保优化的稳定性。
*   **对比方法**：
    *   **工业级闭源库**：cuBLAS、cuDNN、cuSPARSE。
    *   **学术界 SOTA**：Astra（针对 SGLang 算子的优化框架）。
*   **评估指标**：编译成功率、执行正确性、**复杂度加权加速比**（根据算子理论复杂度 $O(N)$ 分配权重，使大规模数据的性能表现占更高权重）。

### 4. 资源与算力
*   **硬件环境**：所有内核编译和性能分析均在 **NVIDIA RTX 4090 GPU** 上完成。
*   **模型使用**：使用了 **OpenAI o4-mini** 和 **DeepSeek-V3.2** 作为底层 LLM 驱动智能体。
*   **实验参数**：设定迭代轮数 $R=3$，调试轮数 $D=3$。
*   **训练时长**：论文未提及模型训练，因为该方法基于现有的预训练模型进行推理和迭代优化。

### 5. 实验数量与充分性
*   **实验规模**：共进行了 **100 个优化任务**（50 个算子 × 2 种精度）。
*   **消融实验**：
    *   测试了不同迭代轮数和调试轮数对成功率的影响。
    *   对比了“无性能分析”、“全量性能分析”与“过滤性能分析”的效果，验证了过滤策略在降低 Token 成本（减少 30-40%）的同时保持了高性能。
*   **客观性与公平性**：实验采用了统一的测试框架，基准线（Baseline）均经过严格验证，且对比了行业标准的闭源库，评估过程较为客观。

### 6. 主要结论与发现
*   **性能卓越**：CUDAMaster 在多数算子上实现了显著加速，整体性能比 Astra 提升了约 **35%**。
*   **比肩专家**：在稀疏矩阵向量乘法（SpMV）、2D 卷积和点积运算中，生成的代码性能**达到甚至超过了 cuSPARSE、cuDNN 和 cuBLAS**。
*   **模型差异**：OpenAI o4-mini 在生成成功率和优化深度上普遍优于 DeepSeek-V3.2。
*   **瓶颈转化**：优化过程成功将大量“内存延迟受限”的任务转化为“内存带宽受限”或“计算受限”，显著提升了硬件利用率。

### 7. 优点
*   **场景广泛性**：突破了 AI 算子的范畴，首次系统性地评估了 LLM 在科学计算和稀疏计算中的 CUDA 优化能力。
*   **硬件反馈闭环**：引入了基于 Otsu 算法的自动瓶颈分类，使 LLM 能够像人类专家一样根据 Profiler 数据进行决策。
*   **复杂度加权指标**：提出的评估指标更符合高性能计算的实际需求，即更关注大规模计算下的加速效果。
*   **端到端自动化**：不仅生成内核，还自动处理编译工具链和调试，实用性强。

### 8. 不足与局限
*   **硬件覆盖有限**：实验仅在 RTX 4090 上进行，未测试在 H100 等数据中心级 GPU 或国产算力平台上的表现。
*   **算法复杂度上限**：对于极高复杂度的算法（如 FFT 或大规模排序），LLM 仍难以完全通过自动调优达到极致性能，可能需要更深层的算法重构。
*   **依赖闭源模型**：系统高度依赖高性能的闭源 LLM API，对于离线环境或开源小模型的适配性有待验证。
*   **多卡支持缺失**：目前仅针对单内核优化，未涉及多 GPU 并行或分布式通信算子的优化。
