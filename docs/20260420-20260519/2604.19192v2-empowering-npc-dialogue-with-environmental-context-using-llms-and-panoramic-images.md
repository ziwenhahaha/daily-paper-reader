## Abstract
We present an approach for enhancing non-playable characters (NPCs) in games by combining large language models (LLMs) with computer vision to provide contextual awareness of their surroundings. Conventional NPCs typically rely on pre-scripted dialogue and lack spatial understanding, which limits their responsiveness to player actions and reduces overall immersion. Our method addresses these limitations by capturing panoramic images of an NPC's environment and applying semantic segmentation to identify objects and their spatial positions. The extracted information is used to generate a structured JSON representation of the environment, combining object locations derived from segmentation with additional scene graph data within the NPC's bounding sphere, encoded as directional vectors. This representation is provided as input to the LLM, enabling NPCs to incorporate spatial knowledge into player interactions. As a result, NPCs can dynamically reference nearby objects, landmarks, and environmental features, leading to more believable and engaging gameplay. We describe the technical implementation of the system and evaluate it in two stages. First, an expert interview was conducted to gather feedback and identify areas for improvement. After integrating these refinements, a user study was performed, showing that participants preferred the context-aware NPCs over a non-context-aware baseline, confirming the effectiveness of the proposed approach.

### 1. 核心问题与整体含义（研究动机与背景）

- **问题核心**：传统游戏 NPC（非玩家角色）依赖预脚本对话树，缺乏对周围环境的感知与空间理解能力，导致对话僵硬、难以响应玩家动态行为，降低沉浸感。
- **研究空白**：现有 NPC 对话系统虽有自适应行为研究，但缺少将**实时环境感知**融入对话推理的方案；LLM 在游戏中的应用多聚焦于对话生成，较少与视觉感知结合。
- **解决思路**：利用全景图像 + 计算机视觉（语义分割）提取 NPC 周围环境信息，结构化为 LLM 可理解的输入，使 NPC 能动态引用附近物体、地标与空间关系。

---

### 2. 方法论：核心思想、关键技术细节与流程

#### 2.1 核心思想

将 NPC 的"环境感知"转化为结构化 JSON 描述，与预设的 Supporting Prompt 一同输入 LLM，使 NPC 在回答玩家询问时能基于真实的场景物体生成沉浸式描述。

#### 2.2 技术流程（如图 1 系统结构）

1. **全景图像采集**：在 NPC 视线高度拍摄四张 90° 图像（前后左右），合成为 360° 全景图。引擎使用 Unreal Engine 5（UE5）。
2. **语义对象分割（RAM++）**：
   - 使用修改后的 Recognize Anything Model（RAM++）对四张独立图像进行开放集图像标注。
   - 优势：相比连续全景图，模型在有限视场图像上表现更佳；自动按方向分组输出（left、right、in-front、behind）。
   - 输出格式示例：
     ```json
     {"left": ["cabinet", "pottery", ...], "right": ["altar", "candle", ...], ...}
     ```
3. **径向对象选择（Radial Object Selection）**：
   - 在 UE5 中使用包围球（半径 10 米），碰撞检测返回球内所有物体。
   - 获取物体名称（需规范命名）+ NPC 指向物体的方向向量。
   - 输出示例：
     ```
     Simple_Shelf2, VEC:X=-0.940 Y=-0.340 Z=0.000
     ```
4. **输入组合阶段**：将语义分割 JSON + 径向选择数据 + Supporting Prompt 组合为完整的上下文提示，发往 LLM（如 GPT-4 API）。
5. **视角转换规则**：由于玩家面向 NPC，方向向量需翻转（如 NPC 左侧 → 玩家右侧），由 Supporting Prompt 中内置的转换指南约束。
6. **对话交互**：玩家通过聊天界面输入问题，LLM 结合上下文生成回复，历史记录在同一实例内维护。

#### 2.3 关键技术选型理由

- **RAM++ vs. SAM**：RAM++ 支持开放集识别，无需针对特定游戏资产重新训练，推理高效，适合实时游戏场景。
- **四张独立图像 > 全景图**：模型训练数据为常规 FOV 图像，直接处理全景图性能下降，分张处理并附加方向标签可保证准确性。

### 6. 主要结论与发现

- **系统有效性**：用户研究中，完整系统（Answer 1）获得压倒性偏好，最高 52/58 人选择（Scene 1 Q2），最低 42/58（Scene 2 Q2 & Q4）。
- **评分差异**：Answer 1 平均评分仅比 Answer 2 高约 4%，差异不悬殊但偏好明显。
- **专家反馈**：完整系统输出最具沉浸感，对象位置描述准确，但缺少物体间精细关系（如深度量化、相对位置）和详细属性描述。
- **幻觉问题**：仅使用 Supporting Prompt 时（Test 3），LLM 大量虚构环境细节，几乎不适用于有视觉画面的游戏。
- **径向选择的局限**：依赖规范命名规范；若物体名称无意义，该通道价值有限。

### 7. 优点

1. **端到端流程完整**：从全景图采集 → 语义分割 → 场景图提取 → Prompt 组合 → LLM 生成，全链路清晰可复现。
2. **模块化解耦**：语义分割与径向选择可独立使用或组合，适配不同游戏项目的命名规范情况。
3. **视角转换逻辑**：巧妙通过 Prompt 指令处理 NPC 与玩家视角不一致问题，无需额外模型。
4. **工程落地友好**：使用 UE5 蓝图实现，依赖预训练模型（RAM++、GPT-4），无需大规模本地训练。
5. **评估严谨**：专家访谈 + 用户研究双阶段验证，量化偏好数据支撑结论。
6. **开放集识别**：RAM++ 不依赖游戏特定资产标签，可迁移至新场景。

### 8. 不足与局限

1. **算力信息不透明**：未披露本地硬件配置，外部 API 调用不可控，难以评估部署成本。
2. **场景泛化性不足**：仅 2 个场景测试，室内外各一，无法全面评估复杂或非典型环境的鲁棒性。

4. **长对话未验证**：实验仅涉及 1–4 轮短对话，未评估多轮交互中上下文累积、漂移或冲突问题。
5. **物体关系缺失**：系统仅输出方向（前后左右），缺少精确距离、物体间遮挡/包含等空间关系信息。
6. **命名依赖**：径向选择通道依赖游戏资产的规范命名，游戏工业化流程配套要求较高。
7. **实时性挑战**：全景图采集 → 分割 → LLM 推理的完整链路延迟未公开，在 AAA 级游戏中是否满足实时要求存疑。
8. **对比方法单一**：未与现有商用 NPC 系统或基于规则的环境感知基线做横向对比。
