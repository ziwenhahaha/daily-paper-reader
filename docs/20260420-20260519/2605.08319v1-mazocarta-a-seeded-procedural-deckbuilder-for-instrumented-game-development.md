## Abstract
Mazocarta is a seeded procedural tactical deckbuilder implemented in Rust, compiled to WebAssembly for browser play, and executable natively for simulation. Its primary technical contribution is not the invention of a new deckbuilding genre, but the construction of an instrumented game-development reference artifact: the same rules engine supports interactive play, native command-line simulation, automated end-to-end tests, save/load fixtures, and local-area multiplayer. This paper describes Mazocarta's architecture, deterministic run model, reproducible balance probes, and QR-mediated WebRTC pairing for local multiplayer. An evaluation snapshot over 1,000 deterministic seeds shows that the simulation pipeline can produce reproducible development signals. In the evaluated configuration, single-player and two-player autoplay win rates were 36.1% and 34.9% over 1,000 deterministic seeds, respectively. These rates are not presented as final player-facing balance metrics, but as repeatable probes for future balance shifts and regressions. Mazocarta is positioned as a playable open-source reference artifact for instrumented game development: deterministic regression checks, automated playtesting workflows, balance probes for game mechanics, and browser-native local multiplayer all exercise one shared production rules core.

## 1. 核心问题与整体含义（研究动机和背景）

- **核心问题**：传统游戏开发中，平衡调整、回归检测和自动化测试往往依赖于独立于生产代码的简化模型或外部工具，导致测量对象与实际游戏之间存在不一致。此外，套牌构建类游戏（deckbuilder）具有复杂的组合交互（卡牌协同、地图生成、敌人行为、长线经济决策），仅靠人工测试难以发现所有潜在问题。

- **背景**：现代游戏开发对自动化 playtesting、确定性回归检查和可重复的平衡探测有强烈需求。现有的研究多依赖外部 agent、遥测数据或 UI 级封装，缺乏对同一生产规则核心的端到端覆盖。

- **研究定位**：Mazocarta 并非要发明一种新的卡牌构建玩法，而是要构建一个**仪器化游戏开发参考 artifact**——让交互式游戏、命令行仿真、自动化测试、保存/加载 fixture 和本地多人同步全部共用同一个生产级规则引擎。

## 2. 方法论：核心思想、关键技术细节、算法流程

### 2.1 核心思想

- **一次实现，多端复用**：所有规则逻辑在 Rust 中实现一次，同时支持浏览器交互（WebAssembly）、本地命令行仿真（native actor）以及其他测试表面。
- **确定性（Determinism）**：战斗、卡组顺序、地图生成、奖励、事件和敌人行为均基于显式种子（seed）可完整复现。
- **仪器化即产品**：将自动化视为一等公民（first-class product surface），而非事后补加的测试脚本。

### 2.2 关键技术细节

- **规则核心**：Rust 实现，统一驱动浏览器客户端、WASM 渲染层和本地 actor。游戏机制围绕三个有符号轴（signed axes）构建：
  - **Focus** → 影响输出伤害
  - **Rhythm** → 影响护盾获取
  - **Momentum** → 影响能量与敌人意图输出
  - 这些轴之间的关系由内容定义（content-defined），而非硬编码，使得下游开发者可以在不改变规则表面的前提下调整卡牌、敌人、奖励和模块。

- **仿真 Actor**：手写的启发式策略，执行以下评分优先级：胜利 → 防御生存 → 免费出牌 → 准备动作 → 正向交换。该 actor 旨在作为确定性基线而非最优玩家模型，能够快速检测明显难度偏移或回归。

- **WebRTC 本地多人配对**：
  - 使用 **QR 码 / 手动粘贴**作为信令路径，替代中心化的匹配服务器。
  - 连接后，共享地下城状态（地图结构、当前节点、已访路径等）与每英雄独立状态（牌组、血量、credits）分离传输。
  - 大型 payload 可压缩并切分为动画传输帧。

- **验证表面**：单元测试、保存/加载 fixture、Playwright E2E 浏览器测试、QR 相机伪路径测试、多人 soak 运行，覆盖完整的确定性回放与回归检测能力。

### 2.3 架构流程

> Reproducible Inputs → **One Rust Rules Core** → Browser Play + Native Actor + Verification Harness + WebRTC Multiplayer → Development Signals (Balance / Debug / Regression)

## 7. 优点（方法或实验设计亮点）

1. **架构统一性**：浏览器交互、本地仿真、自动化测试和多人同步共用同一个 Rust 规则核心，消除模型-游戏不一致风险。
2. **确定性回放能力**：固定种子 + 序列化状态，可精确定位回归源头，对调试极其友好。
3. **多层次验证表面**：单元测试、E2E（Playwright）、save/load fixture、soak smoke 测试逐级覆盖，从细粒度规则到完整流程均有保障。

5. **可比较的基线**：固定种子 × 固定策略生成的可复现胜率，使得后续代码改动（规则调整、内容更新）可立即对比，支撑快速迭代。
6. **内容可扩展性**：三个有符号轴的 content-defined 设计，允许在不改动规则表面的情况下重新平衡卡牌、敌人和模块。
7. **开源可复现**：MIT 许可，代码公开，可作为教学或进一步研究的 artifact。
