## Abstract
Large Language Models (LLMs) have shown strong potential for narrative generation, but their use in complex, multi-layered role-playing game (RPG) worlds is still limited by issues of coherence, controllability, and structural consistency. This paper explores a dependency-aware, multi-stage prompt pipeline for procedural RPG content generation that models narrative dependencies through structured intermediate representations. The approach decomposes generation into sequential stages: world building, non-player character creation, player character creation, campaign-level quest planning, and quest expansion. Each stage conditions on structured JSON outputs from previous stages. By enforcing schemas and explicit data flow, the pipeline reduces narrative drift, limits hallucinations, and supports scalable creation of interconnected narrative elements. The system is evaluated qualitatively through human-centered analysis across multiple independent runs. Outputs are assessed using criteria such as structural completeness, internal consistency, narrative coherence, diversity, and actionability. Results show that the pipeline consistently generates logically sound and structurally valid RPG content, without quality degradation as complexity increases. Separating high-level campaign planning from detailed quest expansion improves both global structure and local storytelling. These findings suggest that dependency-aware prompt pipelines with structured intermediate representations are an effective design pattern for LLM-based procedural content generation. This approach may also generalize to other domains requiring sequential reasoning over evolving contextual states.

## 2. 方法论：核心思想、关键技术细节与流程

### 核心思想

提出一种**依赖感知的多阶段提示管道**（Dependency-Driven Prompt Pipeline），将 RPG 内容生成分解为顺序执行的多阶段过程，每阶段显式基于前序阶段的结构化 JSON 输出进行条件化，从而在生成过程中强制保留叙事依赖关系。

### 关键技术细节

| 组件 | 说明 |
|------|------|
| **管道控制器（Pipeline Controller）** | Python 编排模块，负责按预定义顺序执行阶段、强制依赖关系、管理数据流 |
| **大语言模型（LLM）** | 通过 OpenAI API 访问 GPT-5，零样本设置，无需微调 |
| **提示模板（Prompt Templates）** | 参数化提示定义，嵌入前序阶段的结构化上下文，约束输出为预定义 JSON 模式 |
| **结构化数据存储（Structured Data Store）** | 持久化中间和最终 JSON 工件，支持复用、检查和验证 |
| **JSON 解析与验证** | 验证模型输出是否符合预定义模板；早期阶段解析失败则管道停摆，保留原始文本供人工检查 |

### 管道流程（五个阶段）

```
1. 世界构建（World Generation）
   └── 输出：城市、周边地点、关键建筑、政治/派系结构
   
2. NPC 生成（NPC Generation）
   └── 输入：世界 JSON
   └── 输出：多个角色（含身份、人格、技能、动机、秘密、社会关系）
   
3. 玩家角色生成（Player Character Generation）
   └── 输入：世界 JSON + 部分 NPC
   └── 输出：主角结构化表示（含关系绑定）
   
4. 战役级任务生成（Quest Generation）
   └── 输入：世界 + 玩家 + 所有 NPC
   └── 输出：多个关联任务（含标识、描述、目标、任务给予者、奖励、任务间依赖）
   
5. 扩展任务生成（Extended Quest Generation）
   └── 输入：完整上下文 + 具体任务
   └── 输出：详细、多步骤叙事体验（含对话、玩家决策点、后果）
```

### 关键设计原则

- **显式依赖管理**：每阶段将所有前序 JSON 序列化为上下文注入，而非依赖隐式自然语言编码。
- **结构化约束**：所有输出强制符合预定义 JSON 模式，减少歧义、支持确定性解析。
- **两层次任务设计**：战役层规划（全局叙事骨架）与任务层展开（局部细节）分离，平衡全局一致性与局部深度。
- **零微调策略**：所有行为控制通过提示设计、结构化条件和输出约束实现。

### 核心发现

1. **无质量退化**：尽管叙事复杂性递增，后续阶段未见显著质量下降，表明顺序条件化策略有效保留了上下文信息。
2. **叙事一致性得以维持**：任务和扩展任务均未出现叙事崩溃或不一致，与单提示/平面生成方法中常见的问题形成对比。
3. **两层次任务设计有效**：将战役层规划与任务层展开分离，支持全局叙事秩序与详细本地叙事的平衡（战役层连贯性 4.40，扩展任务连贯性 4.20，可操作性高达 4.60）。
4. **结构化表示的优势**：JSON 模式强制约束减少了幻觉，支持确定性解析和跨阶段复用。
5. **管道可扩展性**：多阶段设计缓解了 token 限制约束，每阶段在可管理的上下文大小内生成更丰富的结构化输出。

## 7. 优点

### 方法设计亮点

1. **显式依赖建模**：将 RPG 生成从隐式自然语言上下文提升为结构化数据依赖，确保跨层级一致性。
2. **两层次任务规划**：分离战役级（高层骨架）与任务级（局部展开），兼顾全局连贯与局部深度。
3. **零微调策略**：仅通过提示工程和结构化约束实现行为控制，降低落地门槛，便于复现和迁移。
4. **模块化与可扩展**：各阶段为独立模块，可灵活增减、替换或独立测试；JSON 工件作为"胶水语言"便于与游戏引擎集成。
5. **应用驱动评估**：聚焦于 RPG 设计的实际需求（可操作性、结构完整性），而非通用文本指标。

### 实验亮点

1. **多次独立运行**：20 次完整管道执行评估了系统在重复条件下的鲁棒性，而非仅依赖单次生成实例。
2. **全类别覆盖**：评估覆盖从世界到扩展任务的所有内容类型，保留依赖关系完整性。
3. **定性定量结合**：Likert 量表评分与定性笔记结合，便于识别模式和失败模式。

## 8. 不足与局限

### 实验覆盖与设计局限

- **缺乏基准对比**：未与单提示、平面管道或其他方法在同一框架下对比，难以量化依赖感知的实际增益幅度。
- **无自动化指标**：未采用 BLEU、ROUGE 等标准文本评估指标（虽作者论证了其局限性），限制了大规模可比性。
- **Token 预算固定**：统一 32,768 token 可能导致不同阶段的有效信息密度差异（世界/任务信息密集 vs. 角色信息稀疏）。

### 偏差风险

- **评估者同源性**：4 名评估者均参与系统开发，虽能深入理解设计意图，但可能高估系统表现；未进行正式 inter-rater reliability 分析。
- **无外部验证**：缺少独立游戏设计师或真实玩家的反馈，评估可能偏离实际使用需求。

### 应用限制

- **模型依赖**：实验使用 GPT-5（特定版本），结论可能对模型架构、训练数据或上下文处理差异敏感。
- **JSON 模式约束**：严格模式虽减少幻觉，但可能限制创造性表达的上限。
- **领域泛化未验证**：方法在 RPG 叙事生成中表现良好，但未探索在开放式互动故事、模拟设计或教育内容等其他领域的效果。
- **解析失败处理**：最终扩展任务阶段解析失败最频繁，需人工干预恢复原始文本，增加工作流摩擦。

### 方法论局限

- **温度固定**：统一温度 1.0 无差异化控制，可能在某些阶段引入不必要的随机性。
- **无对话记忆**：阶段间无隐藏状态保留，仅依赖显式 JSON 上下文，可能遗漏隐式叙事线索。
