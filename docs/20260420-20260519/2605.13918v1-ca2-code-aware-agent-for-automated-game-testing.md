## 摘要
自动化游戏测试对于验证游戏功能至关重要，但它仍然是一项成本高昂且耗时的过程。人工测试往往遗漏边缘情况，而现有的自动化方法难以实现完整的代码覆盖率。先前的工作已经探索了强化学习（RL）在游戏测试中的应用，但没有利用内部代码信号（如调用栈）。我们提出了代码感知智能体（CA2），它利用调用栈信息学习有效的测试策略。该智能体接收当前函数调用轨迹以及游戏状态，并学习到达特定的目标函数。我们为两类环境（1）基于状态的和（2）基于图像的进行仪器化，并支持高效的调用栈提取。通过实验评估，我们发现 CA2 在不使用调用栈信息的非代码感知基线上取得了一致的改进。我们的结果表明，像调用栈这样的代码信号能够实现更有效、更有针对性的游戏测试。

## Abstract
Automated game testing is important for verifying game functionality, but it remains a costly and time-consuming process. Manual testing often misses edge cases, and current automated methods struggle to provide full code coverage. Prior work has explored reinforcement learning (RL) for game testing, but without leveraging internal code signals such as the call stack. We present Code Aware Agent (CA2), which uses call stack information to learn effective testing strategies. The agent receives the current function call trace along with the game state and learns to reach specific target functions. We instrument two types of environments, 1) State-based and 2) Image-based, with support for efficient call stack extraction. Through experimental evaluation, we find that CA2 achieves consistent improvement over the non-code aware baselines, which does not leverage call stack information. Our results show that incorporating code signals like the call stack enables more effective and targeted game testing.

## 1. 论文的核心问题与整体含义（研究动机和背景）

### 研究问题
- **自动化游戏测试的高成本与高耗时**：游戏功能验证仍是成本高昂且耗时的过程
- **人工测试的局限性**：人工测试往往遗漏边缘情况（edge cases）
- **现有自动化方法的不足**：当前自动化方法难以实现完整的代码覆盖率

### 核心观察
先前的工作大多聚焦于**状态覆盖率最大化**，但相同游戏状态可能触发不同的函数调用，仅靠状态覆盖无法保证有意义的代码覆盖。

### 核心贡献定位
提出**代码感知智能体（CA2）**，将调用栈（call stack）信息引入强化学习框架，使智能体能够针对特定函数目标进行测试，从而实现更有效的功能级代码覆盖。

## 2. 论文提出的方法论

### 核心思想
将代码级别的程序信号（如调用栈）作为额外的观测信息输入给强化学习智能体，使其能够理解动作与游戏逻辑之间的关系，进而执行更有针对性的功能测试。

### 关键技术细节

#### （1）离线强化学习框架
采用**离线学习**设置，仅使用人类测试者收集的专家轨迹数据进行训练，无需在线交互：

- **代码感知 MDP**：定义为五元组 $(S, C, A, P, G)$，其中 $C$ 为有限函数集合，$G \subset C$ 为目标函数集合
- **调用栈表示**：在 $t$ 时刻的调用栈 $s_{code}^t$ 为有序函数列表 $[f_1^t, f_2^t, \ldots, f_F^t]$

#### （2）三种基础算法架构

| 算法 | 描述 |
|------|------|
| **GC-BC** | 目标条件行为克隆，适用于离散/连续动作空间 |
| **GC-IQL** | 目标条件隐式 Q 学习，使用 expectile 回归 |
| **GC-DT** | 目标条件决策变压器，使用自回归序列建模 |

#### （3）调用栈编码方法（三选一）

- **简单 Token 嵌入**：$\text{CS}_{emb} = \sum_{i=1}^{F} E(f_i)$ — 直接查表后求和
- **线性投影**：$\text{CS}_{emb} = \sum_{i=1}^{F} W \cdot E(f_i)$ — 加入可学习投影矩阵
- **多头自注意力（MHSA）**：$\text{CS}_{emb} = \sum_{i=1}^{F} \text{MHSA}([E(f_1), E(f_2), \ldots, E(f_F)])_i$ — 捕获函数间依赖关系

#### （4）决策变压器（GC-DT）的轨迹格式

$$\tau = \langle g, s_{code}^1, s_1, a_1, s_{code}^2, s_2, a_2, \ldots, s_{code}^T, s_T, a_T \rangle$$

- 目标 $g$ 被前置作为条件
- 每步包含：调用栈、状态、动作的嵌入
- 训练目标：$\pi(a_t | g, s_{code}^{\le t}, s_{\le t}, a_{<t})$

## 3. 实验设计

### 环境与 Benchmark

| 环境 | 类型 | 状态空间 | 动作空间 | 函数数 | 目标数 |
|------|------|----------|----------|--------|--------|
| **Crafter** | 基于图像（像素） | 64×64×3 | 离散 [0,16] | 293 | 94 |
| **Godot** | 基于状态（向量） | 5×1 | 连续 [-1,1] | 34 | 24 |

### 对比方法

- **BC**：标准行为克隆（非目标条件）
- **GC-BC**：目标条件行为克隆（Simple vs CA2）
- **GC-IQL**：目标条件隐式 Q 学习（Simple vs CA2）
- **GC-DT**：目标条件决策变压器（Simple vs CA2）

### 评估指标
1. **成功率（Success Rate）**：达到目标函数的episode占比
2. **Episode 返回值**：累积奖励
3. **函数覆盖率（Function Coverage）**：唯一触发的函数数量

### 数据集
- **Crafter**：300条轨迹，约 10 万样本
- **Godot**：200条轨迹，约 2 万样本

## 6. 论文的主要结论与发现

### 核心发现

1. **CA2（代码感知）显著优于非代码感知基线**：在所有指标上均取得一致改进

2. **GC-DT 表现最佳**：相比 GC-IQL 和 GC-BC，决策变压器架构因使用上下文窗口而能推理更长时域并更有效条件化目标函数

3. **多头自注意力编码效果最好**：MHSA 能捕获调用栈元素之间的依赖关系，性能显著优于简单嵌入和线性投影

4. **更长的上下文有助于代码感知智能体**：
   - 无 CA2 时，性能在 10-20 上下文处饱和
   - 有 CA2 时，20-30 的上下文带来更好性能（Godot 成功率从 30% 提升到 47%）

### 关键数值总结

| 配置 | Crafter 成功率 | Godot 成功率 | Crafter 函数覆盖 | Godot 函数覆盖 |
|------|----------------|--------------|------------------|----------------|
| GC-DT + CA2 (MHSA) | **56.47%** | **47.15%** | **96.92%** | **79.49%** |
| GC-DT Simple | 48.78% | 41.22% | 92.96% | 71.96% |

## 7. 优点

### 方法亮点
- **首次将调用栈信息引入强化学习游戏测试**：填补了代码感知测试的研究空白
- **环境无关设计**：通过在 Godot（GDScript/Python）和 Crafter（Pygame）两个引擎上的验证，证明框架的可迁移性
- **MHSA 编码的创新应用**：有效处理变长调用栈的时序依赖问题

### 实验亮点
- **多维度评估**：从成功率、Episode 返回值、函数覆盖率三个角度综合评估
- **消融实验充分**：验证了编码方法、上下文长度、算法选择的影响
- **可复现性高**：提供了详细的超参数表和环境配置

## 8. 不足与局限

### 实验覆盖不足
- 仅在两个相对简单的环境中验证，未涉及大型商业游戏
- 函数选择（过滤每帧调用函数）虽合理但可能遗漏边界情况

### 方法局限
- **计算开销**：实时收集函数级代码覆盖和调用栈信息在高复杂度游戏中成本较高
- **函数平等对待**：将所有函数视为同等优先级，未考虑开发者实际需求
- **离线训练依赖**：依赖预先收集的专家数据，对数据质量敏感

### 可扩展性挑战
- 随着游戏代码库增大，仪器化开销可能成为瓶颈
- 当前未探索跨游戏的知识迁移

### 未来改进方向
- 探索选择性仪器化，仅跟踪开发者指定的高优先级函数
- 研究更小粒度、模块化的测试策略
- 尝试将 CA2 早期集成到开发流程中
