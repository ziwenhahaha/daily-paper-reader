## Abstract
Game development sits at the intersection of creative design and intricate software engineering, demanding the joint orchestration of game engines, real-time loops, and tightly coupled state across many files. While Large Language Models (LLMs) and code agents now solve isolated programming tasks with ease, they consistently stumble when asked to produce a fully playable game from a high-level design, collapsing under cross-file inconsistencies, broken scene wiring, and logical incoherence. We bridge this gap with OpenGame, the first open-source agentic framework explicitly designed for end-to-end web game creation. At its core lies Game Skill, a reusable, evolving capability composed of a Template Skill that grows a library of project skeletons from experience and a Debug Skill that maintains a living protocol of verified fixes - together enabling the agent to scaffold stable architectures and systematically repair integration errors rather than patch isolated syntax bugs. Powering this framework is GameCoder-27B, a code LLM specialized for game engine mastery through a three-stage pipeline of continual pre-training, supervised fine-tuning, and execution-grounded reinforcement learning. Since verifying interactive playability is fundamentally harder than checking static code, we further introduce OpenGame-Bench, an evaluation pipeline that scores agentic game generation along Build Health, Visual Usability, and Intent Alignment via headless browser execution and VLM judging. Across 150 diverse game prompts, OpenGame establishes a new state-of-the-art. We hope OpenGame pushes code agents beyond discrete software engineering problems and toward building complex, interactive real-world applications. Our framework will be fully open-sourced.

## 1. 论文核心问题与整体含义

### 研究背景
- 游戏开发是软件工程领域最具挑战性的任务之一，需要同时协调**游戏引擎**、**实时循环**和**跨多文件的紧耦合状态**
- 传统LLM虽能解决孤立编程任务，但在端到端生成完整可玩游戏时频繁失败

### 三大典型失败模式
1. **逻辑不一致 (Logical Incoherence)**：模型丢失全局状态追踪，导致游戏冻结或关键机制无法实现
2. **引擎特定知识缺失 (Engine-Specific Knowledge Gaps)**：通用模型忽视或误用引擎抽象，倾向于从零重写而非正确使用框架原生系统
3. **跨文件不一致 (Cross-File Inconsistencies)**：单个文件看似合理，但整体项目因资源键不匹配、场景连接错误、配置字段缺失等破坏

### 核心贡献
提出 **OpenGame**——首个用于端到端网页游戏创建的**开源智能体框架**，将代码智能体能力从离散编程任务拓展至复杂交互式应用。

## 2. 论文方法论

### 2.1 核心架构：Game Skill

Game Skill 由两个可演化组件构成：

**Template Skill（模板技能）**
- 从单一游戏无关的元模板 M₀ 开始
- 通过经验积累逐步发展成**模板库 L**
- 演化出 5 个专业化模板家族：
  - 重力驱动横版 (gravity-based side view)
  - 俯视角连续运动 (top-down continuous motion)
  - 离散网格逻辑 (discrete grid logic)
  - 路径/波次动态 (path/wave dynamics)
  - UI驱动玩法 (UI-driven gameplay)
- 目的：缩小代码生成的搜索空间，稳定项目结构

**Debug Skill（调试技能）**
- 维护一个**活的调试协议 P**，从构建、测试、运行时结果中持续更新
- 每次失败记录结构化条目：(错误签名，根因，经验证修复方案)
- 包含轻量级**预执行验证**，检查高频不一致类别（如资源键不匹配）
- 目的：系统性解决跨文件集成错误而非修补孤立语法问题

### 2.2 基础模型：GameCoder-27B

基于 **Qwen3.5-27B** 构建，采用三阶段训练管道：

| 阶段 | 内容 | 作用 |
|------|------|------|
| **持续预训练 (CPT)** | 使用开源 Phaser/JS 仓库、官方文档、社区教程 | 建立游戏循环、物理系统、资源使用的强先验 |
| **监督微调 (SFT)** | 用 gpt-codex5.1 策划复杂多步游戏设计提示，minimax2.5 生成高质量目标方案 | 教模型将抽象创意意图转化为具体代码结构 |
| **强化学习 (RL)** | 组件级逻辑（单文件玩法逻辑）通过单元测试执行反馈评估 | 将模型锚定在确定性、可执行逻辑上 |

### 2.3 自主智能体工作流（六阶段）

```
1. 初始化与分类 → 2. 脚手架搭建 → 3. 设计生成 → 
4. 资源合成 → 5. 代码实现 → 6. 验证与自修复
```

**关键技术细节**：
- **Physics-First 分类规则**：按物理约束而非模糊类型标签分类任务
- **Three-Layer Reading Strategy**：渐进加载 API 摘要 → 目标源文件 → 实现指南，最大化即时显著度
- **Template Method Pattern**：复制模板文件并覆盖指定钩子方法，而非从头编写
- **多模态资源生成**：generate-game-assets 合成背景、角色动画、静态物品、音频；generate-tilemap 将 ASCII 布局转换为 Phaser Tilemap JSON

### 2.4 算法流程 (Algorithm 1)

```
输入：用户规格 x，元模板 M₀，模板库 L，调试协议 P
输出：可运行游戏项目 y

1. 从 L 选择模板家族 T（初始化为 M₀）
2. 实例化 T 生成项目骨架 y
3. 在 y 的扩展点生成游戏特定内容
4. 重复（直到收敛）：
   a. 运行验证（构建、测试、运行），使用 P 引导
   b. 如有失败：用 P 诊断并修复 y
   c. 如模式新颖：追加 (signature, cause, fix) 条目到 P
5. 可选：从 y 提取可复用片段并合并到 L
6. 返回 y
```

## 3. 实验设计

### 数据集与 Benchmark

**OpenGame-Bench**
- **规模**：150 个任务，150 个独特自然语言提示
- **覆盖五大游戏类型**：
  - 横版平台游戏 (Platformer)
  - 俯视角射击 (Top-Down Shooter)
  - 街机经典 (Arcade Classics)
  - 策略游戏 (Strategy)
  - 益智/UI游戏 (Puzzle/UI)
- **数据来源**：公开游戏开发比赛的策划库 + AI 辅助设计简报
- **约束**：所有提示均要求使用 Phaser 3 框架

| **Intent Alignment (IA)** | VLM 对结构化需求规范的加权通过率 |

## 6. 主要结论与发现

### 6.1 总体性能

使用 **Claude Sonnet 4.6** 的 OpenGame 达到最优：

| 指标 | 数值 |
|------|------|
| **Build Health** | 72.4 |
| **Visual Usability** | 67.2 |
| **Intent Alignment** | 65.1 |

### 6.2 关键发现

1. **自定义模型竞争力**：GameCoder-27B 在 BH 和 VU 上优于所有直接 LLM 基线，接近大型商业智能体框架

2. **三阶段训练价值**：
   - CPT 主要提升 Build Health
   - SFT 最大提升 Intent Alignment（+1.9）
   - RL 继续提升 VU 和 IA

3. **智能体组件贡献**（消融实验 II）：
   - Hook-Driven Implementation 是最关键约束（去掉 BH -10.1，IA -11.6）
   - Three-Layer Reading Strategy 去除后 IA -8.6
   - Physics-First Classification 影响相对较小

4. **Game Skill 演化价值**（消融实验 III）：
   - 仅用静态元模板严重瓶颈（BH 60.5 → 72.4）
   - 完整调试协议（预执行验证）将性能推至最优

5. **迭代调试效率**：零次调试 IA 仅 58.4，三次迭代后快速收敛至平台期

6. **类型差异显著**：
   | 类型 | IA 分数 |
   |------|---------|
   | 平台游戏 | 76.8（最强） |
   | 俯视角射击 | 71.4 |
   | 街机/动作 | 66.5 |
   | 策略 | 58.2 |
   | 益智/UI | 52.6（最弱） |

   策略和益智类游戏状态管理更抽象，逻辑解耦时难以被自动调试检测。

## 7. 优点

### 方法论亮点

1. **首个开源端到端游戏生成智能体框架**，填补了该领域空白
2. **Game Skill 设计精妙**：Template Skill + Debug Skill 组合，既稳定结构又累积修复知识，符合软件工程最佳实践
3. **三阶段领域适配训练管道**：CPT 建立引擎先验 → SFT 对齐创意意图 → RL 锚定可执行逻辑，层次分明
4. **Physics-First 分类**：避免模糊类型标签，直接映射到物理约束和交互模式
5. **Template Method Pattern**：强制使用钩子覆盖而非从头编写，显著减少生命周期管理错误

### 实验亮点

1. **三维评估指标**：区分构建稳定性、视觉质量、意图满足，比二元成功/失败更细粒度
2. **系统化消融实验**：覆盖模型、架构、演化三个维度，因果链条清晰
3. **类型分解分析**：揭示不同游戏类型的相对难度，引导未来研究方向
4. **迭代调试曲线**：量化自修复的价值，指导实际部署中的迭代预算

## 8. 不足与局限

### 实验覆盖局限

1. **仅限 2D 网页游戏**：未涵盖 3D 游戏、主机/桌面或移动游戏，框架泛化性待验证
2. **单一框架依赖**：仅测试 Phaser 3，切换到 Unity/Unreal 时行为未知
3. **闭源模型黑盒**：部分基线（GPT-5.1、Gemini 3.1 Pro）无法分析内部失败原因

### 方法论局限

1. **性能天花板**：即使完整 OpenGame，仍有约 **34.9%** 的权重需求未满足，游戏生成本质困难
2. **抽象逻辑难以调试**：策略/益智游戏中逻辑解耦导致的静默失败难以自动检测
3. **依赖外部工具**：generate-game-assets 等依赖 Tongyi/Doubao 等多模态模型，成本和质量可控性存疑
4. **资源键管理**：虽通过资源协议减轻，但仍需人工配置

### 应用风险

1. **生成内容安全性**：未讨论游戏内容过滤或知识产权问题
2. **计算成本**：完整六阶段工作流 + 多次迭代调试，生产环境成本较高
3. **长时游戏支持**：评估仅涵盖短时长可玩性，长剧情/存档系统未覆盖
