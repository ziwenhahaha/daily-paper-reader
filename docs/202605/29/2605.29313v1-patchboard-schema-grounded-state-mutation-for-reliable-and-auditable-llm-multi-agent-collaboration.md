## 摘要
LLM 多智能体系统常通过自然语言对话或松散结构的共享记忆进行协调，使得中间状态难以验证、归属和审计。
我们提出 PatchBoard，一种基于模式的协作架构，用经过验证的 JSON Patch 变更替代智能体间的对话，作用于共享的结构化状态。
一位架构师智能体构建任务专属的模式与工作流规则，而确定性内核在事务性提交前，依据模式约束、角色专属写入契约以及运行时不变量对每项提议的状态变更进行验证。
在 630 个匹配的 ALFWorld 回合中，PatchBoard 实现了 84.6% 的成功率，而 LangGraph 为 30.8%，Flock 为 61.6%；与此同时，PatchBoard 将每个成功任务的 token 数降至 45.5k，而 LangGraph 为 368.3k，Flock 为 64.2k。

## Abstract
LLM multi-agent systems often coordinate through natural-language dialogue or loosely structured shared memory, making intermediate state difficult to validate, attribute, and audit. We introduce PatchBoard, a schema-grounded collaboration architecture that replaces inter-agent dialogue with validated JSON Patch mutations over a shared structured state. An Architect agent constructs a task-specific schema and workflow rules, while a deterministic kernel validates each proposed state mutation against schema constraints, role-specific write contracts, and runtime invariants before committing it transactionally. On 630 matched ALFWorld episodes, PatchBoard achieves an 84.6% success rate, compared with 30.8% for LangGraph and 61.6% for Flock, while reducing tokens per successful task to 45.5k, compared with 368.3k and 64.2k, respectively.

### 1. 核心问题与整体含义（研究动机和背景）

**问题背景：**
- LLM 多智能体系统（如 AutoGen、ChatDev、MetaGPT、AgentVerse）通常通过自然语言对话或松散结构的共享记忆进行协调
- 这种协调方式存在关键缺陷：对话历史随轮数增长而膨胀，任务事实与元讨论混杂，难以明确哪些中间输出应被视为已提交状态
- 下游智能体可能读取未验证的观察、过时的计划、格式错误的中间声明或失败的修复尝试，导致错误在后续推理、工具调用或环境操作中被放大

**核心问题：**
- 中间状态难以验证（validate）
- 中间状态难以归属（attribute）
- 中间状态难以审计（audit）
- 污染的中间状态可悄然影响所有后续智能体

### 2. 论文提出的方法论

**核心思想：**
PatchBoard 用**经过验证的 JSON Patch 变更**替代智能体间的开放对话，作用于共享的结构化 JSON 状态，实现"schema-grounded"（模式锚定）的协作。

**关键技术组件：**

- **Architect 智能体**：在任务初始化时调用一次，将用户请求编译为任务蓝图（B），包含：
  - 全局状态模式（Σ）
  - 工作智能体规格与角色指令
  - 读/写路径授权
  - 上下文预算（C）
  - 工作流规则（R，触发器-条件-动作格式）

- **确定性内核（Deterministic Kernel）**：唯一的可变 committed 状态的组件，负责：
  - 初始化全局状态
  - 构建有边界的智能体视图
  - 验证 JSON Patch 提案
  - 事务性提交接受的补丁
  - 记录事务日志
  - 基于事件调度后续智能体

**核心验证公式：**
```
Accept_at = Syntax(Δ_at) ∧ Auth(Δ_at, W_a) ∧ (Ŝ_{t+1} ≠ ⊥) ∧ Valid_Σ(Ŝ_{t+1}) ∧ Inv_B(S_t, Δ_at, Ŝ_{t+1})
```
其中：
- Syntax(Δ_at)：JSON Patch 语法检查
- Auth(Δ_at, W_a)：角色写入契约授权验证
- Valid_Σ(Ŝ_{t+1})：应用后状态是否符合模式 Σ
- Inv_B：运行时不变量检查

**架构特点：**
- 每个 patch 都是路径寻址的，编辑意图在字段级别显式表达
- 破坏性操作（如 remove）默认禁用
- 拒绝的 patch 被记录但不修改 committed 状态
- 提供完整的 replayable 事务日志

### 7. 优点

**方法设计亮点：**
- **清晰的可靠性边界**：确定性内核作为唯一的状态修改组件，智能体只返回候选 patch
- **路径级角色隔离**：通过写入契约防止一个角色悄然覆盖分配给另一个角色的中间产品

- **Replayable 轨迹**：给定初始状态、蓝图和事务日志，可重建状态转换序列

**实验设计亮点：**
- **严格匹配设置**：所有系统使用相同的 gamefiles、seeds、模型、配置
- **多层对比**：不仅对比端到端系统，还对比组件级别的贡献（黑板控制、消融研究）
- **故障注入评估**：系统性测试结构有效性与语义有效性的边界
- **诊断研究**：HotpotQA 明确标注为局限性诊断，非跨任务迁移主张

### 8. 不足与局限

**结构限制：**
- **Schema 有效性 ≠ 语义正确性**：Schema-valid 的声明仍可能为假、不完整或缺乏支持
- **蓝图质量依赖**：过于稀疏的模式可能阻塞有用进展，过于宽松的模式削弱角色隔离

**覆盖范围限制：**
- **主要在 ALFWorld**：任务状态和成功条件明确定义
- **开放性设置**：软件开发、科学发现、长篇生成等更开放的任务需要更丰富的模式和更复杂的验证

**工程开销：**
- Schema 构造、上下文切片、patch 验证、事务日志均带来工程复杂度
- 适用场景：审计性、归属性、状态完整性是核心需求的场景

**语义验证限制：**
- HotpotQA 诊断显示：结构有效工作流仍可能产生不支持的声明
