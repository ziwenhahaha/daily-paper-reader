## Abstract
Generating a game is not the same as making one that can be played. Despite advances in code generation, existing approaches treat game generation as one-shot translation from prompt to artifact, leaving interaction-level failures undetected. We argue that evaluating and improving game generation requires a player, and study two roles for graphical user interface (GUI) agents in this process: (1) as an objective evaluator, for which we introduce PlaytestArena, a new evaluation environment that pairs 200 browser-based game generation tasks across eight genres with rubrics of expected in-play behaviors, adjudicated by a GUI agent that loads each build in a browser and plays it; and (2) as a subjective playtester, for which we propose Play2Code, where a game agent and a GUI agent operate in a sustained loop with shared memory, turning game generation into a dialogue between coding and playing. Our experiments show that even frontier models struggle to generate playable games directly, while Play2Code achieves a 66.8\% rubric pass-rate, improving over single-pass and agentic-coding baselines by 37.1 and 14.6 points respectively. Further analysis shows that GUI playtester feedback is more traceable than a human report, yet idiosyncratic in ways reminiscent of human testers, establishing game playtesting as a critical testbed for interactive code generation. Our project website is available at https://continual-game-generation.vercel.app/.

### 1. 核心问题与整体含义（研究动机和背景）

**核心问题**：当前大语言模型在代码生成方面取得显著进展，但将游戏生成视为从提示词到产物的单次翻译过程，导致交互层面的问题（如按钮无响应、精灵渲染失败、胜利条件无法触发等）无法被检测和修复。游戏质量存在于"游玩"而非"代码"中，但现有方法缺乏有效的评估和改进机制。

**研究背景**：
- 游戏产业长期依赖人类测试人员，但AI生成游戏的速度远超人类测试能力
- 大多数代码生成任务可通过测试或规范检查质量，但游戏生成无法通过静态分析评估
- GUI智能体是唯一能在与人类相同的呈现界面上与游戏交互的代理类型

**核心论点**：评估和改进游戏生成需要一个"玩家"，而GUI智能体是实现这一角色的自然候选者。

### 2. 论文提出的方法论

#### 2.1 PlaytestArena：客观评估器

**核心思想**：构建一个评估环境，让GUI智能体通过实际游玩来评判生成游戏的质量。

**关键技术细节**：
- **游戏任务构建**：200个浏览器游戏生成任务，覆盖8种类型（解谜、策略、卡牌、平台跳跃、动作、管理、射击、其他）
- **提示词构造**：平均长度131个token（ median 119），指定状态性游戏玩法、胜负条件、玩家输入和反馈行为
- **评分标准构建**：1,548条评分标准，平均每游戏7.7条，覆盖5个维度——机制（Mechanics）、控制（Controls）、进度（Progression）、界面（Interface）、视觉反馈（Visual Feedback）
- **评估流程**：GUI智能体加载游戏、游玩、通过点击和按键与游戏交互、追踪观察哪些评分标准被满足

**GUI评估器可靠性**：在32个游戏的分层样本上，GUI评估器与人类标注者的原始一致率达84.2%（Cohen's κ = 0.64），接近人类间一致性水平（κ = 0.66）。

#### 2.2 Play2Code：主观测试者与迭代改进系统

**核心思想**：将GUI智能体纳入生成循环，使其成为改进信号源，将游戏生成转变为编码与游玩之间的持续对话。

**关键技术细节**：
- **双智能体架构**：
  - **游戏智能体（Game Agent）**：生成或修改代码，输出HTML/CSS/JS游戏
  - **GUI智能体（GUI Agent）**：加载游戏游玩，仅依赖记忆和游戏指南，无源码访问权限
- **迭代循环**：每轮包含生成/修订步骤（游戏智能体）和游玩报告步骤（GUI智能体）
- **输出制品**：
  - **游玩摘要**：描述游玩过程中的事件和交互行为
  - **可操作修复列表**：将观察到的失败映射到具体代码级变更（如"敌人保持静止应巡逻→验证敌人巡逻触发条件"）
- **三层记忆系统**：
  | 层级 | 生命周期 | 作用域 | 内容 |
  |------|----------|--------|------|
  | Episode Memory | 任务内 | 共享 | 轮次历史、摘要、修复列表、尝试记录 |
  | Skill Memory | 跨任务 | 智能体私有 | 游戏智能体：代码模式；GUI智能体：交互策略 |
  | World Memory | 跨任务 | 所有智能体共享 | 游戏规则、类型原型、设计原则 |
- **终止条件**：GUI智能体判断构建满足提示意图，或达到最大轮数（Rmax=5）

### 8. 不足与局限

1. **游戏格式单一**：当前仅支持HTML/CSS/JS浏览器游戏，未涵盖原生引擎或3D游戏

2. **测试集规模有限**：200个游戏的策划集合可能无法覆盖真实世界的全部游戏场景多样性

3. **记忆架构通用性**：当前跨所有游戏共享记忆架构，游戏特定或类型感知的记忆结构可能带来更大提升

4. **高复杂度游戏瓶颈**：复杂游戏改进有限，瓶颈在于GUI智能体能力而非改进循环本身

5. **人类差异性未完全建模**：不同人类测试者有不同的测试偏好，当前GUI智能体模拟单一类型测试者

6. **计算成本**：多轮迭代需要大量模型调用，实际部署需考虑成本效益

7. **实时性游戏挑战**：快节奏实时游戏（如射击类）因智能体推理延迟与游戏tick率不匹配而面临额外挑战
