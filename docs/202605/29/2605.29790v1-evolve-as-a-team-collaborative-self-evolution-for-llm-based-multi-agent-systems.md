## 摘要
基于大型语言模型的多智能体系统（MAS）已成为处理复杂和长时程任务的有效范式。然而，在实际任务中，MAS 在执行过程中常常出现各种失败，且这类失败在设计阶段难以根除。这促使了经验驱动的 MAS 演进方式，即系统基于自身执行经验进行改进。然而，这种演进极具挑战性，因为 MAS 的经验是长期且复杂的，交织着多个智能体的执行链和通信消息，使得难以确定需要改进的内容。为应对这一挑战，我们提出了 Meta-Team——一种基于协作式自我演进的经验驱动 MAS 演进框架。Meta-Team 保存每个智能体的执行上下文并协调任务后的通信，使智能体能够交换分布式证据以支持演进。基于此设计，Meta-Team 实施多尺度自我演进，将执行经验转化为可复用的改进，涉及智能体行为、跨智能体协作以及团队层面的组织。在六个长时程智能体基准测试中，Meta-Team 始终优于单智能体系统、人工设计的 MAS 以及已有的 MAS 演进方法；进一步分析表明，Meta-Team 能够实现更可靠且可扩展的 MAS 自我演进。

## Abstract
LLM-based multi-agent systems (MAS) have emerged as an effective paradigm for complex and long-horizon tasks. However, in real-world tasks, MAS often exhibit various failures during execution and such failures are difficult to eliminate during design. This motivates experience-driven MAS evolution, where a system improves based on its own execution experience. Yet such evolution is challenging because MAS experience is prolonged and intricate, interleaving multiple agents' execution chains and communication messages, which makes it difficult to identify what should be improved. To address this challenge, we propose Meta-Team, an experience-driven MAS evolution framework based on collaborative self-evolution. Meta-Team preserves the execution context of each agent and coordinates post-task communication, enabling agents to exchange distributed evidence for evolution. Building on this design, Meta-Team conducts multi-scale self-evolution, transforming execution experience into reusable improvements to agent behaviors, inter-agent coordination, and team-level organization. Across six long-horizon agent benchmarks, Meta-Team consistently outperforms single-agent systems, hand-crafted MAS, and prior MAS evolution methods; further analyses demonstrate that Meta-Team enables more reliable and scalable MAS self-evolution.

- **核心观点**：论文倡导"**团队协作演进**"原则——MAS 不仅应作为团队执行，也应作为团队演进。

## 2. 方法论（核心思想、关键技术细节、算法流程）

### 2.1 核心思想：协作式经验组织

论文提出三种经验组织方案的对比：
- **全局方案（Global Scheme）**：将所有智能体轨迹扁平化后喂入单一分析器 → 长轨迹下上下文和推理负担过重，失败归因准确率急剧下降。
- **局部方案（Local Scheme）**：每个智能体独立分析自己的轨迹 → 缺乏跨智能体全局视角，难以识别协作失效的根源。
- **协作方案（Collaborative Scheme）**：每个智能体保留本地执行上下文，通过多轮通信交换处理后的本地发现和下游影响追踪 → 既保留局部保真度，又恢复跨智能体感知。

### 2.2 关键技术细节

- **经验定义**：执行经验 $e \equiv (x, \tau, r)$，其中 $\tau$ 是多智能体轨迹（多个智能体执行链 $\tau_i$ 与跨智能体事件 $B$ 的交错），$r$ 是任务结果（如评估分数）。
- **更新公式**：$M_{k+1} = \Omega(M_k, E_{1:k})$，从经验序列中更新 MAS 配置。

### 2.3 三层多尺度自我演进

Meta-Team 在每次任务后对团队进行三层演进：

| 演进层级 | 目标 | 关键操作 |
|---------|------|---------|
| **L1：智能体层演进** | 改进个体智能体脚手架 $s_i$ | 智能体根据自身轨迹和跨智能体证据进行反思，更新工具、策略或行为模式 |
| **L2：交互层演进** | 改进协作机制 $P$（智能体间协议） | 更新队友画像（teammate profiles），标定对队友能力与局限的理解；更新成对协作笔记 |
| **L3：团队层演进** | 改进共享脚手架 $S$（团队宪法、成员构成） | 通过集体讨论决定是否引入新角色、移除冗余角色、调整组织结构或修订团队宪法 |

### 2.4 算法流程

```
for each task x_k:
    1. 执行 MAS M_{k-1}，收集经验 e_k = (x_k, τ_k, r_k)
    2. 并行进行 L1 演进：为每个智能体调用 Ω_L1
    3. 进行 L2 演进：为每对交互的智能体对调用 Ω_L2
    4. 进行 L3 演进：汇总前两层摘要，更新团队宪法和成员池
    5. 如有需要，根据更新后的 M 重试任务
    6. 提交更新后的 MAS M_k
```

### 2.5 实现细节

- **开放编制机制**：维护候选智能体池，动态招募活跃子集；通过 `LIST POOL`、`START AGENT`、`STOP AGENT`、`FINALIZE`、`TERMINATE` 等原语协调。
- **消息总线**：智能体间通信通过追加式消息总线，支持异步收发。
- **提交门控**：更新提交前检查角色一致性、工具可用性、格式有效性和预算约束。

## 6. 主要结论与发现

1. **Meta-Team 在所有6个基准上均取得最优性能**，平均超越手工 MAS 6.6%，超越最强 MAS 演进基线 MASFly 6.3%。
2. **协作式经验组织优于全局和局部方案**：在 TraceElephant 失败归因基准上，协作方案在短轨迹（≤128K）和长轨迹（>128K）两个区间均取得最佳 Agent Accuracy 和 Step Accuracy。

4. **可扩展性验证**：Meta-Team 在上下文规模从 8K 增长到 256K 时保持性能优势；在 Python 上进化后能泛化到 C、C++、Java 语言。
5. **成本效率**：在受限演化预算下，Meta-Team 实现最佳性能-成本权衡，超越其他演进方法。

## 7. 优点

1. **架构洞察深刻**：明确识别出 MAS 执行与演进之间的架构错配——执行时分布式、演进时集中式，重新引入单点瓶颈。
2. **团队反思哲学**：引入组织心理学中的团队反思（team reflexivity）概念，将智能体从被动诊断目标转变为主动贡献者，激活内生反馈。
3. **多尺度演进设计**：L1/L2/L3 三层演进覆盖智能体行为、交互模式和组织结构，互补性强，避免单一层面的局部优化。
4. **实验设计严谨**：使用饱和度分析排除过简单基准；在6个挑战性基准上与9个基线公平对比；提供统计显著性验证和泛化性测试。
5. **可扩展性强**：在不更新底层 LLM 参数的情况下，通过脚手架和协调规则的显式更新实现进化，具备模型无关性。

## 8. 不足与局限

1. **不涉及底层基础设施演化**：Meta-Team 仅关注脚手架和组织层面的演进，不修改工具 API、环境 harness、验证器或记忆后端等执行基础设施——这些通常需要任务特定的工程投入。
2. **不更新模型参数**：演进依赖任务后反思和显式脚手架更新，虽具可解释性和模型无关性，但限制了将重复团队行为内化到基础模型的能力。结合微调或强化学习是未来方向。
3. **依赖任务后通信**：协作式自我演进需要智能体愿意在任务后分享信息和反思结果，实际部署中可能受限于通信协议或隐私约束。
4. **统计充分性有限**：部分评估列（如 GAIA）的性能提升虽为正，但未达到统计显著水平，可能因样本量或评估方差限制。
5. **成本敏感性**：虽然 Meta-Team 在受限预算下表现良好，但其协作通信和多尺度演进仍引入额外开销，在极端成本约束场景下可能受限。
