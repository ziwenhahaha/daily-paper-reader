## Abstract
Effective multi-agent systems cannot be designed by selecting prompts or communication graphs in isolation. Agent behavior depends on the information an agent receives, while the usefulness of a communication edge depends on how the receiving agent interprets and uses that information. We propose \textbf{TCP-MCP} (Topology-Coupled Prompting for Multi-Agent Collaborative Problem-Solving), a co-evolution framework that searches agent prompts and communication topologies as a unified genome. TCP-MCP uses an initialization-time landscape probe to calibrate early search behavior, and then relies on Pareto-front diagnostics to adapt exploration under three objectives: task performance, token cost, and structural complexity. Using the same DeepSeek-V3.2 backbone across all methods, TCP-MCP achieves 82.66\%, 89.96\%, and 96.61\% accuracy on MMLU-Pro, MMLU, and GSM8K, respectively. Across the three benchmarks, it consistently outperforms automated graph-generation baselines and achieves competitive accuracy relative to debate-style systems, while using up to 5.69$\times$ fewer tokens than those systems at the reported operating points. These results show that jointly evolving prompts and communication structure provides a practical route to cost-aware and task-adaptive multi-agent system design in controlled evaluations.

## 1. 论文的核心问题与整体含义

### 研究动机
- 多智能体系统（MAS）的设计涉及两个关键组件：**智能体提示（prompts）** 和 **通信拓扑（communication topologies）**
- 现有方法往往将这两者分离优化：固定通信结构单独优化提示，或将拓扑与提示分别处理
- 核心观察：**提示行为与拓扑结构是耦合的**——同一个提示在一种通信模式下有效，在另一种模式下可能失败；一条通信边的有用性取决于接收智能体如何解读和使用该信息

### 背景问题
- 单智能体推理方法（Complex CoT、Self-Consistency）仅改进单模型推理，未考虑角色分工与通信
- 多智能体协议（LLM-Debate、LLM-Blender）引入交互聚合，但通常固定通信结构
- 拓扑设计方法（GPTSwarm、G-Designer）专注于图结构优化，提示往往被当作固定或独立组件

## 2. 论文提出的方法论

### 核心思想
TCP-MCP（Topology-Coupled Prompting for Multi-Agent Collaborative Problem-Solving）将**提示与通信拓扑视为统一的基因组**进行协同进化，而非独立优化。

### 关键技术细节

#### 2.1 基因组表示
- 候选系统表示为统一基因组：`M = (G, P)`
  - `G = (V, E)`：有向通信图（节点为角色智能体，边为通信链路）
  - `P`：节点级角色-提示分配
- 提示模板独立于拓扑存储，运行时通过 `BuildPrompt(pi, N^-(vi; G))` 注入拓扑相关上下文，使同一提示模板在不同拓扑下产生不同行为

#### 2.2 多目标优化问题
公式定义：
```
F(M) = (A(M), -C(M), -ln(1 + K(G)))
```
- `A(M)`：任务准确率（最大化）
- `C(M)`：token成本（最小化）
- `K(G) = |V| + |E|`：图结构复杂度（最小化，含对数惩罚）

#### 2.3 初始化阶段：景观探测
- 使用 **Fitness Distance Correlation (FDC)** 在初始化时进行景观探测
- 公式：`FDC = Spearman({S(Mj)}, {d(Mj, M*)})`
- 偏好得分 `S(M) = A(M)^k / ((C(M)/T0)^γ · K̄(G)^β)`
- 仅用于初始化校准，进化过程中不再重复计算

#### 2.4 协同进化操作符

**结构交叉（PMSI）**：
- 交叉传递**拓扑子图模块**，而非提示片段
- 节点按角色名精确匹配或按边界度（in_degree + out_degree）匹配
- 受体节点保留原提示模板，移植节点继承供体提示

**变异操作**：
- **拓扑变异**：`{ADD_EDGE, REMOVE_EDGE, ADD_NODE, REMOVE_NODE}`
- **角色变异**：节点角色改变时，PromptMutator 生成新提示保持对齐
- **激进变异**：引入较大结构变更（链、树、星形等），由停滞信号控制激活概率

#### 2.5 环境选择与多样性保持
- 使用 **NSGA-II** 加约束支配进行环境选择
- 动态准确率下限：`τ = best_acc(t) - δ`
- **分箱 niche**：基于token成本和结构复杂度的分位数分箱，限制每箱容量
- **结构配额**：防止过早坍缩至平凡低节点拓扑
- **外部精英档案**：定期重新引入验证过的高性能基因组

#### 2.6 自适应控制
- 控制指数 `s = w_HV·s_HV + w_acc·s_acc + w_tok·s_tok + w_div·s_div`
- 映射到激进变异概率：`p_stag = σ(αs + β)`
- 早期侧重多样性，后期侧重超体积和准确率

## 6. 论文的主要结论与发现

### 性能结论
| 指标 | 发现 |
|------|------|
| **准确率** | TCP-MCP在MMLU（89.96%）、MMLU-Pro（82.66%）、GSM8K（96.61%）均超越G-Designer等自动化拓扑设计基线 |
| **成本效率** | 相比LLM-Debate，MMLU-Pro上用**5.69倍更少token**，MMLU上用**4.59倍更少token**，GSM8K准确率更高且用**4.09倍更少token** |
| **Pareto权衡** | 选定点强调准确率，但并非最低token方案；搜索暴露了性能-成本-复杂度的明确权衡 |

### 关键发现
1. **联合优化优于顺序优化**：AB4（先提示后拓扑）全面弱于完整TCP-MCP，证明提示与拓扑必须协同变化
2. **结构交叉至关重要**：移除结构交叉导致MMLU-Pro最大降幅（-2.51pp）
3. **变异不可或缺**：移除变异不仅降低准确率，还导致1.46×更多token消耗
4. **自适应控制效益数据集依赖**：主要稳定MMLU搜索，对MMLU-Pro和GSM8K影响较小

### 泛化能力
- 跨数据集迁移：性能下降1.20-1.54pp仍保持竞争力
- 跨骨干迁移：GPT-4上TCP-MCP拓扑仍优于基线，尤其在MMLU-Pro（78.63%）

## 7. 优点

### 方法亮点
1. **统一基因组设计**：首次将提示与拓扑表示为耦合的统一搜索对象，而非分别优化
2. **Prompt Minimal Inheritance (PMI)**：结构交叉传递拓扑子图，提示继承最小化，保持模板稳定性同时允许运行时行为适应
3. **Pareto多目标自适应搜索**：同时优化准确率、token成本、结构复杂度，通过前沿诊断自适应调整探索压力
4. **初始化景观探测**：使用FDC校准早期搜索偏置，避免盲目探索
5. **丰富的多样性保持机制**：分箱niche、结构配额、外部精英档案，防止早熟收敛

### 实验设计亮点
1. **严格控制变量**：所有方法使用相同主干模型、相同API设置，确保公平比较
2. **开发集-测试集严格分离**：避免数据泄露，报告仅在held-out测试集上的结果
3. **操作点选择规则透明化**：基于开发集Pareto前沿的固定选择规则，方法论清晰
4. **多维度迁移验证**：跨数据集、跨骨干验证泛化能力

## 8. 不足与局限

### 实验覆盖局限
1. **单一主干模型**：仅使用DeepSeek-V3.2，未在GPT-4o、Claude等其他模型族上验证
2. **固定开发子集**：搜索预算和初始化协议固定，未系统探索不同预算下的表现
3. **消融实验未做统计显著性测试**：因多种子运行成本高，消融结果为方向性证据而非统计验证

### 方法论局限
1. **静态拓扑限制**：实验聚焦于静态选择的拓扑，未探索动态通信、工具交互或长时域适应
2. **选定点token消耗较高**：虽远低于LLM-Debate，但可能高于部分基线方案
3. **搜索成本较高**：搜索阶段消耗965M tokens、55.4小时，算力成本显著

### 偏差风险
1. **Pareto选择过程可能偏向特定目标权重**：开发集选择规则虽固定，但权重设置反映设计者偏好
2. **迁移实验暗示但不保证泛化性**：跨数据集转移仍有性能损失，目标场景变化时需重新搜索

### 应用限制
1. **API依赖**：实验依赖外部LLM API，未评估本地模型或开源模型的表现
2. **拓扑设计假设**：基于有向无环图（DAG）约束，可能不适用于需要循环通信的复杂工作流
