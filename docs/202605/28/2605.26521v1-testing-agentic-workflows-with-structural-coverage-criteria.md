## 摘要
多智能体系统越来越多地展示显式的工作流结构：智能体、工具、工具访问规则、限制和委托路径。现有评估主要依赖端到端任务成功率、基准分数、最终响应质量或提示级检查，这些仅提供有限的证据表明所声明的协调结构实际上已被执行。这使得难以评估测试套件的充分性，或检测工具访问、限制和智能体间委托方面的结构回归。我们通过一种针对多智能体工作流规格的结构测试方法来解决这一缺口。该方法将每个工作流表示为类型化协作图，在可达智能体、允许的工具边、受限的工具边和委托边上派生覆盖义务，并使用覆盖驱动的生成结合基于 DSPy 的场景实现来产生可执行测试。图表固定了必须覆盖的内容；DSPy 将这些义务实现为自然语言场景，其见证在运行时被检查。我们在 OpenAI Agents SDK 风格的工作流上实现该方法，并在十个 SDK 派生的基准上评估，涵盖 49 个可达智能体、47 个工具和 403 个结构义务。生成的场景在有限的细化预算内见证了 54/75 的允许工具义务和 36/48 的委托义务。对抗性受限工具标准触发了 23/248 的受限调用违规，将在探测下保持限制的工作流与存在具体错误路由失败的工作流区分开来。这些结果表明，结构性覆盖为多智能体工作流测试提供了有用的充分性层：它不取代语义或端到端评估，而是揭示声明的智能体、工具访问规则、限制和委托路径是否已被执行。

## Abstract
Multi-agent systems increasingly expose explicit workflow structure: agents, tools, tool-access rules, restrictions, and delegation paths. Existing evaluations rely largely on end-to-end task success, benchmark scores, final-response quality, or prompt-level checks, which provide limited evidence that this declared coordination structure has actually been exercised. This makes it difficult to assess test-suite adequacy or detect structural regressions in tool access, restrictions, and inter-agent delegation. We address this gap with a structural testing approach for multi-agent workflow specifications. The approach represents each workflow as a typed coordination graph, derives coverage obligations over reachable agents, allowed tool edges, restricted tool edges, and delegation edges, and uses coverage-driven generation with DSPy-based scenario realization to produce executable tests. The graph fixes what must be covered; DSPy realizes those obligations as natural-language scenarios whose witnesses are checked at runtime. We implement the approach for OpenAI Agents SDK-style workflows and evaluate it on ten SDK-derived benchmarks comprising 49 reachable agents, 47 tools, and 403 structural obligations. Generated scenarios witness 54/75 allowed-tool obligations and 36/48 delegation obligations within a bounded refinement budget. The adversarial restricted-tool criterion elicits 23/248 restricted-call violations, separating workflows whose restrictions hold under probing from workflows with concrete misrouting failures. These results show that structural coverage provides a useful adequacy layer for multi-agent workflow testing: it does not replace semantic or end-to-end evaluation, but reveals whether declared agents, tool-access rules, restrictions, and delegation paths have been exercised.

## 1. 核心问题与整体含义（研究动机与背景）

多智能体系统（Multi-agent Systems）日益普及，这些系统包含显式的工作流结构：智能体（agents）、工具（tools）、工具访问规则、限制（restrictions）和委托路径（delegation paths）。然而，现有的评估方法主要依赖端到端任务成功率、基准分数、最终响应质量或提示级检查，这些指标只能提供极为有限的证据，表明所声明的协调结构实际上已被执行过。这导致以下核心问题：

- **难以评估测试套件的充分性**：即使工作流成功完成任务，也可能存在某些工具从未被调用、某些受限工具访问从未被测试、或某些委托路径从未被探索的情况。
- **难以检测结构回归**：工作流定义的细微变更（如新增工具边、删除限制或修改委托路径）可能不会立即导致端到端失败，但现有的评估方法仍会通过，从而遗漏重要的结构性义务未观察的情况。
- **安全合规场景的需求**：在安全、政策或合规敏感的环境中，从业者通常需要可追溯的证据，证明相关的工具访问规则、限制和协调路径已被测试。

**研究空白**：端到端任务成功不是多智能体系统中结构覆盖的可靠代理指标，而结构性充分性对于多智能体工作流的测试开发不足。

## 2. 论文提出的方法论

### 核心思想

将多智能体工作流视为**类型化协调图（Typed Coordination Graph）**，从中派生结构性覆盖义务，并使用覆盖驱动的测试生成方法来生产可执行测试。具体而言：

- **图模型固定必须覆盖的内容**：工作流结构（智能体、工具、允许边、限制边、委托边）决定了测试义务，而不是任意的提示。
- **DSPy 用于受控场景实现**：DSPy 将每个义务实现为自然语言场景，但场景是否真正覆盖该义务由运行时见证决定，而非由语言模型自行判断。

### 关键技术细节

**工作流规格表示**：
- 可达智能体（`Areach`）：可从入口点通过零或多步委托可达的智能体集合。
- 允许工具边（`Eallow`）：记录哪些智能体可以调用哪些工具。
- 受限工具边（`Erestrict`）：记录智能体被明确禁止调用的工具。
- 委托边（`Edel`）：描述控制或责任如何在智能体之间转移。

**类型协调图定义**：
```
G(S) = <VA, VT, Eallow, Erestrict, Edel>
```
其中 VA 是智能体节点集，VT 是工具节点集。

**四种覆盖标准**：
1. **C1（智能体覆盖）**：每个可达智能体至少被一个测试观察到。
2. **C2（允许工具覆盖）**：每个可达允许智能体-工具边被行使。
3. **C3（受限工具覆盖）**：每个可达受限智能体-工具边通过显式受限访问观察被行使（即存在被拒绝的尝试，而非仅仅缺失调用）。
4. **C4（委托覆盖）**：每个可达委托边被观察到。

**覆盖义务集**：
```
Ω(S) = <Areach, Tallow, Trestrict, D>
```
其中 `Tallow = {(a,t) ∈ Eallow | a ∈ Areach}`，`Trestrict = {(a,t) ∈ Erestrict | a ∈ Areach}`，`D = {(a,b) ∈ Edel | a ∈ Areach ∧ b ∈ Areach}`。

**覆盖函数定义**：
```
Cov(X, O, W) = |{ω ∈ O | ∃x ∈ X: W(x, ω)}| / |O|
```
当 O 为空时定义为 1（空规范满足覆盖）。

### 算法流程

**五步管道**：

1. **规范化工作流规格 S**：记录可达智能体、工具、允许/限制边、委托边。
2. **构建可达协调图 G(S)**：从入口点计算可达性。
3. **派生结构义务**：提取 Ω(S) 四个义务集。
4. **覆盖驱动测试生成**：
   - 使用四个确定性规则（R1-R4）为每个义务创建类型化见证目标（Witness Objectives）：`Reach(a)`、`UseTool(a,t)`、`RestrictTool(a,t)`、`Delegate(a,b)`。
   - 合并兼容目标以减少套件大小（仅允许可达性目标作为次要目标合并）。
   - 使用 DSPy 模块将每个目标实现为自然语言场景（每种标准有专门签名）。
5. **运行时执行与覆盖测量**：通过运行时适配器重放场景并验证结构性见证。

**DSPy 场景实现**：
- 每种覆盖标准有专门的 DSPy 签名（`RealizeReach`、`RealizeUseTool`、`RealizeRestrictTool`、`RealizeDelegate`）。
- 使用有界细化循环（`dspy.Refine`），每次尝试生成候选提示 z，执行后根据二进制奖励评分：
  - r = 1：执行产生所需结构性证据。
  - r = 0：提示泄漏禁止标识符或未产生预期见证。
- 最多 N = 5 次尝试，若成功则接受该测试。

## 6. 论文的主要结论与发现

### RQ1：义务提取与场景实现

- 提取器在全部十个工作流上成功终止并生成格式良好的协调图。
- 义务层无需人工干预即可生成类型见证目标。
- 场景生成器在细化预算内为所有目标生成候选自然语言场景。
- 多跳路由目标需要更多实现尝试。

### RQ2：允许工具与委托见证

- **允许工具见证**：成功见证 45/65 个义务（69.2%）。
- **委托见证**：成功见证 31/41 个义务（75.6%）。

**分类发现**：
1. **单跳目标均匀通过**：`oai_customer_service`、`oai_message_filter`、`social_media_agent_system`、`deep_research_clone`（7/7 允许工具义务通过）。
2. **扇出协调器通过多数目标**：`oai_research_bot`、`oai_financial_research`、`octagon_vc_agents`（19/23 通过）。
3. **多跳和运行时模糊目标导致多数失败**：`ydmitry_deep_research` 和 `value_investment` 中的失败主要因多跳或工具-委托模糊性。

### RQ3：受限工具对抗见证

- 在 248 个受限工具义务中，23 个（9.3%）触发了受限调用违规。
- **零违规工作流**（4 个）：`oai_customer_service`（0/4）、`oai_message_filter`（0/2）、`deep_research_clone`（0/2）、`oai_research_bot`（0/12）。
- **产生违规的工作流**（6 个）：例如 `social_media_agent_system`（2/3）、`oai_financial_research`（7/42）、`octagon_vc_agents`（6/132）。
- **诊断价值**：C3 反转极性——较高的违规计数标识具体加固目标，而非积极结果。

### RQ4：故障注入鲁棒性

- 两种故障模式（`fail:error` 和 `fail:malformed`）下，鲁棒性见证均满足（2/2 通过）。
- 机械见证无法检测语义捏造（malformed 输出产生看似合理的内容仍被接受）。
- 证明相同结构测试工具可支持故障处理行为测试。

## 7. 优点

1. **填补研究空白**：首次将结构覆盖概念系统性地引入多智能体工作流测试，而非仅依赖端到端任务成功。
2. **明确的覆盖模型**：定义了四种结构充分性标准（C1-C4），提供了结构覆盖的可衡量定义。
3. **覆盖驱动的生成**：通过义务优先的方法替代任意提示生成，确保测试的针对性。
4. **受控的 DSPy 实现**：使用专用签名和有界细化循环，运行时见证决定场景有效性，而非表面合理性。
5. **诊断价值而非仅报告通过/失败**：受限工具测试揭示具体结构弱点（加固目标），而非仅给出安全断言。
6. **可扩展的测试工具**：同一结构测试工具可用于正常执行、受限访问探测和故障注入鲁棒性测试。
7. **真实代码提取的基准**：十个工作流来自真实 SDK 示例和公开第三方项目，无需人工构造。

1. **框架限制**：所有实验均在 OpenAI Agents SDK 风格工作流上进行，图模型虽声称框架无关，但提取器和运行时适配器为 SDK 专用，外部有效性存疑。

4. **生成器可达性限制**：有界细化循环（N=5）并非总能引导对话穿越多跳委托链，复杂拓扑可能持续产生未实现目标。
5. **结构覆盖 ≠ 语义正确性**：覆盖标准衡量结构性义务的观察，不测量最终答案的事实正确性、用户满意度、政策合规性或鲁棒性。
6. **基准规模限制**：演示和研究规模工作流，未评估生产规模部署或极深委托链场景。
7. **基准归一化的主观性**：某些工作流需将外部编排风格适配为 SDK 声明边，虽然文档化但可能影响图形状和派生义务。
8. **缺乏与替代方法的比较**：未与其他测试生成或评估策略进行公平比较，难以评估相对于基线的改进。
9. **C3 极性解释的复杂性**：受限工具见证的成功解释需要仔细理解（成功 = 违规被触发，这在限制执行视角下是负面结果）。
