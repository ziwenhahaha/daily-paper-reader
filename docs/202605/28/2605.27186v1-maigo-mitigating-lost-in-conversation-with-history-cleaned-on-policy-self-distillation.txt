Abstract 

Large language models often solve tasks from a fully specified prompt but degrade when the same requirements unfold over multiple turns, known as the lost-in-conversation (LiC) gap. We trace part of this degradation to self-contamination: intermediate assistant replies enter later context and carry early deviations forward. Motivated by this mechanism, we propose MAIGO, an on-policy self-distillation method that reduces this contamination using history-cleaned references from the model’s own policy. For middle turns, MAIGO re-moves prior assistant replies while preserv-ing the user-visible sharded prefix; for answer turns, it distills from paired full-view refer-ences conditioned on the completed user-side dialogue. A reliability weight downweights middle-turn samples that disagree with the clean reference. MAIGO requires no veri-fier rewards, state labels, or inference-time scaffolding. Under the LiC paired-view pro-tocol with deterministic verifiers, MAIGO im-proves Qwen2.5-7B-Instruct SHARDED accu-racy from 52.8 to 66.1 and the SHARDED/-FULL ratio from 66.5% to 84.1%, while keep-ing FULL accuracy within 2.3 points. These results show that self-contamination is a train-able component of the LiC gap. 

1 Introduction 

Large language models (LLMs) have shown strong ability to follow complex instructions when all requirements are specified in a single prompt (Ouyang et al., 2022; Liu et al., 2024a; Yang et al., 2025b,a). In real assistant use, however, requirements are often accumulated, clarified, and revised across multiple turns. This conversational setting exposes the Lost-in-Conversation (LiC) gap : a model may solve a task in the FULL view, where all requirements are provided at once, but

fail in the SHARDED view, where the same re-quirements are revealed progressively before the final answer (Laban et al., 2025). Because the two views contain the same task requirements, this gap is not simply a failure of single-turn compe-tence, but a failure to solve from a dialogue history shaped by both user turns and the model’s own earlier responses. This distinguishes LiC from or-dinary long-context access failures, where relevant information may be present but used unevenly de-pending on where and how it appears (Liu et al., 2024b), and aligns with evidence that extended reasoning depends on intermediate planning and optimization states rather than only on the final prompt (Zheng et al., 2026b). Existing LiC-oriented methods improve multi-turn robustness, but they do not directly supervise the intermediate turns where a model can con-taminate its own future context. RLSTA (Chen et al., 2026), for example, anchors optimization to the single-turn full-prompt view through scalar GRPO feedback and reports gains on contex-tual inertia. Other reward-based methods, includ-ing RLAAR (Li, 2025) and ICPO (Wang et al., 2026b), shape the policy with verifiable-accuracy or illocution-calibrated rewards, while inference-time methods such as Grounded Continuation (He et al., 2026) introduce a runtime verifier. These ap-proaches provide answer-level rewards, full-view anchors, or runtime checks, but they do not pro-vide dense supervision for middle-turn assistant replies . As a result, premature assumptions, over-confident partial solutions, or stale intermediate states can still be written into the dialogue history and later treated as context by the same model. This creates a compounding self-contamination effect ,analogous to the distributional drift studied in se-quential prediction and imitation learning (Ross et al., 2011; Bengio et al., 2015). The key idea is to train the model on references that preserve the user-visible conver-sational prefix while removing the assistant outputs that could otherwise bias the reference distribu-tion. For middle turns, MAIGO constructs history-cleaned references by deleting previous assistant replies but keeping the sharded user-side prefix in-tact. This teaches the model how to respond to the current conversational state without inheriting its own earlier mistakes. For answer turns, MAIGO distills from a paired full-view reference condi-tioned on the completed user-side dialogue, align-ing the final answer with the fully specified task. A reliability weight further reduces the influence of middle-turn updates when the sampled response diverges from the clean reference. The method requires no verifier rewards, no additional state annotations, and no inference-time scaffolding. We evaluate MAIGO on Math, Actions, Database, and Code under the LiC FULL-vs-SHARDED paired-view protocol with determin-istic verifiers. On Qwen2.5-7B-Instruct, MAIGO improves average SHARDED accuracy from 52.8 to 66.1 and raises the SHARDED/FULL ratio from 66.5% to 84.1% , while no FULL task changes by more than 2.3 points relative to the base model. These results show that reducing self-contamination can substantially narrow the LiC gap without compromising performance on fully specified tasks. Our contributions are threefold: • We identify self-contamination from interme-diate assistant replies as a trainable contrib-utor to the LiC gap, connecting multi-turn degradation to the compounding effects of self-conditioned dialogue histories. • We propose MAIGO, a turn-role-specific on-policy self-distillation method that uses history-cleaned middle-turn references, paired full-view answer references, and reliability-weighted supervision to reduce self-contamination without verifier re-wards, state annotations, or inference-time scaffolding. • We provide a controlled evaluation across math, tool-call generation, text-to-SQL, and Python code generation under the LiC paired-view protocol, showing substantial SHARDED gains while preserving FULL per-formance. 

2 Related Work 

Lost-in-Conversation and its mitigations. LiC introduced a controlled FULL/SHARDED com-parison: a task that is solvable from a complete single-turn instruction is split into user turns, and performance is measured after the same informa-tion has been revealed conversationally (Laban et al., 2025). The key implication is that the multi-turn gap is not explained by lack of task com-petence alone. A model that solves the FULL version can still fail in SHARDED form because of early assumptions, premature answer attempts, over-reliance on previous attempts, and weak re-covery once the missing information arrives. Agrowing line of subsequent studies characterizes this regime as a family of multi-turn state failures rather than a single error type, including attention drift over goal tokens, conversational inertia, and constraint or temporal-scope violations (Dongre 2et al., 2026; Wan et al., 2026; Liu et al., 2026; Canaverde et al., 2026; Kruthof, 2026; Atri et al., 2026; Zheng et al., 2026c). Existing mitigations mostly attack this regime through scalar rewards, anchoring, or runtime checks: verifiable-accuracy, abstention, or clarification rewards (Li, 2025; Wang et al., 2026b; Qiu et al., 2026), single-turn-anchor GRPO (Shao et al., 2024; Chen et al., 2026), and inference-time dependency verification (He et al., 2026). MAIGO is complementary but adopts a different lens: it treats assistant-generated history itself as the trainable failure mode and supplies dense, history-cleaned reference distributions on the student’s own SHARDED rollout. 

On-policy distillation and context-conditioned self-teaching. On-policy distillation (OPD) trains on student-generated rollouts rather than fixed reference outputs, reducing the train–test mismatch that can undermine imitation in se-quential decoding (Agarwal et al., 2024). Its self-distilled variant (OPSD) uses the same model as student and teacher under different contexts, with dense token-level matching on sampled rollouts (Zhao et al., 2026), and has been extended to context-conditioned self-distillation (Ye et al., 2026) and easier-context-to-harder-context transfer for long-context generation (Zhang et al., 2026). Recent studies further show that on-policy signals are not uniformly reliable: peer-rollout contexts, temporal curricula, scope-aware schedules, and teacher-choice analyses all suggest that not every on-policy update should be weighted equally (Yu et al., 2026; Wang et al., 2026a; Zheng et al., 2026a; Zhu et al., 2026). MAIGO follows this methodological line but adapts it to sharded conversations, where the main failure mode is self-contamination from earlier assistant replies. Rather than using a uniform reference context, it constructs turn-role-specific references: history-cleaned supervision for middle turns and paired-view recovery supervision for answer turns. A divergence-adaptive weight further modulates middle-turn updates, connecting the method to reliability-aware OPD while targeting the distinctive structure of multi-turn LiC failures. 

3 Method 

MAIGO is an on-policy self-distillation method for sharded multi-turn tasks. The method trains on the student’s own sharded rollouts, but computes reference distributions under cleaner training-time contexts. This allows the student to learn from the histories it actually induces, while preventing its own earlier replies from contaminating the refer-ence signal. The training signal has three components. First, middle turns are supervised with history-cleaned sharded references that preserve the user-visible prefix but remove previous assistant replies. Sec-ond, the final answer turn is supervised with a completed-task reference that preserves the full user-side sharded dialogue and appends a canoni-cal reserialization of the paired FULL specification. Third, a separate FULL-view branch regularizes the model toward its original full-prompt behavior. MAIGO requires no verifier rewards, no extra state labels, and no inference-time scaffolding.

3.4 Training Objective 

The training objective has two branches. The sharded trajectory branch combines answer-turn re-covery with weighted history-cleaned middle-turn distillation, and the FULL-view branch acts as a preservation regularizer: 

Ltrain = (1 −z)( Lfinal + Lmid ) + z Lfull ,z ∼ Bernoulli( ρ), (17) where ρ controls how often the preservation branch is sampled; with probability 1 − ρ, the update uses the sharded trajectory branch. 4 Experiments 

We report SHARDED accuracy as the primary metric and FULL accuracy as a complete-prompt preservation check. 

4.1 Evaluation Setup 

We follow the Lost-in-Conversation FULL-vs-SHARDED paired-view protocol: each item has a complete-prompt view and a sharded conversa-tional view with the same requirements revealed across user turns (Laban et al., 2025). Following LiC and subsequent LiC-based training studies, SHARDED evaluation uses a model-assisted user simulator that reveals at most one unrevealed shard per turn, lets the evaluated assistant respond, and then decides whether an answer attempt should be scored (Laban et al., 2025; Li, 2025). All meth-ods use the same fixed DeepSeek-V3.1-Terminus simulator for user turns and answer-attempt clas-sification (Liu et al., 2024a), along with the same shard files, prompts, seeds, and deterministic scor-ers. Fine-tuning uses disjoint sharded training sets of 6,400 examples for Math, Actions, and Code, and 6,397 examples for Database; Appendix A gives construction details. Outputs that fail to parse, are empty, violate the response protocol, or are trun-cated by the generation budget count as incorrect. For MAIGO, these scorers are evaluation-only, not training rewards. 

Tasks and metrics. We use the four objectively scored task families from LiC—the same subset adopted by recent LiC-based training studies (Li, 2025; Chen et al., 2026; Qiu et al., 2026)—and exclude the two judge-scored ones (summary, cap-tioning) so that every cell is auditable by a de-terministic verifier. Math is instantiated by LiC-style GSM8K with numeric matching (Cobbe et al., 2021); Actions by BFCL parallel/parallel-multiple tool calls with deterministic AST matching (Patil et al., 2025); Database by fixed-schema sharded text-to-SQL on Spider no-easy with execution matching (Yu et al., 2018); and Code by Hu-manEval Python function generation scored by unit tests (Chen et al., 2021). Together, they cover nu-meric, tool-call, SQL, and program outputs, giv-ing deterministic probes of different contamination channels. 

Models and baselines. The primary compari-son uses Qwen2.5-7B-Instruct, with 3B rows as a smaller-backbone validation (Yang et al., 2025b). Baselines are per-task SFT, GRPO with sparse ver-ifiable rewards (Shao et al., 2024), and RLSTA, which uses the stronger single-turn view as a GRPO anchor (Chen et al., 2026). All trained methods use Low-Rank Adaptation (LoRA) adapters (Hu et al., 2022); adapter weights are merged into the base model before evaluation. All methods use the same paired FULL/SHARDED prompts under a matched 100-step fine-tuning budget. We report the mean of four training seeds for each fine-tuned method. Appendix A.2 reports optimizer, context, and generation settings.

5 Conclusion 

We studied sharded multi-turn task solving as a self-generated-history problem: when requirements ar-rive across turns, intermediate assistant replies en-ter later context and can hinder final completion. MAIGO addresses this with adaptive on-policy self-distillation on the student’s own sharded roll-outs, combining history-cleaned middle-turn su-pervision, answer-turn recovery, and full-prompt preservation. Across math, tool/action generation, SQL, and Python code generation, MAIGO im-proves SHARDED completion over SFT, sparse-8reward RL, and single-turn-anchor RL while keep-ing FULL performance comparable. These results suggest that robust multi-turn solving benefits from supervising student-generated histories under refer-ence contexts that preserve user-provided require-ments while removing earlier assistant replies. 

Limitations 

MAIGO may improve a language model’s abil-ity to preserve and recover task requirements over longer conversations, but the same capability could also make harmful or inappropriate objectives more persistent. The reference views used during train-ing can also encode biased, incorrect, or unsafe requirements, and those errors may be distilled into the student model. Our evaluation focuses on objectively scored tasks, so it does not estab-lish safety, privacy protection, or refusal behav-ior in open-domain dialogue. Deployment should therefore pair the method with safety filtering, per-mission controls, and careful handling of sensitive information.

A.1 Tasks and Metrics 

Each task is evaluated in paired FULL and SHARDED views: FULL presents the complete task in one prompt, while SHARDED reveals the same requirements across user turns and scores the final answer. The evaluation uses GSM8K ( N =1319 , numeric answer match), Ac-tions ( N =382 , tool-call AST match), Database (N =715 , SQL execution match), and HumanEval (N =164 , unit tests). Table 4 summarizes how the paired views are constructed. We follow the LiC principle that FULL and SHARDED expose the same task re-quirements, with SHARDED delaying their presen-tation across user turns. The conversion follows the fixed task-family protocols used for our LiC-style evaluation and is shared by all methods; the table reports the information needed to interpret the comparison at a protocol level. For SHARDED interaction, we retain the sim-ulator roles from the LiC protocol: an evaluated assistant, a model-assisted user model that decides which unrevealed shard to reveal next, and a system component that classifies answer attempts and ex-tracts the candidate answer for the task verifier. Our experiments use a fixed DeepSeek-V3.1-Terminus model for the user model and answer-attempt clas-sifier. This model controls only the interaction en-vironment around the evaluated assistant; the shard-ing protocol, turn budget, answer extraction inter-face, and deterministic task verifiers remain aligned with the LiC evaluation protocol. Every model and training method is evaluated with the same simula-tor model, same shard files, same prompt templates, same turn budget, and same task verifier. Outputs that fail to parse, are empty, violate the response protocol, exceed the context limit, or reach the interaction limit are counted as incorrect. This convention keeps FULL and SHARDED com-parable: a model must both produce an answer in the expected format and solve the underlying task. 

A.2 Optimization and Baselines 

All trained methods are optimized as LoRA adapters (Hu et al., 2022) and evaluated after merg-ing the adapter weights into the base model. Ta-bles 5 and 6 summarize the optimization budget and MAIGO-specific settings used in the main experiments. The causal com-parison for this component is the corresponding uniform-weight ablation in the main ablation table. The main comparison rows are matched fine-tuning comparisons under a common adapter, train-ing set construction, optimization budget, and eval-uation setup. The RL baselines use their own sparse task rewards; RLSTA additionally uses the paired FULL view as its single-turn anchor reward. No method receives extra evaluation-time state or a different SHARDED prompt. 

A.3 Training-Only Reference Views 

The reference contexts are used only for comput-ing distillation losses during training. Middle turns. The middle-turn reference keeps the user turns through the current shard and task-specific system information that is shared by the paired views. It removes earlier assistant replies and withholds both unrevealed user turns and the full task view.

It adds a canonical serialization built from the completed dialogue and its paired FULL view, then removes earlier assistant replies such as pre-mature calls, draft SQL, or partial code attempts. B.4 Offline Shard Construction 

Shard construction is an offline preprocessing step. Runtime evaluation and fine-tuning consume the resulting shard lists and do not call the shard con-structor. We use fixed task-family prompt pro-grams following the LiC paired-view protocol: GSM8K uses arithmetic segmentation followed by conversational rewriting; Actions uses BFCL re-quest segmentation and conversational rewriting while keeping function schemas in the task system prompt; Database uses a fixed-schema delayed-query construction; and Code uses HumanEval-style function-specification segmentation, conver-sational rewriting, and validation.
