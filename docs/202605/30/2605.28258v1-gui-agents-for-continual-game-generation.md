## 摘要
生成游戏与制作可玩游戏的区别不容忽视。尽管代码生成技术取得进展，现有的方法将游戏生成视为从提示到产物的单次翻译，忽视了交互层面的失败。我们认为，评估和改进游戏生成需要一个玩家，并研究图形用户界面（GUI）智能体在此过程中的两个角色：（1）作为客观评估者，我们引入PlaytestArena，这是一个新的评估环境，它将跨八个类型的200个浏览器游戏生成任务与预期游戏行为的评分标准配对，由GUI智能体在浏览器中加载每个构建并游玩来进行评判；（2）作为主观测试者，我们提出Play2Code，其中游戏智能体和GUI智能体在共享内存的持续循环中协同工作，将游戏生成转变为编写代码与游玩的对话。我们的实验表明，即使是前沿模型也很难直接生成可玩的游戏，而Play2Code达到了66.8%的评分标准通过率，分别比单次生成和智能体编程基线高出37.1和14.6个百分点。进一步分析表明，GUI测试者的反馈比人类报告更具可追溯性，但在某些方面与人类测试者相似，从而将游戏测试确立为交互式代码生成的重要试验平台。我们的项目网站位于 https://continual-game-generation.vercel.app/。

## Abstract
Generating a game is not the same as making one that can be played. Despite advances in code generation, existing approaches treat game generation as one-shot translation from prompt to artifact, leaving interaction-level failures undetected. We argue that evaluating and improving game generation requires a player, and study two roles for graphical user interface (GUI) agents in this process: (1) as an objective evaluator, for which we introduce PlaytestArena, a new evaluation environment that pairs 200 browser-based game generation tasks across eight genres with rubrics of expected in-play behaviors, adjudicated by a GUI agent that loads each build in a browser and plays it; and (2) as a subjective playtester, for which we propose Play2Code, where a game agent and a GUI agent operate in a sustained loop with shared memory, turning game generation into a dialogue between coding and playing. Our experiments show that even frontier models struggle to generate playable games directly, while Play2Code achieves a 66.8\% rubric pass-rate, improving over single-pass and agentic-coding baselines by 37.1 and 14.6 points respectively. Further analysis shows that GUI playtester feedback is more traceable than a human report, yet idiosyncratic in ways reminiscent of human testers, establishing game playtesting as a critical testbed for interactive code generation. Our project website is available at https://continual-game-generation.vercel.app/.

### 1. 核心问题与整体含义

- **研究背景**：当前基于大语言模型（LLM）的游戏代码生成方法，将游戏生成视为从提示词到产物的单次转换（one-shot translation），导致交互层面的可玩性缺陷无法被检测
- **核心问题**：游戏质量存在于"游玩"而非"代码"层面；代码编译通过不代表游戏可玩，交互逻辑缺失（如按钮无响应、精灵渲染失败、永不触发的胜利条件）无法通过静态分析发现
- **研究动机**：游戏行业依赖人类测试员进行实际游玩并反馈体验，但LLM生成速度远超人类测试能力，需要可扩展的自动化替代方案；GUI智能体因其与人类相同的感知-决策-行动界面而成为自然候选
- **核心论点**：评估和改进游戏生成需要一个"玩家"，而GUI智能体是唯一能在人类测试员所使用的相同呈现表面上与游戏交互的替代方案

### 2. 方法论：核心思想与技术细节

#### 2.1 PlaytestArena：客观评估环境

- **构建方式**：
  - 200个浏览器游戏生成任务，涵盖8种类型（解谜、策略、卡牌、平台跳跃、动作、管理、射击、其他）
  - 每个任务包含生成提示词和配套评分标准（Rubric），共计1,548条评分标准，平均每游戏7.7条
  - 评分标准覆盖5个维度：机制（Mechanics）、控制（Controls）、进度（Progression）、界面（Interface）、视觉反馈（Visual Feedback）
  - 提示词平均长度131个token，足够具体但保持自包含

- **评估流程**：
  - GUI智能体在浏览器中加载生成的HTML/CSS/JS游戏包
  - 通过截图观察渲染界面
  - 执行点击、按键等交互动作
  - 追踪观察到的行为与每条评分标准的符合程度
  - 最终得分为通过率（Passed Criteria / Total Criteria）

- **GUI智能体验证**：
  - 小规模可行性研究：20个游戏、118个关卡
  - 三个后端模型（GPT-5.4、Claude Sonnet 4.6、Kimi K2.5）独立游玩
  - GPT-5.4达到Pass@20=0.82，接近人类参考0.92
  - 验证智能体具备三项核心能力：**观察**（描述渲染屏幕）、**推理**（连接当前状态与目标）、**行动**（产生可观察的状态转换）

#### 2.2 Play2Code：主观测试者与持续改进机制

- **核心思想**：将游戏生成从单次翻译转变为编码与游玩的持续对话，游戏智能体（Game Agent）与GUI智能体通过共享记忆系统协同演化

- **双智能体协同架构**：
  - **Game Agent**：负责生成、调试、修补游戏代码；读取GUI反馈，决定采纳或放弃哪些修复建议；输出自包含HTML游戏包
  - **GUI Agent**：在浏览器中游玩构建版本；不访问源代码，仅依赖渲染界面和游戏指南；输出游玩总结（Play Summary）和可操作修复列表（Actionable Fix List）
  - 两智能体在共享游戏运行时周围形成闭环，每轮包含"生成/修补"和"游玩并报告"两个阶段

- **三层记忆系统**：
  - **Episode Memory**（任务内共享）：积累轮次历史、总结、修复列表、尝试记录
  - **Skill Memory**（跨任务私有）：每个智能体积累的专有know-how，如游戏智能体的代码模式、GUI智能体的交互策略
  - **World Memory**（跨任务共享）：通用游戏规则、常见原型、设计原则

- **轮次终止条件**：GUI智能体判定当前构建已满足生成提示意图，或达到最大轮次Rmax=5；平均有效轮次3.24轮，93.5%的任务在达到Rmax前终止

### 6. 主要结论与发现

- **生成可玩性差距**：即使是前沿模型，直接生成可玩游戏的能力有限；Direct LLM平均通过率仅29.7%

- **Play2Code显著提升**：
  - 达到**66.8%评分标准通过率**
  - 比单次生成基线提升**37.1个百分点**
  - 比OpenGame代理编码基线提升**14.6个百分点**

- **GUI游玩反馈的价值**：
  - 代码层面和编译层面的检查无法发现运行时交互问题（如无响应的控件、缺失的状态转换、永不触发的胜利条件）
  - Play2Code持续优于OpenGame，证实游玩测试信号的独特价值

- **轮次改进特性**：
  - 评分分数随轮次单调递增
  - 早期轮次暴露基本交互失败（如缺少输入响应）
  - 后期轮次处理更细致问题（如视觉效果）

- **记忆系统各层独立贡献**：
  - Episode Memory贡献最大（无记忆基线64.1% → 完整72.3%）
  - Skill Memory对重复性实现陷阱（平台跳跃跳跃弧度、塔防波次调度）有增益
  - World Memory对 underrepresented 类型有跨任务知识迁移效果

- **GUI智能体不可或缺**：
  - 移除GUI智能体后跌至58.7%，低于甚至"无记忆"基线
  - GUI智能体提供的基于执行的反馈能发现代码自检系统性遗漏的问题

- **游戏复杂度调制效应**：
  - 低复杂度游戏：前几轮即暴露大部分潜在缺陷，3-5轮收敛
  - 中等复杂度游戏：从迭代循环获益最大，五轮持续单调改进
  - 高复杂度游戏：改进有限，GUI智能体触发复杂机制能力不足成为瓶颈

- **GUI vs. 人类测试者**：
  - GUI反馈**更具可追溯性**：每步行动、观察、诊断均有记录，锚定在可恢复的帧序列中
  - 不同GUI后端产生**不同侧重的反馈**：GPT-5.4聚焦功能bug和控制响应；Sonnet 4.6均匀关注体验问题；Kimi K2.5关注视觉美学维度
  - 这种差异性与人类测试员的个体差异相似
  - GUI智能体在早期轮次与人类轨迹接近，但复杂游戏中人类仍保持优势

- **GUI评估器可靠性**：
  - 与人类标注者逐标准一致率84.2%，Cohen's κ=0.64，在人类-人类协议范围内（κ=0.66）
  - 游戏级别排名Spearman ρ=0.87，Pearson r=0.88，方法间比较对评估器残余噪声稳健

- **双角色方法论**：
  - 客观评估者（PlaytestArena）：提供标准化、可复现的游戏可玩性benchmark
  - 主观测试者（Play2Code）：将评估反馈转化为优化信号，实现编码-游玩闭环

- **系统设计优雅**：
  - 三层记忆系统（Episode/Skill/World）沿生命周期和范围两个维度组织，实现经验跨轮次和跨任务复用
  - 游戏智能体与GUI智能体职责解耦但协同，避免相互干扰

- **实验全面扎实**：
  - 多粒度验证：从单关卡Pass@K可行性研究 → 完整benchmark → 消融实验 → 人类对比
  - 多维度消融：记忆各层独立消融 + GUI智能体核心贡献验证
  - 可靠性验证：GUI评估器与人类标注者盲评对比，确认评估器偏差在可控范围

- **可解释性**：GUI智能体输出结构化报告（观察日志、游玩总结、可操作修复列表），比人类测试报告更可追溯

- **实用导向**：聚焦HTML浏览器游戏，与现有GUI智能体的操作表面天然对齐，易于部署

### 8. 不足与局限

- **格式覆盖有限**：
  - 当前仅覆盖HTML/CSS/JS浏览器游戏
  - 未扩展到原生游戏引擎或3D游戏格式
  - 未来工作需适配更广泛的游戏类型

- **数据集代表性**：
  - 200个游戏虽覆盖8种类型，但为人工筛选的精选集
  - 不涵盖真实世界游戏场景的全部多样性
  - 存在选择偏差风险：精选游戏可能比真实生成产物更规范

- **记忆系统通用性**：
  - 当前所有游戏共享统一记忆schema
  - 游戏特定或类型感知的记忆结构可能带来更大收益
  - 不同游戏类型对记忆复用模式需求差异未被充分探索

- **GUI智能体能力瓶颈**：
  - 高复杂度游戏中，GUI智能体触发复杂机制能力有限
  - 精确计时任务（如平台跳跃）仍是弱点
  - 推理延迟 vs. 游戏帧率存在潜在冲突

- **人类对比的不对称性**：
  - 人类测试员在Play2Code循环中不写记忆条目
  - 每次观察仅存在于当期修复报告中
  - 信息获取不对称可能导致人类优势被部分放大

- **评估维度局限**：
  - 评分标准聚焦可观察行为（机制、控制、进度等）
  - "游戏性"的主观体验维度（难度曲线、挫败感、趣味性）难以通过行为准则完全捕捉
  - 人类测试员的"感受判断"仍无可替代
