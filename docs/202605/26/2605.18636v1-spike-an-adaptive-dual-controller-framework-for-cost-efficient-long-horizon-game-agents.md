## 摘要
开放世界游戏中的长时域多模态智能体必须在紧张的 token 和延迟预算下在许多低级交互中保持目标导向。现有方法经常在每步昂贵推理和可能导致漂移、重复失败和恢复不良的反应式执行之间权衡。我们的关键思想是在局部稳定段上复用战略推理，并在事件边界处重新调用它。我们提出 SPIKE，这是一种用于成本高效长时域游戏控制的自适应双控制器框架。其战略控制器执行低频全局规划、失败分析和恢复，而其反应控制器在严格的 token 预算下处理快速局部执行。事件触发器监控视觉变化、任务进度、重复动作和失败信号，以决定控制应保持反应式还是升级到战略推理。层次记忆将状态-动作记忆库（SA-MB）中的短期经验复用与状态动作知识图（SA-KG）中的结构化证据分离，允许每个控制器检索其需要的上下文。这种设计在多个反应步骤上复用战略提议，在计划变陈旧时支持本地覆盖，并将昂贵的推理保留在额外推理有用的时刻。在 StarDojo 的 Lite-100 分割上，SPIKE 将 Lite-100 成功率（SR）相比最强的 Lite-100 基线提高了 5.0 个百分点（相对 38.5%），将预算 SR 相比最强的预算基线提高了 9.3 个百分点（相对 75.6%）。它还分别降低了 54.9% 的 token 消耗和 40.8% 的延迟。消融显示事件触发、反应覆盖和异构记忆各自对成功和恢复有贡献，支持选择性推理而非每步推理。

## Abstract
Long-horizon multimodal agents in open-world games must stay goal-directed across many low-level interactions under tight token and latency budgets. Existing approaches often trade off costly per-step reasoning against reactive execution that can drift, repeat failures, and recover poorly. Our key idea is to reuse strategic reasoning across locally stable segments and reinvoke it at event boundaries. We present SPIKE, an adaptive dual controller framework for cost-efficient long-horizon game control. Its Strategic Controller performs low-frequency global planning, failure analysis, and recovery, while its Reactive Controller handles fast local execution under a strict token budget. An Event Trigger monitors visual change, task progress, repeated actions, and failure signals to decide when control should stay reactive or escalate to strategic reasoning. Hierarchical Memory separates short-term experience reuse in the State-Action Memory Bank (SA-MB) from structured evidence in the State Action Knowledge Graph (SA-KG), allowing each controller to retrieve the context it needs. This design reuses strategic proposals over multiple reactive steps, supports local override when plans become stale, and reserves expensive reasoning for moments where extra deliberation is useful. On the Lite-100 split of StarDojo, SPIKE improves Lite-100 success rate (SR) by 5.0 percentage points (38.5% relative) over the strongest Lite-100 baseline and Budgeted SR by 9.3 points (75.6% relative) over the strongest budgeted baseline. It also reduces token consumption by 54.9% and latency by 40.8%. Ablations show that event triggering, reactive override, and heterogeneous memory each contribute to success and recovery, supporting selective reasoning rather than reasoning at every step.

## 1. 核心问题与整体含义

### 研究动机
开放世界游戏中的长时域多模态智能体需要在严格的 token 和延迟预算下，在大量低层交互中保持目标导向。当前的两种主流方法存在明显缺陷：

- **单步推理型方法**（如 CRADLE）：能够有效推理，但每步都调用大模型，导致 token 消耗和延迟极高
- **纯反应型方法**（如 ReAct）：执行成本低，但在漂移、执行失败、重复和恢复方面表现脆弱

这形成了**规划-延迟-记忆三难困境**：更深的推理提升决策质量但增加成本，轻量级反应执行快速但视野受限，简单的记忆复用会引入检索噪声。

### 核心问题
如何在保持任务完成成功率的同时，最小化推理成本和延迟？具体而言：

- 何时应该触发昂贵的战略推理
- 如何在局部执行和全局规划之间分配控制权
- 如何设计记忆系统以支持不同层级的决策需求

## 2. 方法论

### 核心思想
将战略推理的成本分摊到局部稳定的执行段上，仅在事件边界处重新调用战略规划。关键洞察是：长时域游戏控制的代价-收益不匹配——常规步骤需要快速执行，而罕见的场景变化、停滞和失败需要昂贵的推理。

### 关键技术架构

**自适应双控制器框架**包含三个核心组件协同工作：

#### 2.1 Strategic Controller（战略控制器）
- **职责**：低频全局规划、失败分析和恢复
- **工作流程**：信息收集 → 自我反思（可选）→ 任务推理 → 动作提议
- **输出**：为接下来 T 步提供短期动作提议
- **激活条件**：当周期性刷新、卡住状态或恢复信号需要深思熟虑的推理时激活

#### 2.2 Reactive Controller（反应控制器）
- **职责**：在严格 token 预算下处理常规步骤
- **工作机制**：
  - 遵循最新战略提议
  - 当新证据使提议失效时，可进行有限的局部修正（避免、重新定位、工具重选）
  - 不能改变当前子目标
  - 修正失败则将控制权交还事件触发器

#### 2.3 Event Trigger（事件触发器）
基于可重现的代理事件检测器，使用固定阈值检测以下信号：

```
g_t = I[(c_t ≥ T) ∨ (d_t > τ_v) ∨ (z_t ≥ τ_z) ∨ (r_t ≥ τ_r ∧ z_t ≥ τ_rz) ∨ (ℓ_t ≥ τ_ℓ)]
```

其中：
- `c_t`：距离上次战略调用以来的反应步数
- `d_t`：视觉不连续性检测（1 - cos 相似度）
- `z_t`：进度停滞计数
- `r_t`：行为循环检测
- `ℓ_t`：执行失败级别（0-3）

**失败级别定义**：
- ℓ=0：正常执行
- ℓ=1：孤立局部问题，触发有限重试
- ℓ=2：硬执行错误或低进度循环，升级并携带失败轨迹
- ℓ=3：重复硬失败，无效化当前计划，强制顶层重规划

#### 2.4 Hierarchical Memory（分层记忆）

**State-Action Memory Bank (SA-MB)**：
- 用于反应控制器的快速局部动作先验
- 检索评分函数：
  ```
  S_MB(i|x_t) = ρ_i(α_C·C_i + α_L·L_i + α_R·R_i + α_P·P_i)
  ```
- 综合考虑：Jaccard 词汇重叠、token 频率余弦相似度、奖励 EMA、成功率和时效性

**State-Action Knowledge Graph (SA-KG)**：
- 用于战略控制器的结构化证据图
- 检索评分函数：
  ```
  S_KG(j|x_t) = β_C·C_j + β_P·P_j
  ```
- 从最近状态节点开始，通过向量搜索保持匹配度 > τ_KG 的候选

**融合机制**：
```
S_fuse = λ_MB·S̃_MB + λ_KG·S̃_KG
```
未匹配的图动作得分为零。

### 闭环执行流程

```
for t = 1 to H:
    1. 观察 o_t = (v_t, u_t, f_t)，更新历史
    2. 计算视觉变化 d_t、停滞 z_t、重复 r_t、失败级别 ℓ_t
    3. Event Trigger 决定 g_t（战略或反应）
    4. 若 g_t=1：检索 SA-KG，用 Strategic Controller 更新提议
      否则：检索 SA-MB，用 Reactive Controller 发出动作
    5. 执行动作，接收反馈
    6. 仅在成功、可解释恢复或规划器验证后写入记忆
```

## 6. 主要结论与发现

### 6.1 核心性能提升

| 指标 | 结果 | 相对提升 |
|------|------|----------|
| Lite-100 SR | 18.0% | +38.5%（vs CRADLE 13.0%） |

| Latency/Step | 43.5s | -40.8%（vs CRADLE 73.5s） |
| Recovery/Stuck Ratio | 2.75 | +10.4%（vs CRADLE 2.49） |

### 6.2 关键发现

1. **Pareto 最优性**：SPIKE 在成功率-成本权衡上位于 Pareto 前沿，既不单纯增加计算也不牺牲质量

2. **任务类别差异**：
   - 农业任务成功率最高（34.9%）
   - 战斗和社交任务最难（8.3%、10.7%）

3. **事件触发的作用**：
   - 默认设置提供最佳成功-成本平衡
   - 过早触发增加战略调用但降低成功率
   - 移除周期刷新导致最大性能下降

4. **记忆组件贡献**：
   - SA-MB 移除：SR -3.0pp
   - SA-KG 移除：SR -3.7pp
   - 分层记忆优于全局记忆（12.3% → 18.0%）

5. **跨域泛化**：在 RDR2 上，SPIKE 达到 56.4% SR，比 CRADLE 高 12.8pp，token 减少 48.5%，延迟降低 47.1%

## 7. 优点

### 7.1 方法创新

1. **成本感知推理分配**：首次将战略推理建模为预算分配问题，在事件边界上进行

2. **双控制器协同设计**：
   - Strategic Controller 处理需要全局视角的决策
   - Reactive Controller 在局部执行中快速响应
   - 明确的优先级和覆盖机制防止冲突

3. **多层事件检测**：融合视觉变化、进度停滞、行为循环和失败级别四种信号

4. **异构记忆架构**：
   - SA-MB 支持快速局部重用
   - SA-KG 提供结构化战略证据
   - 写入过滤防止错误信息积累

### 7.2 实验设计亮点

1. **多维度评估**：成功率、成本（token）、延迟、恢复率全面覆盖

2. **控制变量严谨**：
   - 所有方法使用相同后端、环境接口、动作空间
   - 仅方法特定组件不同

3. **消融层次清晰**：从框架级到组件级逐步隔离贡献

4. **泛化验证**：跨视觉域和任务类型测试框架迁移性

### 7.3 工程贡献

1. **实用阈值设计**：固定阈值适用于同类游戏族，无需任务级调整

2. **记忆更新策略**：仅记录成功或规划器验证的进展，防止噪声积累

3. **模块化架构**：各组件可独立替换或升级

## 8. 不足与局限

### 8.1 实验覆盖局限

1. **单一主要后端**：Qwen3.5-397B 的结果可能不完全迁移到其他模型架构

2. **RDR2 任务规模**：13 个任务相对有限，统计可靠性有待提升

3. **任务类别偏差**：战斗和社交任务仍是瓶颈，可能需要类别特定的适配

### 8.2 方法论局限

1. **固定阈值依赖**：
   - 事件触发器依赖手工设定的固定阈值
   - 跨游戏迁移需要重新校准
   - 无法自适应不同游戏的交互节奏

2. **视觉子任务难题**：
   - 视觉变化细微时可能误触发或漏触发
   - 游戏状态变化但屏幕差异不大时难以检测

3. **记忆失效风险**：
   - 检索到的记忆可能过时
   - 写入过滤虽有限但仍可能引入噪声

4. **资源约束**：
   - 恢复和重规划仍依赖显式模型调用
   - 成本和延迟仍是部署的实际约束

### 8.3 潜在偏差风险

1. **选择偏差**：Lite-100 和 RDR2 任务可能不能代表所有开放世界游戏

2. **评估指标局限**：
   - Recovery/Stuck Ratio 依赖执行日志，在纯视觉设置中可能不准确
   - Budgeted SR 的调用预算可能与实际部署场景不同

3. **对比基线**：未与更近期的端到端训练方法（如 NitroGen）对比

### 8.4 应用限制

1. **无特权环境 API**：依赖屏幕观察，可能对视觉混淆更敏感
