## 摘要
由大型语言模型驱动的多智能体系统（LLM-MAS）擅长复杂任务，但不可靠智能体仍是系统级可靠性的关键瓶颈。因此，自动失败归因至关重要，但现有方法，例如直接预测智能体-错误对和智能体优先失败归因，依赖智能体的局部日志，容易遗漏只有在完整交互轨迹上才显现的全局失败，例如跨步骤不一致和智能体间协调错误。此外，直接预测失败会引入巨大的组合搜索空间，阻碍细粒度归因。为应对这些挑战，我们提出 VerifyMAS，这是一个用于智能体失败归因的假设验证框架。VerifyMAS 不直接预测故障智能体和错误类型，而是针对完整轨迹构造并验证失败假设。这种基于验证的方法将归因分解为轨迹级错误验证和细粒度智能体定位，提供一种错误优先的归因方式，既能捕捉全局失败模式，又能显著减少搜索空间。我们进一步引入基于假设的数据构造策略，该策略以结构化错误分类为基础，并微调一个专门的 LLM 验证器模型，用于轨迹级失败验证和智能体归因。在 Aegis-Bench 和 Who&When 上的实验表明，VerifyMAS 能够持续改进包括开源 Qwen 和基于 API 的 GPT 模型在内的多种骨干模型，且在长多智能体轨迹上超过先前方法而不牺牲推理效率。

## Abstract
Large language model-driven multi-agent systems (LLM-MAS) excel at complex tasks, yet unreliable agents remain a key bottleneck to system-level reliability. Automatic failure attribution is therefore critical, but existing approaches, such as direct prediction of agent-error pairs and agent-first failure attribution, rely on local logs of agents and miss global failures that only manifest over full interaction trajectories, such as cross-step inconsistencies and inter-agent coordination errors. Moreover, directly predicting failures induces a large combinatorial search space, hindering fine-grained attribution. To address these challenges, we propose VerifyMAS, a hypothesis verification framework for agent failure attribution. Instead of directly predicting faulty agents and error types, VerifyMAS formulates and verifies failure hypotheses against full trajectories. This verification-based approach decomposes attribution into trajectory-level error validation and fine-grained agent localization, providing an error-first attribution approach that captures global failure patterns while substantially reducing the search space. We further introduce a hypothesis-based data construction strategy grounded in a structured error taxonomy and fine-tune a specialized LLM verifier model for trajectory-level failure verification and agent attribution. Experiments on Aegis-Bench and Who&When show that VerifyMAS consistently improves diverse backbone models, including open-source Qwen and API-based GPT models, outperforming prior methods without sacrificing inference efficiency for long multi-agent trajectories.

### 核心问题
- **不可靠智能体是关键瓶颈**：在协作设置中，单个智能体的失败会通过整个系统传播，最终导致错误或次优的结果。智能体失败可能表现为多种形式：偏离任务要求、误解角色、忽略关键信息、产生不一致推理、或过早终止流程。

### 现有方法的局限
1. **直接预测（Direct Prediction）**：将错误类型和故障智能体组合为单一预测任务，存在以下问题：
   - 缺乏明确的证据基础
   - 难以捕获跨交互步骤的长程依赖
   - 倾向于产生表面级错误标签
   - 诱导出智能体和错误类别的大组合搜索空间

2. **智能体优先归因（Agent-first Attribution）**：使用 CoT 提示先识别负责智能体再归因错误，但存在缺陷：
   - 每个智能体的日志只提供完整轨迹的部分观察
   - 容易使模型偏向局部交互上下文
   - 无法有效捕获仅在完整交互轨迹中才显现的全局失败（如跨步骤不一致、跨智能体协调错误）

## 2. 论文提出的方法论

### 核心思想
**VerifyMAS** 提出了一个**假设验证（hypothesis verification）框架**，其核心思想是：
- 不直接预测故障智能体和错误类型，而是**先针对预定义的失败模式构建假设**
- 然后**验证这些假设是否被完整轨迹所支持**
- 采用"**错误优先（error-first）**"归因范式，优先验证错误类型再定位智能体

### 关键技术细节

#### 2.1 问题形式化
- **轨迹定义**：$\tau = (I, l_1, ..., l_T)$，其中 $I$ 是初始任务指令，$l_1, l_2, ..., l_T$ 是 $K$ 个 LLM 智能体 $A = \{a_1, a_2, ..., a_K\}$ 在 $T$ 个交互步骤中产生的日志序列。
- **错误空间**：由预定义的 $M$ 种错误类型组成的分类法 $Y = \{y_1, ..., y_M\}$。
- **地面真值标注**：$G(\tau) = \{(a^*_1, y^*_1), (a^*_2, y^*_2), ...\}$，表示导致系统失败的具体智能体-错误对。

#### 2.2 零样本失败归因流程

**步骤一：错误假设验证**
- 对每个错误类型 $y_m \in Y$，构建自然语言假设 $h_m$（例如："An agent deviated from the specified task requirements in the trajectory."）
- 使用 LLM 验证器对轨迹 $\tau_i$ 和假设 $h_m$ 进行三向验证：
  - **A (Entail)**：轨迹提供明确或足够强的证据表明假设的失败发生
  - **B (Neutral)**：证据混杂、不完整、弱，或对最终结果的影响不确定
  - **C (Contradict)**：轨迹明确表明假设为假或与实际发生的不一致
- 形式化：$\hat{z}_{i,m} = \arg\max_{z \in \{A,B,C\}} f_\theta(z | \tau_i, h_m, P, A_i)$

**步骤二：智能体级归因**
- 仅对标记为 Entail (A) 的假设进行智能体归因
- 从当前候选集合 $A_i$ 中选择负责的智能体
- 支持多智能体负责同一失败的情况
- 最终归因结果：$\hat{G}(\tau_i) = \{(a_n, y^m) | y^m \in Y, a_n \in A_i\}$

#### 2.3 假设验证引导的有监督微调（SFT）

**训练数据构建策略**：
1. **Entail 样本构建**：直接从地面真值失败标注构建正例
   - 输入：$(\tau_i, h_m, P, A_i)$
   - 输出：$\{"label": "A", "agent": a\}$

2. **Contradict 样本构建**：使用反事实关键词
   - 定义每个错误类型的反证据关键词表（如表 7 所示）
   - 当轨迹中出现这些关键词时，对应的错误假设标记为 Contradict

3. **Neutral 样本构建**：使用邻近错误类型映射
   - 根据语义相关性或易混淆性，将错误类型分组（如表 8 所示）
   - 未被明确驳回但缺乏支持的相关错误假设作为 Neutral

4. **稀有智能体重采样**：缓解长尾分布问题
   - 统计每个智能体作为故障智能体的出现频率
   - 对频率低于阈值的稀有智能体的正例进行重复采样

- 在保持强性能的同时不会引入过度的推理开销

## 7. 优点

### 7.1 方法创新性
1. **错误优先范式**：不同于传统的"智能体优先"方法，采用"错误优先"归因策略，先验证错误假设再定位智能体
2. **三向假设验证**：引入 Neutral 类别处理证据不明确的情况，避免强制将不确定案例归类为正类或负类
3. **分解式归因**：将失败归因分解为轨迹级错误验证和细粒度智能体定位，显著减小组合搜索空间

### 7.2 训练策略设计
1. **假设验证引导的 SFT**：设计了基于假设验证的微调策略，将轨迹-假设对与标注的智能体-错误对结合
2. **精细的负样本构建**：
   - 使用反证据关键词构建 Contradict 样本
   - 使用邻近错误类型映射构建 Neutral 样本
   - 提供细粒度的负监督信号
3. **稀有智能体重采样**：有效缓解训练数据中的长尾分布问题

### 7.3 全局错误捕获能力
- 能够捕获仅在完整交互轨迹中才显现的失败模式（如跨步骤不一致、跨智能体协调错误）
- 在全局错误和混合错误上的显著提升证明了方法的优越性

### 7.4 广泛的适用性
- 模型无关：可应用于各种规模的语言模型
- 无需额外训练即可通过提示工程实现零样本推理
- 可与 SFT 结合进一步提升性能

### 7.5 实验设计严谨性
- 多模型、多数据集、多评估维度的综合验证
- 消融实验完整，验证各组件的贡献
- 效率分析全面

## 8. 不足与局限

### 8.1 实验覆盖局限
- **基准测试有限**：仅在 Aegis-Bench 和 Who&When 两个数据集上评估，虽然它们涵盖了多种任务域，但可能无法完全代表所有实际应用场景
- **领域覆盖**：主要关注数学、科学、编程等任务，对其他领域（如对话、创意写作）的适用性待验证

### 8.2 方法局限
1. **保守性可能不足**：验证驱动的设计可能导致在某些轨迹上的预测不够保守，可能倾向于过度归因
2. **假设质量依赖**：方法性能依赖于预定义错误分类法的完整性和准确性，如果错误类型覆盖不全，会影响归因效果
3. **多智能体责任分配**：当多个智能体共同导致一个失败时，准确分配责任仍然具有挑战性

### 8.3 适用性限制
- **特定错误模式**：针对预定义的 14 种错误类型设计，对新出现的错误模式可能需要重新定义假设
- **轨迹长度敏感**：虽然声称对长轨迹高效，但当轨迹非常长时（超过 8192 token），可能需要额外的截断或分段策略
- **计算资源需求**：SFT 需要 2 个 H200 GPU，虽然不是特别高，但对于资源受限的场景可能难以应用

### 8.4 偏差风险
- **假设偏差**：预定义的错误假设可能存在偏差，某些边界情况可能被错误归类
- **训练数据偏差**：基于 Aegis-Bench 训练的模型可能对特定的智能体协作模式过拟合
- **领域迁移**：跨领域迁移时的性能衰减未充分评估

### 8.5 未来改进方向
- 论文中提到的改进方向：
  1. 探索额外的补充监督以实现更精确的失败定位
  2. 研究更保守的预测策略以减少过度归因
  3. 扩展到更多的错误模式和更广泛的应用场景
