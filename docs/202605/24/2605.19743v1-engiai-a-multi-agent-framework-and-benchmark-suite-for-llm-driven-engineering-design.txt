## ABSTRACT 

Large Language Model (LLM) agents are increasingly applied to engineering design tasks, yet exist-ing evaluation frameworks do not adequately address multi-agent systems that combine simulation, retrieval, and manufacturing preparation. We introduce a benchmark suite with three evaluation dimensions: (1) a workflow benchmark with seven prompt styles targeting distinct cognitive de-mands—including direct tool use, semantic disambiguation, conditional branching, and working-memory tasks; (2) a Retrieval-Augmented Generation ( RAG ) benchmark with gated scoring isolating retrieval contributions to parameter selection; and (3) an High Performance Computing ( HPC ) bench-mark evaluating end-to-end ML training orchestration on a SLURM cluster. Alongside the benchmark we present E NGI AI, a Multi-Agent System ( MAS ) reference implementation built on LangGraph that operationalizes the benchmark by coordinating seven specialized agents through a supervisor architecture, unifying topology optimization, document retrieval, HPC job orchestration, and 3D printer control. Across four LLM backends and two EngiBench problems, proprietary models achieve 96–97% average task completion on Beams2D, while open-source 4B-parameter models reach 55– 78%, with clear generational improvement. Conditional branching proves most challenging, with task completion dropping to 20–53% for the conditional style on Photonics2D. RAG gating confirms near-perfect retrieval-augmented scores ( ≈1.0) versus near-zero without retrieval, validating the evaluation design. On HPC orchestration, one model completes all pipeline steps in 100% of runs while another drops to 50%, revealing that multi-step instruction following degrades over long-running workflows. 

Keywords multi-agent systems · large language models · topology optimization · retrieval-augmented generation 

## 1 Introduction 

Engineering design is inherently complex, spanning conceptual layout, simulation-driven optimization, and manufactur-ing preparation, each relying on specialized software and iterative refinement [ 1 ]. In recent years, various Machine Learning ( ML ) models have been adopted to streamline the generation of design candidates, particularly through inverse design and surrogate models [ 2 ]. While these models can generate high-quality designs, they often function as “black boxes” that are inflexible and difficult for non-ML experts to interact with or interpret.

Concurrently, the proliferation of Large Language Models ( LLM s), specifically in the form of chatbots such as ChatGPT [ 3], has enabled users to interact naturally with complex systems through conversational interfaces. This creates an opportunity: LLM agents can serve as a natural-language bridge between engineers and specialized design tools, making ML-driven workflows accessible without requiring deep expertise in the underlying models. This paper introduces a benchmark suite for evaluating LLM -driven engineering workflows along three dimen-sions—workflow execution, retrieval-augmented parameter selection, and High Performance Computing ( HPC ) training orchestration—together with E NGI AI, a multi-agent reference implementation that operationalizes the benchmark. Accessible through a web-based interface, E NGI AI coordinates multiple engineering tools through natural-language interaction, with a modular architecture that allows new capabilities to be added via additional agents or tool APIs. The benchmark suite and reference implementation together evaluate the following research questions: 

RQ1 - Workflow Performance: Which cognitive demands—explicit parameters, qualitative descriptions, seman-tic disambiguation, conditional branching, derived computation, and multi-export tracking—most degrade workflow task completion and design quality, across proprietary and open-source LLM backends? 

RQ2 - Model Robustness: How robust is design performance to LLM model choice and configuration? 

RQ3 - Tool Usage: How does tool-calling efficiency relate to design performance—are correct designs and efficient tool use co-occurring competences? 

RQ4 - RAG Improvement: How much does retrieval contribute to parameter selection accuracy, and can the contribu-tion be cleanly isolated from parametric memorization? 

RQ5 - HPC Orchestration: How reliably do current proprietary LLM agents complete long-horizon HPC training pipelines, and at which pipeline stages does multi-step instruction-following degrade? We contribute (1) a benchmark suite comprising seven workflow prompt styles targeting distinct cognitive de-mands—direct tool use, semantic disambiguation, conditional branching, derived parameter computation, and working-memory tasks—evaluated across two engineering problems and four LLM backends; (2) a gated RAG scoring mecha-nism that isolates the contribution of document retrieval to parameter selection, preventing agents from earning credit through prior knowledge or alternative information channels; (3) an HPC training benchmark evaluating end-to-end or-chestration of generative model training on SLURM clusters—a capability not addressed by existing agent benchmarks; and (4) a LangGraph [ 4 ]-based multi-agent reference implementation that operationalizes the benchmark, integrating topology optimization, RAG, HPC orchestration, and 3D printer control through a supervisor architecture. The source code and evaluation framework are planned for public release. A video demonstration of the system is also available. 

## 2 Related Work 

Engineering design is classically decomposed into the phases of task clarification, conceptual design, embodiment design, and detail design culminating in manufacturing preparation [ 5 ], with iteration across and within these phases consistently identified as a defining characteristic of design [ 1]. Modern ML -augmented design adds two enabling layers: large-scale training infrastructure for surrogate and generative models (e.g., EngiOpt [ 6]), and agentic coordination 

between heterogeneous tools, recently formalized as Intelligent Design 4.0 [ 7]. We organize E NGI AI around a representative selection of six capabilities that map onto these phases: multi-agent orchestration handles natural-language entry and inter-tool routing (task clarification); physics simulation and topology optimization cover the analysis side of embodiment and detail design; RAG prevents parameter hallucination by grounding decisions in domain documents; STL export supports the manufacturing handoff; HPC orchestration enables ML -based design training; and structured benchmarking isolates model capability from prompt sensitivity. Existing systems cover subsets of these phases but, to our knowledge, no prior work addresses all of them within a single framework (Table 1). 

2.1 Multi-agent orchestration 

Recent surveys [ 8, 9 ] document the rapid growth of agentic AI—systems employing LLM -based agents to accomplish complex task sequences—across scientific domains. General-purpose frameworks such as AutoGen [ 10 ], CrewAI [ 11 ], the OpenAI Agents SDK [ 12 ], and LangGraph [ 4] provide multi-agent orchestration infrastructure, while the AI Co-Scientist [ 13 ] demonstrated that structured agent orchestration can outperform single-model reasoning on complex research tasks. Jiang et al. [7 ] situate these developments within a four-stage evolution of intelligent design, proposing an ontological framework for multi-agent end-to-end design automation (ID 4.0). In mechanical engineering, several MAS shave applied these ideas to specific design tasks: topology optimization and Finite Element Analysis ( FEA ) [ 14 , 15 , 16 ], 2EngiAI: A Multi-Agent Framework and Benchmark Suite for LLM-Driven Engineering Design PREPRINT 

industrial structural analysis in aerospace [ 17 ], automotive conceptual design [ 18 ], early-stage product design [ 19 ], mechatronics [ 20 ], and design-state graph coordination across multiple LLM backends [ 21 ]. However, these systems focus on conceptual design phases and do not extend to manufacturing export, HPC -based training, or retrieval-augmented parameter selection. 

2.2 Tool integration and manufacturing 

Bridging the gap from design to physical production requires standardized tool interfaces and manufacturing-aware agents. The MCP [22 ] addresses the former by standardizing tool integration across heterogeneous services, though Wang et al. [ 23 ] show that even frontier models struggle with complex multi-step tool orchestration. On the man-ufacturing side, Jadhav and Farimani [ 24 ] demonstrate a hierarchical agent system for real-time Fused Deposition Modeling ( FDM ) print monitoring, but their pipeline does not include simulation or design optimization. For standard-ized access to engineering problems, Felten et al. [6 ] introduce EngiBench, providing a unified Python Application Programming Interface ( API ) for common mechanical engineering problems and ML -based design optimization via the companion EngiOpt framework. No existing system integrates tool orchestration with both physics-based simulation and retrieval-augmented reasoning. 

2.3 Retrieval-augmented engineering 

RAG [ 25 , 26 ] enables agents to ground their outputs in domain-specific documents rather than relying solely on parametric knowledge. Without retrieval, agents risk hallucinating parameter values or falling back on training-data defaults—a failure mode that is particularly consequential in engineering, where incorrect parameters produce physically invalid designs. Chandrasekhar et al. [ 27 ] pair Llama2-7B with indexed Additive Manufacturing ( AM ) papers to achieve competitive performance with GPT-4 on metal additive manufacturing queries. Khanghah et al. [28 ] extend multimodal RAG to zero-shot anomaly detection in laser powder bed fusion, showing that retrieval from scientific literature improves classification accuracy by 12% over non-retrieval baselines. These applications demonstrate the value of retrieval for individual engineering tasks, but none embed RAG within a multi-agent simulation pipeline where retrieved parameters directly drive downstream tool calls. 

2.4 Agent benchmarking 

Evaluating agent systems requires benchmarks that capture the multi-step, tool-intensive nature of real workflows. Recent surveys [ 29 ] categorize agent evaluation along dimensions such as tool-calling accuracy, multi-step plan-ning, and task completion. Tool-calling benchmarks—ACEBench [ 30 ], ToolSandbox [ 31 ], BFCL [ 32 ], and Tool-Bench [ 33 ]—consistently reveal performance gaps between open-source and proprietary models. Broader agent evaluations such as AgentBench [ 34 ], AgentBoard [ 35 ], and ScienceAgentBench [ 36 ] assess reasoning across diverse environments but target single-agent tasks. Within engineering, FDM-Bench [ 37 ] evaluates LLM s on FDM tasks, and Zhou et al. [ 38 ] introduce a hierarchical benchmark revealing that models struggle with open-ended modeling tasks. For multi-turn professional tasks, τ -bench [ 39 ] reports <50% task success even for state-of-the-art models. However, none of these benchmarks address multi-agent engineering workflows that combine simulation, retrieval, and HPC orchestration. Such workflows expose failure modes absent from standard tool-calling evaluations: agents must chain numeric outputs from one tool into downstream decisions (e.g., reading a simulation result to select conditional parameters), maintain numerical precision when prompts contain competing plausible values for the same parameter, and track multiple parameter sets across sequential tool calls—working-memory and semantic-grounding demands that single-call benchmarks do not elicit. 

## 3 Methods 

3.1 System Architecture 

The E NGI AI system follows a hierarchical supervisor pattern (Figure 1): a central supervisor agent receives each user prompt, classifies the intent, and routes execution to the appropriate specialist agent. Each specialist operates as a self-contained LangGraph state machine—receiving a task, invoking its domain-specific tools, and returning results to the supervisor. The supervisor pattern is motivated by tool-set scalability: prior work shows that LLM s degrade in tool-selection accuracy as the number of available tools grows [ 23 ], so decomposing the tool space across specialized agents keeps each agent’s effective tool list small and bounded. The supervisor can chain multiple agents sequentially (e.g., RAG agent to retrieve parameters, then engineering agent to run the optimization) and enforces termination when repeated re-routing is detected. 3.3 Experimental Setup 3.3.1 Evaluation Problems 

We evaluate on two EngiBench [6] problems spanning different physics domains. 

Beams2D. A 50 × 100 cantilever beam topology optimization problem minimizing compliance subject to a volume fraction constraint [ 44 , 45 ]. This well-studied structural optimization benchmark provides ground-truth solutions for evaluating agent correctness. Key parameters are volume fraction ( volfrac ), force distance ( forcedist ), and filter radius (rmin ). 

Photonics2D. A 120 × 120 photonic device inverse design problem maximizing electromagnetic field overlap for specified wavelengths, testing whether agent capabilities transfer across physics domains. 

3.3.2 Models and Configuration 

We evaluate four LLM backends spanning two categories: proprietary cloud models (GPT-5-mini (gpt-5-mini-2025-08-07 ) [ 46 ], Gemini-3-Flash ( gemini-3-flash-preview )) and open-source 4B-parameter models served locally via Ollama, a lightweight inference server for open-weight models (Qwen3-4B (qwen3:4b-instruct-2507-q8_0 ) [ 47 ], Qwen3.5-4B ( qwen3.5:4b-q8_0 )). This selection enables comparison across both the proprietary–open-source divide and across model generations at fixed parameter count; all four models support structured tool calling and were chosen at cost-efficient tiers to reflect realistic deployment budgets. All calls use temperature = 0 with a fixed API seed to maximize reproducibility (see Section 5.3 for residual variation). Each model–style cell comprises 15 runs (3 random seeds × 5 dataset samples). Beams2D is evaluated on all seven prompt styles; Photonics2D on three (see Section 5.3).

3.4.1 Workflow Evaluation 

We define seven prompt styles spanning distinct cognitive demands (Table 3; representative prompts in Appendix C). The first two establish baselines: F ULL provides all numerical parameters for a direct optimize →simulate →render pipeline, while N ATURAL uses qualitative descriptions (e.g., “lightweight” instead of volfrac = 0 .2), requiring the agent to detect ambiguity and request human clarification. The five W-styles extend F ULL with an STL export step. Each modifies only the export instructions while keeping the optimization and simulation steps identical, isolating the effect of instruction phrasing on task completion. The styles target three categories of failure mode that standard tool-calling benchmarks do not capture. The first category is numerical fidelity : can the agent propagate precise values from prompt to tool call? W-R AND 

tests this directly with explicit randomized export parameters (density threshold, XY scale, Z extrusion, mirror flag). W-D ERIVED raises the bar by specifying parameters as derivation rules (e.g., “use the volume fraction as the density threshold”), requiring arithmetic over values the agent passed to earlier tools. The second is semantic grounding : can the agent select the correct value when the prompt contains competing alternatives? W-D ISTRACT presents two plausible values for the same parameter—one in a preview context, one for manufacturing export—both of which the tool accepts without error. Only the parameter-validation scorer distinguishes correct from incorrect choices, making this failure mode invisible to benchmarks that check only function signatures. The third is stateful reasoning : can the agent condition its actions on runtime information? W-C OND requires the agent to read a compliance value from simulate_design and select the corresponding parameter branch via an if/then rule. W-M ULTI requires two sequential export calls with entirely different parameter sets, testing whether the agent can maintain distinct configurations in working memory rather than merging or dropping one. 3.4.3 HPC Training Evaluation 

This benchmark tests whether the agent can orchestrate a complete ML training pipeline on remote HPC infrastructure. The expected workflow comprises four sequential steps: (1) generate a SLURM training script with the correct algorithm, problem, epochs, and seed; (2) submit the script to the cluster for remote execution; (3) monitor the job until completion; and (4) evaluate the trained model against the dataset baseline using EngiOpt metrics (Maximum Mean Discrepancy ( MMD ), Determinantal Point Process ( DPP ), Ratio of Violated Constraints ( RVC ), and optimality gaps Intermediate Optimality Gap ( IOG ), Cumulative Optimality Gap ( COG ), Final Optimality Gap ( FOG ); defined in Appendix B). We define 10 prompts per style (seeds 1–10) with 100 fixed training epochs, using two prompt formats (Appendix C): EXPLICIT (step-by-step instructions with tool names) and N ATURAL (high-level description requiring the agent to infer the workflow). We report results for the cGAN algorithm; diffusion model results are provided in Appendix D.1. 

3.5 Scoring Methodology 

The workflow evaluation combines three aspects into a single hierarchical score: design quality (65%)—a weighted combination of six sub-metrics covering geometric overlap (Intersection over Union ( IoU ), pixel accuracy), engineering objectives (objective match, constraint match), and manufacturability (2D connectivity, 3D watertightness; individual weights in Appendix A); tool use (20%)—the ratio of correctly matched calls to the maximum of optimal and actual call counts; and task completion (15%)—a binary indicator of whether all required tools were called successfully, with STL export styles additionally requiring parameter match within tolerances (Appendix A). Weights reflect the evaluation priorities: design quality receives the largest share (65%) because it directly measures whether the agent produced a correct engineering artifact; tool use (20%) captures operational cost, as redundant or missing tool calls indicate inefficient agent behavior; and task completion (15%) acts as a binary gating signal that is already implicitly reflected in design quality (a failed task typically yields no designs or low-quality ones). To support alternative weighting schemes, Appendix A reports all individual sub-scores alongside the composite. Two caveats apply when interpreting design quality: (i) the 3D watertightness sub-score is structurally low across all models because our threshold-based STL extraction produces non-manifold meshes (see caption in Tables 8 and 9), so this term reflects a pipeline limitation rather than agent capability; and (ii) since the manufacturability sub-scores are only meaningful for the W-styles that invoke STL export, design quality should be compared per style rather than as a cross-style average. For evaluating generative models ( HPC training benchmark), we additionally report global distribution metrics— MMD for dataset fidelity and DPP for design diversity—as well as engineering performance metrics such as RVC and optimality gaps (IOG, COG, FOG), following Felten et al. [6]; full definitions are given in Appendix B. 

RAG Scoring The RAG evaluation employs a gated scoring mechanism: parameter accuracy dimensions only contribute to the score when the agent invokes the search_documents tool, preventing credit for values reached through alternative means. The score combines per-parameter accuracy with a retrieval indicator (Appendix A.2). 

HPC Workflow Scoring The HPC training benchmark scores workflow orchestration rather than design quality. The primary metric is a weighted composite of step completion (70%), configuration correctness (15%), and evaluation metric extraction (15%). Appendix A.3 provides the full formulation.

## 5 Discussion 

5.1 Key Findings 

Our evaluation yields the following answers to the research questions posed in Section 1: 

RQ1 (Workflow Performance) : GPT-5-mini and Gemini-3-Flash achieve 96–97% average task completion across all seven Beams2D prompt styles (Table 5), indicating that current-generation LLM s can reliably execute structured engineering tool chains when given explicit instructions. The open-source Qwen3.5-4B reaches 78%, a considerable improvement over Qwen3-4B (55%), demonstrating rapid generational progress at fixed model scale. On Photonics2D (Table 6), W-C OND proves considerably harder (53% for the best model), suggesting that conditional reasoning difficulty is problem-dependent and tied to domain familiarity. 

RQ2 (Model Robustness) : Performance is consistent between GPT-5-mini and Gemini-3-Flash. The two open-source 4B-parameter models show that generational improvements can partially compensate for scale: Qwen3.5-4B closes the gap on workflow styles where Qwen3-4B failed (e.g., F ULL TC: 0.00 → 0.73 , W-D ERIVED TC: 0.47 → 0.85 ), though both still underperform proprietary models on the hardest styles. This generational trend extends to Photonics2D, where Qwen3.5-4B more than doubles its predecessor’s W-C OND task completion ( 0.20 → 0.47 ). 

RQ3 (Tool Usage Efficiency) : Tool usage analysis reveals that all models call the correct tools in the expected order for standard workflows, but natural and conditional prompts expose divergent tool-calling patterns. The tool-count analysis (Figure 5) shows that excess tool calls degrade the combined overall score from 0.70 (3 tools) to 0.63 (4–5 tools) without improving design quality, which remains flat at ≈0.5. This demonstrates that tool efficiency directly impacts the combined score through its efficiency component, while the engineering output quality is robust to over-calling. Of note, Qwen3.5-4B achieves optimal tool efficiency on several workflow styles by calling exactly the required tools with no unnecessary invocations, explaining its competitive combined scores despite lower task completion. 

RQ4 ( RAG Augmentation) : RAG -on runs approach a score of 1.0 on most model–prompt combinations (Figure 6), while RAG -off runs score exactly 0 due to the gated scoring mechanism. The Empty RAG condition degrades substantially, confirming that agent accuracy depends on the content returned by the retrieval system, not merely on the presence of the retrieval tool. This validates the gated scoring design: without retrieval, even correct parameter guesses receive no credit, ensuring the benchmark measures retrieval-augmented decision-making specifically. 12 EngiAI: A Multi-Agent Framework and Benchmark Suite for LLM-Driven Engineering Design PREPRINT 

RQ5 (HPC Orchestration) : Within the two cloud models evaluated (Gemini-3-Flash and GPT-5-mini), the HPC agent successfully orchestrates end-to-end ML training pipelines on SLURM -based clusters (Figure 7), including job submission, monitoring, and result retrieval. However, success is model-dependent: Gemini-3-Flash completes the full pipeline in 100% of runs, while GPT-5-mini drops to 70% under explicit instructions and 50% under natural-language prompts at the final evaluation step. This suggests that for the tested proprietary models, multi-step instruction following degrades over long-running workflows, even for models that perform well on shorter tool chains. 

5.2 Comparison to Prior Work 

As established in Section 2 (Table 1), no existing system simultaneously covers all six capability dimensions. We compare our results to the most relevant prior work along each evaluation dimension. 

Workflow Performance The closest multi-agent engineering system is the 9-agent MAS with Design State Graph (DSG) coordination [ 21 ]. Direct numerical comparison is not meaningful because the two systems address structurally different tasks: DSG-MAS targets full design-state management across coupled requirement–design–test loops, whereas our benchmark evaluates tool-calling workflows on structured engineering pipelines. We therefore position DSG-MAS as a complementary contribution rather than a numerical baseline; matched-task comparison would require a shared evaluation framework that does not currently exist. Our workflow benchmark also extends the scope of existing engineering evaluations. FDM -Bench [ 37 ] evaluates LLM on G-code anomaly detection and user queries but does not test multi-step tool orchestration. General-purpose tool-use benchmarks such as ToolSandbox [ 31 ] and ACEBench [ 30 ] report significant performance gaps between open-source and proprietary models, consistent with the degradation we observe for the open-source 4B-parameter models on complex prompt styles, though generational improvements (Qwen3-4B → Qwen3.5-4B) can partially close this gap. Similarly, τ -bench [ 39 ] reports <50% task success ( pass 8 < 25% ) for multi-turn professional tasks; our higher completion rates likely reflect the more constrained, tool-bounded nature of engineering workflows compared to open-ended customer service scenarios. 

RAG Evaluation While RAG is a well-established technique [ 25 , 26 ], its evaluation in engineering contexts has focused on document comprehension rather than actionable parameter extraction. Doris et al. [ 49 ] benchmark multimodal LLM understanding of engineering documentation (rule comprehension, compliance, and extraction), but do not evaluate whether retrieved information is correctly applied to downstream tool calls. Our gated scoring mechanism addresses this gap by isolating the contribution of document retrieval to parameter selection, revealing that agents can reliably extract domain-specific parameters when documents are indexed but cannot produce post-cutoff values without retrieval. 

HPC Orchestration The closest work in automating simulation execution is ALL-FEM [ 16 ], which uses multi-agent LLM collaboration to generate and execute FEniCS code on Google Colab, achieving 71.79% accuracy across 39 FEA benchmarks. However, ALL-FEA targets code generation for simulation execution in a notebook environment rather than HPC job orchestration. To the best of our knowledge, no prior work benchmarks end-to-end LLM -driven HPC training for engineering ML models. The ability to generate Simple Linux Utility for Resource Management ( SLURM )scripts, submit jobs to a production cluster, monitor execution, and evaluate trained models—achieving metrics comparable to official EngiBench baselines—demonstrates a capability not addressed by existing agent benchmarks. 

General-Purpose Coding Agents General-purpose coding agents—such as OpenAI Codex [ 50 ], Claude Code [ 51 ], and OpenClaw [ 52 ]—can write and execute arbitrary code, potentially calling EngiBench and EngiOpt API s directly without pre-bound tool abstractions. Our benchmark suite is agent-agnostic: any system capable of invoking the underlying Python API s—whether through pre-bound tools or generated code—can be evaluated on the same prompt styles and scoring methodology with minimal adaptation. 

5.3 Limitations and Threats to Validity 

The current implementation and evaluation have several limitations. First, the benchmark problems are limited to Beams2D and Photonics2D and do not cover the full range of EngiBench problems. Second, a user study with practicing engineers who could intervene to re-direct the agents was omitted due to time constraints and the cost of recruiting domain experts, though this could be a promising direction of future work. Third, only four LLM backends were tested, primarily due to API cost constraints; the rapid pace of model releases means that newer models may alter the relative performance rankings reported here. Finally, the HPC training benchmark covers only two generative models ( cGAN and diffusion) and two cloud LLM backends (Gemini-3-Flash and GPT-5-mini), reflecting the high per-model 13 EngiAI: A Multi-Agent Framework and Benchmark Suite for LLM-Driven Engineering Design PREPRINT 

evaluation cost and the prohibitively slow local inference of the open-source 4B-parameter models on our available hardware for the long-horizon HPC workflow; RQ5 conclusions are accordingly scoped to current proprietary LLM s on this specific cluster and training setup. We also do not include a single-agent baseline with identical tool access against the multi-agent supervisor; an empirical ablation of the supervisor decomposition lies outside the scope of the present benchmark study. The present analysis is descriptive: we report means and standard deviations but do not provide statistical significance testing, confidence intervals, or sensitivity analyses across composite weights, prompt phrasings, or temperature settings. Regarding threats to validity: despite setting temperature = 0 , LLM API responses exhibit minor run-to-run variation due to floating-point non-determinism and provider-side batching. We mitigate this by averaging over multiple seeds; the resulting standard deviations (Tables 5–6) are small relative to the performance differences between models, indicating that this noise does not affect the conclusions. 

## 6 Conclusion and Future Work 

In this paper, we introduced a benchmark suite for evaluating LLM -driven engineering workflows along three dimen-sions—workflow execution under distinct cognitive demands, gated retrieval-augmented parameter selection, and HPC training orchestration—together with E NGI AI, a multi-agent reference implementation that operationalizes the bench-mark. The system coordinates seven specialized agents—covering topology optimization, ML -based design generation, document retrieval, web search, HPC job management, local command execution, and 3D printer control—through a supervisor architecture built on LangChain and LangGraph. Our evaluation across three benchmark dimensions and four LLM backends yielded several key findings. On workflow tasks, proprietary models (GPT-5-mini and Gemini-3-Flash) achieved 96–97% average task completion across seven prompt styles targeting distinct cognitive demands, while the open-source 4B-parameter models reached 55% (Qwen3-4B) and 78% (Qwen3.5-4B), demonstrating rapid generational improvement at fixed model scale. The gated RAG scoring mechanism confirmed that RAG is essential for parameter selection from post-cutoff literature, with RAG -on scores approaching 1.0 and near-zero scores without retrieval. The HPC training benchmark demonstrated that the two tested cloud models can autonomously orchestrate end-to-end model training on SLURM clusters, achieving metrics comparable to official baselines. These results indicate that LLM -based multi-agent systems can reliably support structured engineering workflows when backed by sufficiently capable foundation models. Several directions follow naturally from the unresolved aspects of our research questions. 

Broader workflow and model coverage (RQ1–RQ3) Our evaluation covers two physics domains, four models, and two open-source generations from the same family. Extending to additional EngiBench problems, larger open-source models (e.g., 70B parameter), and other model families (e.g., Llama, Mistral) would clarify whether the prompt-style failure-mode and tool over-calling patterns we observe are scale-dependent, family-specific, or general. Sensitivity analysis—varying temperature, prompt phrasing, or tool description verbosity—would further quantify the fragility of current performance levels. On tool efficiency specifically, it remains open whether tool-use patterns transfer across problems and whether techniques such as tool-use demonstrations or constrained decoding can reduce unnecessary invocations for smaller models. Beyond coverage, the evaluation exposes concrete failure patterns that suggest targeted interventions. The tool over-calling observed in smaller models—where excess invocations degrade the combined score from 0.70 to 0.63 without improving design quality (Section 4)—could be mitigated through few-shot tool-use demonstrations in the system prompt or constrained decoding that restricts tool selection based on the current workflow state. The conditional-reasoning failures on W-C OND , particularly for Photonics2D where even the best model reaches only 53%, point to a need for structured chain-of-thought: requiring the agent to explicitly extract the simulation result, state the comparison, and declare the chosen branch before issuing the tool call. The semantic-disambiguation failures on W-D ISTRACT —where models select preview-context values instead of export-context values—suggest that parameter extraction should be decoupled from tool invocation, for instance by first generating a structured parameter plan that can be validated before execution. More broadly, the benchmark itself generates structured tool-call traces with ground-truth validation, which could serve as training data for domain-specific fine-tuning of smaller open-source models—using the scorer’s pass/fail signal as a reward to directly optimize tool-calling accuracy in engineering workflows. 

Scaling retrieval and HPC orchestration (RQ4–RQ5) Our RAG evaluation uses a curated index of two papers; testing with larger, noisier collections where relevant information must be identified among contradictory sources would better approximate real engineering practice. Scaling the index to tens or hundreds of papers would test retrieval precision under realistic conditions where relevant information is sparse relative to the corpus size. A further step is 14 EngiAI: A Multi-Agent Framework and Benchmark Suite for LLM-Driven Engineering Design PREPRINT 

adversarial retrieval evaluation: indexing documents that contain plausible but incorrect parameter values alongside the correct source, testing whether the agent can resolve conflicting evidence—analogous to the W-D ISTRACT style but at the retrieval level rather than the prompt level. Retrieval quality itself could also be improved through domain-specific chunking strategies (e.g., preserving table and equation boundaries) or re-ranking models fine-tuned on engineering literature. For HPC , a more demanding test is whether the agent can autonomously write and iterate on training code rather than orchestrating predefined scripts—shifting the evaluation from workflow orchestration to code generation and self-correction. The degradation observed for GPT-5-mini on long-running HPC pipelines (dropping to 50% on natural-language prompts at the final evaluation step) suggests that explicit state tracking—e.g., a structured checkpoint that records which pipeline stages have completed and their outputs—could help maintain instruction adherence across extended multi-step workflows. Finally, as the tool ecosystem grows via MCP [22 ] or tool API s, a key open question is how agent performance scales with the number of available tools.

A.1 Design Quality Metrics 

The design quality score is a weighted combination of six per-design metrics:

In practice, WT is uniformly 0.00 across all configurations because the threshold-and-extrude STL export produces non-manifold meshes; a post-processing repair step would be needed for watertight output. Weights were assigned by engineering judgment, prioritizing geometric fidelity over engineering performance metrics and printability. A post-hoc Analytic Hierarchy Process ( AHP ) analysis [ 53 ] confirmed consistency ( CR = 0.012 ) and produced weights within ±0.03 of the chosen values. The hierarchical workflow score combines design quality with operational metrics: 

Sworkflow = 0 .65 · Sdesign + 0 .20 · Stool + 0 .15 · Scompletion (2) where Stool is the tool-call efficiency, defined as the ratio of correctly matched calls to the maximum of optimal and actual call counts, and Scompletion ∈ { 0, 1} indicates whether all required tools were called successfully. For styles with STL export (W-R AND through W-M ULTI ), task completion additionally requires that the export parameters match the prompt specifications within a tolerance of ±0.05 for floats and exact match for booleans; a successful tool call with incorrect parameters counts as failed. For the N ATURAL style, when the agent correctly requests clarification (Scompletion = 1 ) and does not produce a design, Sdesign is set to 1.0, reflecting that abstention is the correct behavior for underspecified prompts.
