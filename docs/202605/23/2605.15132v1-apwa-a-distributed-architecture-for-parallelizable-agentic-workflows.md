## 摘要
基于大型语言模型（LLM）的自主多智能体系统已在广泛的应用领域中展现出独立解决复杂任务的卓越能力。然而，随着任务规模和复杂性的增长，这些系统在推理、协作和计算扩展方面面临关键瓶颈。这些限制阻碍了多智能体系统实现高吞吐量处理，尽管底层LLM具备并行计算和推理原语。我们提出了智能体并行工作负载架构（APWA），一种旨在高效处理高度可并行化智能体工作负载的分布式多智能体系统架构。APWA 通过将工作流分解为互不干扰的子问题来促进并行执行，这些子问题可使用独立资源处理，无需交叉通信。它支持异构数据和并行处理模式，适用于广泛领域的任务。在我们的评估中，我们展示了 APWA 能够动态将复杂查询分解为可并行化的工作流，并在先前系统完全失效的场景中对更大规模任务实现扩展。

## Abstract
Autonomous multi-agent systems based on large language models (LLMs) have demonstrated remarkable abilities in independently solving complex tasks in a wide breadth of application domains. However, these systems hit critical reasoning, coordination, and computational scaling bottlenecks as the size and complexity of their tasks grow. These limitations hinder multi-agent systems from achieving high-throughput processing for highly parallelizable tasks, despite the availability of parallel computing and reasoning primitives in the underlying LLMs. We introduce the Agent-Parallel Workload Architecture (APWA), a distributed multi-agent system architecture designed for the efficient processing of heavily parallelizable agentic workloads. APWA facilitates parallel execution by decomposing workflows into non-interfering subproblems that can be processed using independent resources without cross-communication. It supports heterogeneous data and parallel processing patterns, and it accommodates tasks from a wide breadth of domains. In our evaluation, we demonstrate that APWA can dynamically decompose complex queries into parallelizable workflows and scales on larger tasks in settings where prior systems fail completely.

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **背景**：基于大语言模型（LLM）的自主多智能体系统已在软件工程、网络安全、医疗、金融、科学研究等广泛领域展现出独立解决复杂任务的能力。
- **核心瓶颈**：随着任务规模和复杂性的增长，现有系统面临三大关键限制：
  - **推理瓶颈**：LLM 的推理和输出质量随任务规模增大而显著下降，严重时完全崩溃。
  - **上下文窗口限制**：当数据量超出 LLM 的上下文窗口时，系统无法处理。
  - **协调瓶颈**：现有多智能体架构（如 Magentic-One）通过集中式编排器协调所有代理，限制了并行化能力——编排器 LLM 必须维护全局视图，导致可扩展代理数仅在数十到数百量级；且仅允许单代理串行处理信息和通信结果，有效并行化不可实现。
- **研究目标**：设计一种类似 MapReduce 在数据处理领域的范式转变，使多智能体系统能够在数据和执行子任务层面实现高并行化，从而高效处理大规模数据并行或任务并行的复杂工作流。
- **核心创新点**：提出 **Agent-Parallel Workload Architecture（APWA）**，一种专为高度可并行化智能体工作负载设计的分布式多智能体系统架构。

## 2. 论文提出的方法论：核心思想、关键技术细节

### 2.1 系统级抽象

APWA 围绕三个核心抽象展开（与现有系统有根本区别）：

| 抽象 | 角色 | 关键设计 |
|------|------|----------|
| **Manager（管理器）** | 负责整体任务分解与协调 | 动态将任务划分为互不干扰的子任务；仅维护全局视图的唯一组件；执行高级任务推理 |
| **Worker（工作器）** | 负责解决具体子任务 | 大量并行（可达数千个）；维护本地执行状态，与管理器解耦；支持多种复杂度（从简单 LLM 调用到多代理子系统） |
| **Executor（执行器）** | 负责底层分布式执行 | 利用 Ray 分布式计算框架；处理工作器在集群节点上的放置、资源分配、超时重试和故障恢复；抽象分布式计算细节 |

### 2.2 分布式智能体计算抽象

为支持大规模并行工作流，系统引入以下关键抽象：

- **并行规划抽象（Parallel Planning）**：管理器代理初始化时即探索本地状态，生成结构化计划对象（含分区策略和执行步骤列表）；支持并行化引导示例；明确将并行分解作为一等设计要素。此外，引入**输出契约（Output Contract）**概念，以表格规范形式定义任务完成的必要前置条件，防止过早终止任务。

- **子任务委托抽象（Subtask Delegation）**：引入**子任务模板（Subtask Templates）**，描述工作单元（含代理配置、任务描述、数据资源引用），支持占位符自动展开，实现从数据规模中解耦的逻辑内容定义，从而高效批量生成并行子任务。

- **数据表抽象（Data Tables）**：针对数据量超出单台机器内存甚至 LLM 上下文窗口的挑战，引入逻辑只读数据表抽象（类似数据库中的表），通过高度紧凑的元数据表示使 LLM 组件能高效交互超大规模数据。支持完整的表代数操作（过滤、投影、连接、分组、聚合等），所有非叶子表由其**世系（Lineage）**完整描述，确保确定性且无需复杂一致性协议。

- **动态代理能力与预设（Dynamic Agent Capabilities & Presets）**：通过外部托管的**能力注册表（Capability Registry）**动态发现和组合代理能力；管理器代理创建代理预设（定义系统提示和一组能力）以适配具体子任务，实现通用性与动态性的统一。

### 2.3 架构实现

- **执行模式**：支持两种执行模式：
  - **全代理模式**：为每个子任务创建隔离的 Docker 容器执行空间（含领导代理和可选的帮助代理），适用于需要工具访问的复杂子任务。
  - **仅 LLM 模式**：对于简单子任务（如摘要、结构化信息提取），通过单次 LLM 调用执行，吞吐量提升 10–100 倍。管理器根据子任务复杂度动态路由。
- **底层基础设施**：基于 **Ray** 分布式计算框架，提供任务并行编程模型，扩展至数十万并发任务和每秒百万级任务吞吐。
- **容错机制**：子任务级自动重试（最多 3 次），处理瞬态故障（如网络分区），逻辑层错误才上报管理器，确保管理器专注于任务语义而非底层基础设施行为。

## 6. 论文的主要结论与发现

1. **可扩展性优势显著**：在输入数据规模增长近两个数量级（166 kB → 10.5 MB）时，APWA 的 Wall-clock 运行时间仅适度增长（157 s → 329 s），同时运行超过 2500 个并发代理；而所有 Baseline 在大规模输入上完全失败（失败率 60%–100%）。

2. **输出质量保持**：APWA 在大规模任务上仍能保持较高的结构分数和语义分数（如 PII-4096 上结构 0.900、语义 0.544），而 Direct 在 4096 条记录上完全无法处理，MegaAgent 的语义分数始终为 0。

3. **子线性扩展**：WebSurfer 实验中，工作量增加 10 倍（10 → 100 话题）仅导致运行时间增长 4.2 倍，显示出有利的扩展特性。

4. **模型配置影响**：较大的规划模型（GPT-5.4）相比小型模型（GPT-5.4-nano）能带来更高的效用分数（结构与语义），但成本增幅较小，体现出良好的性价比。

5. **Baseline 失败模式分析**：
   - **Direct**：受上下文窗口限制，小规模尚可，大规模完全崩溃。
   - **Magentic-One**：遭遇编排失败（代理间路由错误）和上下文爆炸（在 Roman 语料上消耗超过 6.5M 输入 token）两大失败模式。
   - **MegaAgent**：并行化工作分配不当导致任务过早终止或输出格式严重错误（如生成大文件但 JSON 格式错误）。

## 7. 优点

1. **范式创新**：借鉴 MapReduce 在数据处理领域的思想，首次提出为 LLM 智能体系统引入类似的并行化范式转变，具有概念上的突破性。
2. **抽象设计精妙**：数据表抽象配合世系（Lineage）机制，使 LLM 能够通过紧凑元数据高效操作超大规模数据，解决了"甚至元数据也超出上下文窗口"的关键挑战。
3. **解耦设计**：通过将全局协调（Manager）与本地执行（Worker）解耦，允许工作器在子任务内自主探索，避免了全局协调导致的性能瓶颈。
4. **灵活性与通用性**：支持数据并行、任务并行、复制并行三种模式，并通过 Capability Registry 实现动态能力发现，无需预设特定处理模式。
5. **执行效率优化**：通过仅 LLM 模式和全代理模式的动态路由，在简单任务上实现 10–100 倍吞吐量提升。
6. **容错设计完善**：子任务级自动重试机制将瞬态错误处理封装在执行层，使 Manager 能专注于任务语义。
7. **实验全面**：在多个真实数据集（涵盖 PII 检测、结构化提取、层次摘要、Web 浏览）上系统评估，维度丰富，提供了完整的失败率、运行时间、结构与语义分数数据。

## 8. 不足与局限

1. **工作器间无直接通信**：架构限制工作器之间不直接通信，仅通过 Manager 交互，这对需要子任务间协调的工作流不适用（如某些依赖先前子任务结果的迭代式工作流）。
2. **并行化模式泛化未验证**：论文仅在数据并行、任务并行、复制并行和层次并行场景下评估，未验证 APWA 是否能自然泛化至其他并行化模式。
3. **安全与隐私考虑缺失**：未系统性分析系统面临的攻击面（如提示注入攻击、恶意工具、受损工作器等）和数据泄露风险，这是大规模 LLM 智能体系统的重要隐患。
4. **测试环境局限**：所有实验在单台机器上运行（尽管基于 Ray 可扩展至分布式集群），未能充分展示真实大规模分布式场景下的性能表现。
5. **Baseline 模型不一致**：MegaAgent 使用 GPT-4.1-mini 而非其他基线使用的 GPT-5.4 系列，可能影响公平性比较。
6. **缺乏真实工业级部署验证**：实验主要基于学术基准测试，未在实际生产环境中验证其可靠性和鲁棒性。
7. **能力注册表依赖**：系统依赖外部能力注册表来发现和组合代理能力，若注册表本身存在瓶颈或错误，可能影响整体系统效能。
