## Abstract
Reinforcement learning (RL) is increasingly used to improve the reasoning, coding, and tool-use capabilities of large language models, but agentic RL remains prohibitively expensive. Scaling RL to agentic LLMs requires supporting complex workloads, including multi-policy collaborative training, while efficiently using elastic, heterogeneous, and cross-region compute resources. Existing LLM RL systems support some of these capabilities, but each new extension often requires dedicated system engineering. This burden arises from trainer-centered control architectures and the lack of principled abstractions for RL system components. To address these limitations, we propose AstraFlow, a dataflow-oriented RL system that replaces conventional trainer-centered control with principled component abstractions. In AstraFlow, rollout services, dataflow management, and training are decoupled into autonomous components, enabling the system to natively support complex multi-policy agentic RL workloads and efficiently exploit diverse compute resources. We evaluate AstraFlow across math, code, search, and AgentBench workloads, showing that the same system supports multi-policy training, elastic scaling, heterogeneous cross-region execution, and composable data algorithms without system-level code changes. In multi-policy collaborative training, AstraFlow achieves comparable or better accuracy than existing RL systems while speeding up training time by 2.7x.

## 1. 论文的核心问题与整体含义

**研究动机与背景**：
- 强化学习（RL）已广泛用于提升大语言模型（LLM）的推理、编程和工具使用能力
- 将 RL 扩展到智能体式 LLM（Agentic LLM）代价高昂，需要支持复杂的工作负载，包括多策略协作训练
- 现有 LLM RL 系统（如 verl、AReaL 等）采用**以训练器为中心（trainer-centered）的控制架构**，导致系统僵化，难以扩展
- 缺乏对 RL 系统组件的**原则性抽象**，每添加新功能都需要专门的系统级工程改造

**核心问题**：
- 现有系统无法原生支持多策略协作训练、弹性扩展、异构和跨区域执行
- 训练器与 rollout 执行的紧耦合限制了资源利用效率和多策略协调能力

**核心贡献**：
- 提出 AstraFlow，一种**面向数据流（dataflow-oriented）的 RL 训练系统**，以原则性组件抽象替代传统训练器中心控制

## 2. 论文提出的方法论

**核心思想**：
- 关键洞察：**解耦不应仅限于计算分离（rollout 和 training 分开），还应分离控制职责**
- 数据流导向协调：rollout 服务、训练器和数据流层各自运行自主控制循环，仅通过最小化的数据和权重接口交互
- 将 rollout、训练、数据管理和权重传输转化为可组合的系统边界

**三大核心抽象组件**：

### （1）数据流层（Dataflow Layer）
- **角色**：rollout 服务和训练器之间的协调平面
- **功能**：
  - 管理 prompts、轨迹、元数据、训练批次等 RL 数据的自然单元
  - 通过数据可用性和路由协调自主组件（可调节慢/陈旧 rollout 服务的任务分配、优先传输新鲜轨迹、阻止不合适批次）
  - 在多策略训练中，通过轨迹元数据（策略 ID、模型版本、时间戳、奖励统计等）路由策略特定、共享或混合的数据流
- **数据算法接口**：暴露可编程接口，支持选择性 rollout、课程调度、rollout 后过滤、动态采样、回放、数据混合、陈旧纠正等策略，以数据流策略形式实现，无需修改训练器或系统编排

### （2）Rollout 即服务（RaaS, Rollout-as-a-Service）
- **角色**：将轨迹生成与策略优化解耦
- **核心接口**：服务只需消费任务（prompts），生成轨迹（responses），并通过训练器侧权重传输接口刷新权重
- **特性**：
  - 可热插拔：高效智能体推理引擎可即插即用，无需修改训练器或系统编排
  - 弹性容量：添加/移除 RaaS 节点不影响独立训练器控制循环
  - 天然支持异构和跨区域设置：不同延迟、吞吐量和网络带宽的 rollout 服务可共存

### （3）训练器抽象与权重传输（Trainer Abstraction & Weight Transfer）
- **训练器**：从数据流层消费批次，执行优化并发布更新后的权重
- **特性**：
  - 不直接管理 rollout 工作者或协调其他训练器
  - 可替换：现有 RL、SFT 或容错训练后端可通过相同接口参与
  - 多策略支持：每个策略可有独立训练器和权重流，数据流层控制轨迹分配
- **权重传输机制**：
  - 存储模型版本，向 RaaS 节点暴露最新或请求版本
  - 支持全模型传输、稀疏传输和版本感知刷新
  - **异步拉取（pull）机制**：RaaS 节点在适当时机主动拉取权重，权重传输不在训练器关键路径上
  - 高稀疏性（98.9%~99.6%）保证传输效率

## 6. 论文的主要结论与发现

1. **多策略协作训练效率大幅提升**：
   - 在数学 Solver+Verifier 场景中，AstraFlow 达到可比或更优准确率（平均 56.5% vs. 54.4%）
   - 训练时间缩短 **2.7 倍**（77.65s vs. 212.64s/迭代）

2. **系统灵活性验证**：
   - **零代码自动扩缩**：通过代理维护器（Claude Code）驱动，无需系统级代码修改
   - 自动扩缩达到与 11 GPU 固定池相近的准确率（67.9% vs. 68.0%）和墙钟时间，同时 **降低 13% 总 GPU 小时成本**
   - **异构跨区域训练**：在受约束网络下仍保持 67.6% 准确率（接近本地基线 68.0%）

3. **权重传输稀疏性**：
   - 权重增量稀疏度高达 **98.9%~99.6%**
   - 即使在高学习率（5e-6）下仍保持 >97%，传输量压缩 **30 倍以上**

4. **数据算法可组合性**：
   - GRESO、动态采样（DS）、回放可按需组合
   - DS+GRESO 组合以更少 rollout 达到基线准确率

5. **性能对等性**：
   - AstraFlow 在标准数学任务上与 AReaL 准确率相当（差异 <0.6%），训练效率相近（<2%）

## 7. 优点

1. **原则性抽象设计**：
   - 三大组件（数据流层、RaaS、训练器）通过稳定接口交互，架构清晰可扩展

2. **原生多策略支持**：
   - 首个**完全异步**的多策略协作 RL 框架，无需流水线级修改

3. **弹性与异构支持**：
   - 零代码即可实现自动扩缩容和跨区域异构部署

4. **高效权重传输**：
   - 异步拉取稀疏权重传输，传输不在训练器关键路径上

5. **数据算法模块化**：
   - 数据流层暴露可编程接口，数据算法可插拔组合

6. **评估全面**：
   - 覆盖数学、代码、智能体任务；对比现有系统验证；消融实验完整

7. **实用性**：
   - 开源（GitHub）且提供网站文档；可复用现有 SGLang 等推理引擎作为 RaaS

## 8. 不足与局限

1. **算法层面的局限**：
   - AstraFlow 专注于系统抽象，未提出新 RL 优化算法

2. **实验覆盖范围**：
   - 未覆盖所有智能体 RL 设置：长时序网页智能体、机器人环境、安全关键交互系统
   - 智能体任务仅测试 AlfWorld、WebShop、Search，未扩展到更多复杂场景

3. **模拟与受控设置**：
   - 跨区域实验使用网络模拟工具（tc/netem），与真实生产环境可能存在差距

4. **收益依赖性**：
   - 观察到的好处可能依赖工作负载特性（rollout 延迟、训练器吞吐量、网络带宽等）

5. **数据表示假设**：
   - 假设 rollout 任务、轨迹元数据和奖励信号可通过数据流层表示；复杂非结构化数据可能需额外适配

6. **安全性与质量**：
   - 虽然 AstraFlow 使多策略和数据驱动 RL 工作负载更易组合，但模型质量、鲁棒性和安全性仍取决于底层数据、奖励、策略和评估协议
   - 更高效的 RL 基础设施可能加速更强智能体模型的开发，存在滥用风险
