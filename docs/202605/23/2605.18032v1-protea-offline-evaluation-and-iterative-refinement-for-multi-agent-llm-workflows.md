## 摘要
多智能体 LLM 工作流——由多个角色特定 LLM 调用组成的系统——通常优于单一提示基线，但它们仍然难以调试和优化。故障可能源于中间输出中的细微错误，这些错误会传播到下游节点，要求开发者检查冗长的追踪记录并推断需要修改哪个智能体。我们提出了 PROTEA，一个用于离线、测试驱动的多智能体工作流改进的统一界面。PROTEA 执行工作流，使用可配置的评分标准对中间节点输出进行评分，并将每个节点的状态和推理过程叠加在工作流图上，以定位可能的瓶颈。为支持以最终答案参考作为主要监督的复杂系统，PROTEA 执行后向节点评估：它从最终答案参考和图结构中生成候选节点级期望，然后将其与观察到的节点输出进行比较。对于选定的节点，PROTEA 以可编辑的前后对比形式呈现定向提示修订，然后在同一界面中自动重新运行并重新评估工作流，以显示输出变化和评分轨迹。在两个生产相关的工作流中，PROTEA 将文档检查准确率从 64.3% 提高到 83.9%，将推荐 Hit@5 从 0.30 提高到 0.38。在一项由六位经验丰富的 LLM 开发者参与的形成性研究中，参与者对图级定位、节点级推理以及可编辑的前后提示修订给予了高度评价。

## Abstract
Multi-agent LLM workflows -- systems composed of multiple role-specific LLM calls -- often outperform single-prompt baselines, but they remain difficult to debug and refine. Failures can originate from subtle errors in intermediate outputs that propagate to downstream nodes, requiring developers to inspect long traces and infer which agent to modify. We present PROTEA, a unified interface for offline, test-driven improvement of multi-agent workflows. PROTEA executes a workflow, scores intermediate node outputs with configurable rubrics, and overlays per-node states and rationales on the workflow graph to localize likely bottlenecks. To support complex systems where final-answer references are the primary supervision, PROTEA performs backward node evaluation: it generates candidate node-level expectations from final-answer references and graph context, then compares them with observed node outputs. For selected nodes, PROTEA presents targeted prompt revisions as editable before/after comparisons, then automatically reruns and re-evaluates the workflow to show output changes and score trajectories within the same interface. In two production-adjacent workflows, PROTEA improved document-inspection accuracy from 64.3% to 83.9% and recommendation Hit@5 from 0.30 to 0.38. In a formative study with six experienced LLM developers, participants valued graph-level localization, per-node rationales, and editable before/after prompt revisions.

## 2. 方法论

### 核心思想

PROTEA围绕三个核心机制设计：

1. **后向节点评估（Backward Node Evaluation）**：在仅有最终答案参考的情况下，从最终输出反向生成各中间节点的期望输出。
2. **图级诊断支持**：将节点状态（PASS/WARN/FAIL）和评估理由叠加到工作流图上，帮助开发者快速定位瓶颈。
3. **验证型提示优化**：提供可编辑的前后对比提示修订，自动重新运行并评估工作流以验证效果。

### 关键技术细节

#### 2.1 工作流表示

- 工作流被建模为**有向无环图（DAG）**：节点 = 角色特定的 LLM 调用（或带工具的智能体），边 = 中间输出的传递。
- 每个节点包含：指令 $I_v$（prompt）、可选的输出格式要求 $S_v$、评估标准（rubrics）。

#### 2.2 后向节点评估流程

设工作流为 $G = (V, E)$，节点 $v \in V$ 的参考输出 $y^*_{ref,v}$ 定义如下：

- **最终节点**：直接使用最终答案参考 $y^*_{final}$；若不可用，则用人工提供的节点参考 $y^*_v$；否则由 PROTEA 生成。
- **中间节点**：若有人工节点参考则使用；否则通过 **后向生成模块** $M_{gen}$ 从最终答案参考和下游节点需求生成候选期望 $\hat{y}_v$。
- **后向生成模块** $M_{gen}$ 的输入包括：当前节点指令 $I_v$、输出格式 $S_v$、节点在图中的位置、下游子节点的需求、以及最终答案参考（若有）。生成时综合所有直接下游子节点的需求，输出一个支持所有下游路径的期望输出。

#### 2.3 节点评估

- **评估模块** $M_{eval}$：给定节点输出 $y_v$ 和参考 $y^*_{ref,v}$，对每个评估标准 $d$ 产出分数 $\sigma_d(v) \in [0,1]$，加权得到总分 $s(v) \in [0,1]$。
- **状态映射**：通过可配置阈值（默认 PASS ≥ 0.8，WARN ≥ 0.55，FAIL < 0.55）将分数映射为 PASS/WARN/FAIL。
- **输出**：评估分数、简短理由 $R_v$、改进建议。

#### 2.4 提示修订模块

- **提示修订模块** $M_{opt}$：接收当前指令 $I_v$、评估理由和改进建议，输出修订后的指令 $I'_v$ 及变更说明。
- **约束**：保留变量（用户查询、父节点输出）、保持输出格式稳定、避免复制测试内容。
- **防复制检查**：自动拒绝包含测试问题直接复制内容的修订建议。

#### 2.5 迭代流程

1. 运行工作流获取节点输出。
2. 执行后向评估（可选）并评估所有节点。
3. 在图上高亮低分节点，开发者选择瓶颈节点。
4. 查看该节点的输出、参考、理由和修订建议。
5. 编辑（接受/修改/放弃）提示修订。
6. 保存后自动重新运行工作流并重新评估。
7. 在历史视图中比较分数轨迹和输出变化。

- **AUTO LOOP 模式**：自动重复 evaluate→revise→re-evaluate 循环，保留仅在重复检查中显示改进且稳定的修订。

## 3. 实验设计

### 3.1 生产级工作流案例研究

#### Case Study A：企业文档检查

- **工作流结构**：5 节点 DAG（多智能体）。
- **任务**：检查文档是否满足严格策略/风格约束，逐项输出判断及理由。
- **初始状态**：开发者手写的初始提示，非专门为本研究生成。
- **评估指标**：逐项正确率（二值匹配人工标签）。
- **结果**：经过 PROTEA 迭代优化，准确率从 **64.3% 提升至 83.9%**。
- **主要改进类型**：
  - 使中间输出更显式。
  - 收紧节点特定评分标准。
- **协议**：评估和后向期望生成自动化，提示修订由人工撰写和审批。

#### Case Study B：会话式推荐/匹配

- **工作流结构**：6 节点 DAG。
- **任务**：澄清模糊请求、生成候选项、排序、生成最终响应。
- **评估指标**：Hit@5（Top-5 推荐中是否包含目标项）。
- **初始状态**：开发者手写提示。
- **结果**：Hit@5 从 **0.30 提升至 0.38**。
- **协议**：同 Case Study A。

### 3.2 自动迭代压力测试

- **工作流来源**：11 个由 LLM 仅使用工具文档独立生成的 multi-agent 工作流（未参与 PROTEA 开发）。
- **定量分析子集**：5 个初始提示故意设计为极简的工作流（留有自动优化空间）：
  1. HTTP 日志分类（Log triage）
  2. 课程排课（Course scheduling）
  3. 事故工单生成（Incident ticket）
  4. 多步文字问题（Word problem）
  5. 拒绝/澄清支持响应（Refuse/clarify）
- **对比基线**：禁用提示重写的原始工作流三次运行均值。
- **设置**：AUTO LOOP 运行 3 次迭代，每个工作流使用第一个查询。
- **结果**：

| 工作流 | 无重写基线 | 最佳分数 | 提升 |
|--------|-----------|---------|------|
| Log triage | 0.307 ± 0.029 | 0.648 | +0.341 |
| Scheduling | 0.186 ± 0.001 | 0.800 | +0.614 |
| Incident ticket | 0.333 ± 0.110 | 0.840 | +0.507 |
| Refuse/clarify | 0.208 ± 0.027 | 0.390 | +0.182 |
| Word problem | 0.000 ± 0.000 | 0.000 | 0.000 |

- **分析**：5 个工作流中 4 个超过无重写基线，3 个提升超过 0.3 分；Word problem 未改进，因为其评分标准接近二值（最终答案错误即得零分），优化器收到的梯度信号不足。

## 6. 主要结论与发现

1. **效果提升显著**：
   - 文档检查准确率从 64.3% 提升至 83.9%（+19.6%）。
   - 推荐 Hit@5 从 0.30 提升至 0.38（+0.08）。
   - 自动迭代压力测试中，5 个极简提示工作流中 4 个超过基线，3 个提升超过 0.3 分。

2. **主要改进模式**：
   - 使中间输出更显式、更结构化。
   - 收紧节点特定的评分标准（rubrics）。
   - 明确约束传递，减少下游歧义。

3. **用户价值认可**：
   - 开发者高度评价**图级定位**（快速锁定瓶颈节点）、**节点级理由**（使评估结果可操作）、**前后对比的可编辑提示修订**（减少工具切换）。
   - 提出了采用优先级：评估器校准、现有工具链集成、团队协作和可重现性、成本/延迟优化。

4. **AUTO LOOP 的局限**：
   - 在近二值评分标准（如精确匹配）下效果有限，因为优化器收到的梯度信号不足以区分部分改进。

## 7. 优点

1. **统一的开发者工作流**：将执行、评估、诊断、提示修订、验证集成在同一界面，避免工具切换开销。
2. **后向节点评估**：巧妙解决"仅有最终答案标签"这一现实约束，无需为每个中间节点人工标注参考输出。
3. **图级可视化诊断**：将抽象的节点状态叠加到工作流图上，直观引导开发者注意力，降低调试认知负担。
4. **可控的提示优化**：提示修订以可编辑的前后对比呈现，开发者始终掌控决策权，结果通过自动重新评估验证。
5. **AUTO LOOP 支持**：在不要求人工干预的场景下，可自动执行多轮提示优化并保留有效修订。
6. **实用性设计**：保留变量检查、防复制机制、格式稳定性约束等细节，确保提示修订不引入运行时错误。

## 8. 不足与局限

1. **实验覆盖有限**：
   - 案例研究工作流数量少（各 1 个），且受保密限制缺乏细节。
   - 用户研究仅 6 人，缺乏统计显著性。

2. **缺乏关键指标测量**：未量化"修复时间"、"回归率"等开发者关心的实际指标。

3. **结构覆盖局限**：当前仅支持 DAG 结构，循环控制流、主管协调、长期交互式智能体等场景未覆盖。

4. **AUTO LOOP 的二值评分问题**：近二值评分标准（如精确匹配）下，优化器难以获取有效梯度信号，导致自动改进失效。论文建议通过引入部分信用标准（如约束满足、中间事实正确性）缓解。

5. **评估器设计依赖**：Rubric-based LLM 评估器的质量影响整体效果，论文承认需要更细致的评估器设计、法官校准和多人一致性检查。

6. **未涉及架构级修改**：目前仅支持节点内提示修订，不支持添加节点、重连边等架构层面的改动。

7. **成本/延迟**：重复离线评估在大规模测试套件上的计算成本和延迟未被充分讨论。

8. **模型无关性**：未系统测试不同骨干模型对框架效果的影响。
