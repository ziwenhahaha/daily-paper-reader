## 摘要
最近的多智能体系统在解决复杂任务方面展现出巨大潜力。然而，当多个智能体并发编辑共享代码库时，其更改可能悄无声息地产生冲突，不一致的视图导致集成失败。现有的多智能体系统通过工作空间隔离（例如每个智能体一个 git worktree）来解决这一问题，但这只是把冲突解决推迟到事后合并阶段，而此时恢复代价高昂。本文提出 STORM，即面向状态的多智能体协作管理（STate-ORiented Management）。具体而言，STORM 通过调节智能体与共享工作空间的交互来管理智能体状态，确保每个智能体在代码库的一致视图上操作，并在写入时检测并解决冲突。我们在 Commit0 和 PaperBench 基准上针对多种 LLM 评估了 STORM。STORM 在 Commit0‑Lite 上比基于 git‑worktree 的多智能体基线高出 18.7 分，在 PaperBench 上高出 1.4 分，同时实现相当或更优的成本效益。结合单智能体运行，STORM 在两项基准上分别取得 87.6 和 78.2 的最高分，表明显式的状态管理比工作空间隔离更适合作为多智能体协作的基础。STORM 还能无缝嵌入任意多智能体系统。

## Abstract
Recent advances in multi-agent systems have shown great potential for solving complex tasks. However, when multiple agents edit a shared codebase concurrently, their changes can silently conflict and inconsistent views lead to integration failures. Existing multi-agent systems address this through workspace isolation (e.g., one git worktree per agent), but this defers conflict resolution to a post-hoc merge step where recovery is expensive. In this paper, we propose STORM, i.e., STate-ORiented Management for multi-agent collaboration. Specifically, STORM manages agent states by mediating their interactions with the shared workspace, ensuring that each agent operates on a consistent view of the codebase and that conflicting edits are detected and resolved at write time. We evaluate STORM on Commit0 and PaperBench across multiple LLMs. STORM outperforms the git-worktree-based multi-agent baseline by +18.7 on Commit0-Lite and +1.4 on PaperBench, while achieving comparable or better cost efficiency. Combined with single-agent runs, STORM reaches highest scores of 87.6 and 78.2 on the two benchmarks respectively, suggesting that explicit state management is a more effective foundation for multi-agent collaboration than workspace isolation. STORM can also be plugged into any multi-agent system seamlessly.

# 论文详细总结：Multi-agent Collaboration with State Management

## 1. 核心问题与整体含义（研究动机和背景）

**核心问题**：多智能体系统并发编辑共享代码库时，如何高效检测与解决冲突。

**研究背景**：
- 多个LLM智能体并行工作可以解决单智能体无法完成的复杂任务（如软件工程、科学研究）
- 当一个智能体编辑文件时，其推理依赖于其他文件的上下文，这些假设可能在并发环境中失效
- 现有主流方法采用工作空间隔离（如每个智能体一个git worktree），但这只是将冲突推迟到合并阶段
- 文本冲突容易检测，但语义冲突（各自通过测试但合并后失败）难以自动解决
- 事后合并代价高昂，恢复困难

## 2. 论文提出的方法论：核心思想与技术细节

### 核心思想

**STORM（STate-ORiented Management）**通过状态管理中介智能体与共享工作空间的交互，替代传统的工作空间隔离方案。核心洞察：智能体不需要全局一致的代码库视图，只需其实际读取的文件保持不变即可。

### 关键技术

**本地状态一致性（Local State Consistency）**：
- 工作空间 W = {(f, v_f) | f ∈ F}，每个文件维护单调递增的版本号
- 智能体维护读取快照 S_i = {(g, v_obs_g)}，记录读取的文件及版本
- 写入有效性条件：∀(g, v_obs_g) ∈ S_i : v_obs_g = v_cur_g（即读取的文件未被修改）

**写入时冲突控制**：
- 采用乐观并发控制思想：允许并发操作无阻塞执行，仅在冲突实际发生时干预
- 每个文件维护版本计数器（vf），读取返回内容与版本，写入需声明预期版本
- 冲突拒绝后返回：当前目标文件内容、变更diff、过期依赖列表
- 拒绝后授予短期预留（Reservation）防止循环冲突

**意图标注（Intent Annotations）**：
- 结构化注释 `#engineer_id: <short intent>`，标识作者和代码块目的
- 智能体读取文件时可见其他智能体的意图，实现语义层面的协调
- 降低语义冲突（相同函数不同签名、共享数据结构的冲突假设）

### 算法流程

```
Manager Agent
├── 扫描代码库结构，分析依赖
├── 将任务分解为不相交的子任务集
├── 分配给并行 Engineer Agents
└── 审查提交，运行测试，提交更改

Engineer Agent
├── 通过 STORM 读取文件（获取内容+版本）
├── 维护本地读取快照
├── 写入时提交预期版本
└── STORM 验证快照一致性后执行写入
```

## 6. 论文的主要结论与发现

### 主要结论

1. **STORM全面优于基线**：在Commit0-Lite上，STORM比GitWorktree高18.7分（Sonnet: 46.2 vs 24.6），PaperBench上高1.4分（74.1 vs 72.7）

2. **显式状态管理优于隔离**：共享工作空间+写入时冲突检测，比事后合并更高效

3. **可扩展性**：工程师从2增至8，Score_w从38.2%升至69.7%（+31.5），耗时几乎不变（并行执行）

4. **高耦合场景优势显著**：GitWorktree在高度耦合仓库上崩溃至36.3%，STORM保持70.9%

5. **成本效率接近单智能体**：STORM-Combined在Commit0-Lite上达到87.6%（Qwen），几乎无额外开销

6. **任务分解质量是瓶颈**：增加工程师带来的收益递减源于分解质量，而非STORM本身

### 关键发现

- **冲突前置检测**：STORM将冲突从"事后合并"移至"写入时检查"，早期解决与更高通过率相关
- **意图标注有效**：无标注时Score_w下降至26.6%，标注后提升至46.2%
- **单智能体仍有优势**：在简单自包含仓库上，单智能体与STORM相当（如chardet: 99.7% vs 22.1%，反而单智能体更优）

## 7. 优点

### 方法亮点

1. **理论严谨**：形式化定义"本地状态一致性"，将冲突检测归结为版本快照验证
2. **框架通用性**：STORM仅中介文件操作，可无缝嵌入任意多智能体系统
3. **实现完整**：拒绝机制提供完整上下文（当前内容、diff、过期依赖），便于智能体重试
4. **预留机制**：防止反复交替冲突，提升稳定性

### 实验亮点

1. **多模型验证**：跨越三大厂商模型（Sonnet/Qwen/DeepSeek），结果可信
2. **多种隔离策略对比**：软隔离（prompt约束）、GitWorktree（物理隔离）、STORM（状态管理）
3. **规模扩展实验**：验证STORM随智能体数增加的性能表现
4. **失败分析深入**：将失败归类为语义错误、API缺失、范围漂移等，定位系统局限

## 8. 不足与局限

### 方法局限

1. **文件级粒度限制**：两个智能体编辑同一文件的不同函数也会触发误报；共享文件（如`__init__.py`）成为序列化瓶颈
2. **终端绕过**：STORM仅中介`file_editor`工具，不拦截直接文件系统写入（bash中的sed/echo）
3. **无命令协调**：并发shell命令（如格式化器）未被序列化，可能产生竞争条件

### 实验覆盖局限

1. **算力信息缺失**：未报告GPU型号、数量、训练时长等，影响可复现性
2. **Benchmark有限**：仅两个基准（Commit0-Lite、PaperBench），覆盖场景不够全面
3. **长尾场景未测**：高耦合场景下仍有约29%失败率，未深入分析根因
4. **语义冲突检测**：意图标注仅提供协调线索，无法完全消除语义冲突

### 应用限制

1. **非文本文件**：STORM面向代码编辑，不适用于图片、音频等多模态数据编辑
2. **跨仓库协作**：当前设计针对单一代码库，未考虑分布式多仓库场景
3. **动态依赖**：运行时依赖（如动态导入）无法被文件级版本追踪覆盖
