## Abstract 

Large language model (LLM) agents excel at solving complex long-horizon tasks through autonomous interaction with environments. However, their real-world deployment faces a fundamental device–cloud dilemma : on-device models are efficient but often brittle, while cloud models are stronger but costly in computation. State-of-the-art LLM device–cloud routers usually make coarse task-level decisions, which cannot adapt to the changing difficulty of multi-step agent interactions. To address this issue, we present Hera, a step-level device–cloud LLM agent coordinator for long-horizon tasks achieving a strong performance–cost Pareto frontier. Hera adopts a novel two-stage training paradigm: (1) imitation learning for cold-start, followed by (2) reinforcement learning that jointly optimizes task success and cloud usage efficiency. The first stage casts step-level routing as a supervised classification problem: the device agent is replayed on cloud trajectories, with each state labeled by the agreement between device and cloud actions. In the second stage, we perform cost-aware reinforcement learning by grouping identical states across trajectories and updating Hera with labels favoring higher expected return and fewer future cloud calls. We evaluate Hera on ALFWorld, WebShop, and AppWorld, where it consistently outperforms prior methods, achieving 92.5% of the cloud-only success rate with cloud use in only 46.3% of steps. 

## 1 Introduction 

Recent advances in large language models (LLMs) (Singh et al., 2025; Yang et al., 2025a; Liu et al., 2024), combined with agentic frameworks such as ReAct (Yao et al., 2023), have empowered LLM agents to perform complex, long-horizon reasoning and decision-making well beyond static problem solving (Chang et al., 2024). These agents operate in open-ended environments through multi-step interaction loops, using step-wise feedback to guide subsequent actions, and have demonstrated strong performance across diverse domains, including software engineering (Zhang et al., 2024; Yang et al., 2025b), GUI navigation (Vattikonda et al., 2025; Gur et al., 2023), interactive gameplay (Wang et al., 2023; Tan et al., 2025), and robotic control (Zitkovich et al., 2023; Han et al., 2025). Despite their promise, LLM agents in real-world environments face a fundamental device–cloud trade-off. On one hand, deploying advanced agents directly on user-end is infeasible due to stringent resource constraints. Small models (e.g., <10B parameters) that fit within device memory limits typically exhibit significant performance degradation in complex reasoning tasks, often failing to maintain long-term coherence in multi-step trajectory planning (Nozaki et al., 2025; Ong et al., 2025a).

To address this, Hera is proposed as a lightweight step-level device–cloud coordinator trained via a two-stage paradigm: imitation learning for cold-start, followed by reinforcement learning for performance–cost co-optimization. On the other hand, reliance on powerful cloud-based closed-source models introduces substantial network latency and high usage cost, limiting their practicality for real-time applications such as robotic manipulation and interactive GUI navigation (Ravichandran et al., 2025; Wanyan et al., 2025). For example, a cloud-based drone navigation agent may miss dynamic obstacles due to network jitter, whereas an on-device model may lack the “System 2” reasoning depth to plan an optimal global path. To balance local efficiency and global performance, recent work dynamically allocates inference between device- and cloud-side LLMs (Narayan et al., 2025; Li et al., 2025). Existing methods treat each request as an atomic unit, making a one-shot routing decision before inference begins by assign-ing the entire query to a single model based on estimated difficulty, confidence, or preference (Ong et al., 2025a; Zhao et al., 2024). While effective for standalone tasks such as question answering, summarization, and code completion, this paradigm is poorly suited to long-horizon agent settings, where execution unfolds as a sequence of interdependent steps whose difficulty changes with the evolving environment state (Feng et al., 2025; Dong et al., 2026). To address this limitation, this paper investigates step-level device–cloud routing for long-horizon LLM agent tasks, aiming to better balance task success and inference cost through routing decisions made at each environment interaction rather than once per task. Importantly, we find that offloading only a small fraction of critical steps along the trajectory to the cloud is sufficient to match the task success of cloud-only execution (§4). Building on this insight, we propose Hera, a lightweight yet powerful device–cloud coordinator that identifies when cloud assistance is necessary, reducing cloud usage while maintaining strong task completion. Hera follows a two-stage training paradigm, as shown in Figure 1: (1) imitation learning for routing cold-start (§5.1), and (2) reinforcement learning for cost-performance co-optimization (§5.2). We initialize Hera via imitation learning by replaying the device agent on cloud trajectories and labeling each step by device–cloud action consistency. We then refine Hera with cost-aware reinforcement learning: across multiple rollouts, we group identical states and assign step-level preference labels that prioritize higher estimated return and lower expected future cloud usage. Across various challenging agent benchmarks including ALFWorld (Shridhar et al., 2021), WebShop (Yao et al., 2022), and AppWorld (Trivedi et al., 2024), Hera consistently outperforms existing methods, demonstrating superior success–cost co-optimization. Our contributions are threefold: • To the best of out knowledge, Hera is the first step-level device–cloud agent coordinator, addressing the performance–cost trade-off of deploying LLM agents in long-horizon tasks. • We develop a two-stage training framework for Hera: an imitation-learning stage that initial-izes routing from device–cloud action agreement, followed by a state-grouped reinforcement learning stage that jointly optimizes trajectory return and cloud cost. • Experiments on various agent benchmarks show that Hera outperforms state-of-the-art methods, achieving a stronger trade-off between task success and cloud usage. 

## 2 Related work 

LLMs agents. Through carefully designed prompting (Yao et al., 2023; Shinn et al., 2023), en-hanced memory and retrieval mechanisms (Wang et al., 2024; Xu et al., 2025), and integration with external tools (Schick et al., 2023; Xie et al., 2024b), LM agents are capable of executing 2complex, multi-step tasks that were previously beyond reach (Chang et al., 2024; Xie et al., 2024a; Wang et al., 2025). Recent work uses supervised fine-tuning (Xia et al., 2025; Zhang and Zhang, 2024; Li et al., 2026) and, in particular, reinforcement learning (Feng et al., 2025; Mai et al., 2025; Zhang et al., 2026) to enable LLM agents to learn directly from environment interactions rather than static prompts or handcrafted workflows, significantly enhancing long-horizon reasoning and multi-step decision-making and forming a promising paradigm for general-purpose autonomous intelligence (Comanici et al., 2025; Qwen Team, 2026; Liu et al., 2025). However, LLM agents face a fundamental device–cloud dilemma: user-end devices are constrained in memory and energy, limiting LLM scale and performance, while offloading to cloud models adds latency, hindering latency-sensitive applications such as robotics and mobile assistants, and driving up operational costs. 

Device–cloud LLM collaboration. Recent work has explored device–cloud LLM routing, which selectively dispatches each request to models of different capacities to trade off output quality against latency and cost. FrugalGPT (Chen et al., 2024) and Hybrid LLM (Ding et al., 2024) route requests based on estimated quality–cost trade-offs, while Eagle (Zhao et al., 2024) and RouteLLM (Ong et al., 2025b) use historical similarity or preference-based win prediction to select the stronger model under a budget. Another line of work explores tighter device–cloud collaboration, where the cloud decomposes or verifies subtasks and the device executes them over local context to reduce cloud cost. For example, MinionS (Narayan et al., 2025) lets the cloud decompose or verify subtasks while the device executes them over local context to reduce cloud cost. However, these methods rely on request-level routing or coarse subtask decomposition, making them ill suited to long-horizon agent tasks, where evolving environments and step-varying difficulty demand finer-grained coordination. 

## 3 Preliminaries 

Problem setup. We consider a finite-horizon, multi-step agentic task setting in which an LLM-parameterized policy πθ governs an agent interacting with an environment to accomplish a task specified by a description x ∈ X . At each step t ∈ { 1, . . . , T }, the agent observes st ∈ S 

and samples a textual action at ∈ V n according to the conditional distribution πθ (at | st, x ),where V is the LLM vocabulary and n is the maximum output length. The environment subse-quently returns a scalar reward rt ∈ R and the next state st+1 . An episode produces a trajectory 

τ = {(s1, a1, r 1), (s2, a2, r 2), . . . , (sT , aT , r T )} with an associated return R(τ ) (e.g., verifiable, outcome-level task feedback), which is often used to optimize the LLM policy to maximize the expected return of the agent’s trajectories (Schulman et al., 2017; Shao et al., 2024). 

Device–cloud collaboration. Extending the single-agent paradigm, we formalize the device–cloud collaborative system: the device policy πθd performs low-latency inference using a lightweight LLM (typically <3B parameters) under strict memory and compute constraints, while the cloud policy πθc

exploits massive scale (hundreds of billions of parameters) to deliver superior reasoning capabilities at the cost of non-negligible latency and high resource overhead. Let rϕ : S × X → { 0, 1} denote a device–cloud router parameterized by ϕ, where for a task–state pair (s, x ), rϕ(s, x ) = 0 selects local processing by πθd , and rϕ(s, x ) = 1 escalates the request to πθc . Thus, the final action of device–cloud collaborative agents under (s, x ) is given by:

## 4 Do We Need the Cloud LLM for Every Step? 

This section presents an in-depth investigation of the performance gap between device and cloud LLM agents on long-horizon tasks. We use Qwen2.5-7B-Instruct (Yang et al., 2024) as the device model and its closed-source counterpart Qwen-Max (Qwen Team, 2025) as the cloud model, representing a resource-efficient model for local deployment and a compute-intensive, state-of-the-art oracle, respec-tively. We evaluate them on two demanding long-horizon agent benchmarks: ALFWorld (Shridhar et al., 2021), which focuses on embodied household task completion, and WebShop (Yao et al., 2022), which requires multi-step e-commerce navigation and decision-making. More experimental details are shown in § 6.1 and Appendix C. Individually, device and cloud LLMs achieve success rates of 48.6% (3.8%) and 90.9% (35.6%) on ALFWorld (WebShop), respectively, revealing a substantial task-level performance gap. However, not all steps in device-failed trajectories require cloud intervention. As shown in Figure 2a, for tasks where the device fails but the cloud succeeds, we replay the cloud trajectory on the device by providing the same state at each step, and find that 34.9% (39.3%) of steps produce identical outputs on ALFWorld (WebShop). This suggests that an ideal step-level device–cloud agent router would need to invoke the cloud on <25% of steps to match cloud-only task success. We further analyze device–cloud step-level agreement across multiple dimensions, including trajectory position, reasoning length, and predictive entropy. Figure 2b shows the step match rate across trajectory positions (up to 50 steps for ALFWorld and 20 for WebShop). Notably, the match rate varies with step index rather than remaining stable, reflecting heterogeneous task structure and stage-dependent reasoning difficulty. For instance, in ALFWorld, early steps primarily involve environment exploration, while intermediate stages require more complex planning and long-horizon reasoning, leading to increased device–cloud divergence. Case studies are provided later to further illustrate these patterns. Figures 2c and 2d quantify the relationship between device–cloud step-level agreement and action difficulty from the perspectives of reasoning length and predictive entropy, respectively. Reasoning length reflects the depth of deliberation required at each step, while sequence-level predictive entropy captures model uncertainty and is computed as the average token-level entropy:

Interestingly, the device–cloud reasoning-length gap differs markedly 4across the two benchmarks. On ALFWorld, the structured environment and limited action space allow the device model to act more directly, though often without capturing deeper dependencies handled by the cloud. In contrast, WebShop presents a large, unstructured action space—e.g., free-form queries and diverse attribute parsing—causing the weaker device model to produce more verbose deliberation, while the stronger cloud model remains more concise and efficient. Beyond reasoning length, entropy also exhibits a clear separation: the device model is more uncertain on mismatched steps and more confident on those aligned with the cloud, suggesting that the need for cloud intervention is structured by step difficulty rather than random chance. Moreover, entropy exhibits a similar benchmark-dependent pattern: cloud interventions reduce device entropy on ALFWorld, indicating stronger alignment with device reasoning, but increase it on WebShop, where cloud decisions more often fall outside the device model’s familiar decision regime. While the device agent can handle many routine follow-up actions, its gap from the cloud agent is concentrated in decisions requiring deeper context tracking, precise grounding, or complex constraint composition. Accurately identifying these steps could substantially reduce cloud calls while preserving task success, highlighting the promise of step-level coordination. 

## 5 Hera: A Long-Horizon Device–Cloud LLM Agent Coordinator 

This section presents Hera, a step-level device–cloud agent coordinator that optimizes Eq. 2 to maximize task performance while minimizing cloud reliance. Hera consists of a 500M-parameter decoder-only Transformer with a binary classification head to route between device agent πθd and cloud agent πθc . The training of Hera follows a two-stage pipeline (pseudocode in Appendix B): acold-start phase with imitation learning , followed by a refinement phase with reinforcement learning ,ultimately enabling effective performance–cost co-optimization in device–cloud LLM agent systems.

## 6 Experiment 

We conduct comprehensive evaluations of Hera across diverse agentic tasks to show: (1) the strong device–cloud coordination capability of Hera for an optimal performance–cost trade-off; (2) the contribution of each training stage through ablation studies; (3) microbenchmarks, including cloud offloading rate, compute overhead, and analyses of routing patterns. 

6.1 Experimental Settings Benchmarks. We evaluate Hera on three challenging long-horizon agent benchmarks: ALF-World (Shridhar et al., 2021), WebShop (Yao et al., 2022), and AppWorld (Trivedi et al., 2024). ALFWorld is an embodied benchmark with 3,827 tasks across six household activity categories, designed to evaluate multi-step decision-making and interactive control for LLM agents. WebShop is a web-based benchmark for LLM agents on realistic shopping tasks, featuring over 1.1M products and 12k user instructions in a large, diverse action space. AppWorld is a high-fidelity multi-app environment with 9 daily apps and 457 APIs, offering 750 interactive coding-centric agent tasks with robust programmatic evaluation via state- and execution-based unit tests. 

Baselines. We compare Hera with a range of competitive baselines: (1) device-only and cloud-only LLM agents; (2) stochastic step-level routing that randomly selects between device and cloud agents with predefined probabilities; (3) state-of-the-art device–cloud LLM routing methods, including Eagle (Zhao et al., 2024), FrugalGPT (Chen et al., 2024), Hybrid LLM (Ding et al., 2024), and RouteLLM (Ong et al., 2025b). Eagle retrieves similar historical queries and combines global and local ELO scores (Elo, 1967) to route each request to the best LLM. FrugalGPT first routes queries to a device-side model and escalates to a stronger cloud LLM only when a learned scorer deems the response unreliable. Hybrid LLM trains a lightweight router to predict query difficulty and dynamically assign each request to device or cloud model. RouteLLM trains a win-prediction router on preference data and routes to the cloud model when its win probability exceeds a threshold. 

Implementation details. We use the Qwen2.5-7B-Instruct (Yang et al., 2024) as the on-device model and the closed-source Qwen-Max API (Qwen Team, 2025) as the strong cloud oracle, respec-tively (other model pairs in Appendix C.3). Hera is initialized from the Qwen2.5-0.5B backbone (Yang et al., 2024) and replaces the LM head with a binary classification head, resulting in a 494M-parameter model. Training is conducted on 2 ×H100 GPUs, freezing the backbone and updating only the last two attention layers and the classification head (30M parameters). We use AdamW (Loshchilov and Hutter, 2019) optimizer, with learning rates of 4e − 5 (IL) and 1e −5 (RL), batch sizes of 64 (IL) and 256 (RL). We set N = 8 , δ = 0 .5, γ = 1 .3, ϵ = 0 .05 , and β = 0 .1. After training, we use the MNN inference engine (Jiang et al., 2020) to deploy both Hera and device-side LLM on an NVIDIA GeForce RTX 4090 GPU, evaluating end-to-end performance and cost in a realistic device–cloud collaboration setting. 6.2 Overall Results 

Table 1 reports the average task success rate and cost of Hera and other baselines on ALFWorld, Web-Shop, and AppWorld benchmark, where cost includes closed-source cloud API fees and end-to-end latency. The device-side agent yields the smallest latency but the lowest success rates. By contrast, the cloud-only agent boosts success by 42.3%/31.8%/22.0% on ALFWorld/WebShop/AppWorld, respec-tively, at a per-task cost of 11.7/14.7/43.8 ×10 −3 dollars and a latency increase of 51.1/119.7/188.3 seconds. A random router dispatches each step to the device or cloud with a fixed probability (cloud-routing probability in parentheses), yielding a naive success–cost trade-off; however, without step-difficulty or model-preference awareness, it is inefficient—raising the cloud-routing probability from 0.3 to 0.5 improves success by only 5.5% on average while significantly increasing cloud calls, per-task cost, and latency. Notably, reduced cloud-based long-horizon planning lengthens trajectories and introduces redundant steps. The semantic-similarity-based router Eagle yields modest gains over random routing at comparable cloud usage, but still lags behind training-based methods in complex agentic environments. Training-based baselines achieve similar results; FrugalGPT performs d better by conditioning routing on device outputs, but incurs extra latency by waiting for device generation. By contrast, Hera consistently outperforms all baselines, achieving higher task success rates under substantially lower cost. It achieves 92.5% of the cloud-only success rate with only 53.6% of the cloud cost and 69.3% of the end-to-end latency on average. This gain is mainly attributed to the RL stage, whose end-to-end trajectory-level success–cost optimization enables Hera to learn not only state-wise routing decisions but also their interdependence across consecutive steps, ultimately yielding more precise and cost-effective device–cloud coordination.

## 7 Conclusions 

This paper proposes Hera, a step-level device–cloud LLM agent coordinator for long-horizon tasks. Specifically, Hera accurately identifies whether cloud assistance is needed at each interaction state, thus enabling effective task execution with minimal cloud invocations. Hera adopts a two-stage 9training paradigm. First, we perform cold-start imitation learning by replaying the device agent on cloud trajectories and using step-level action consistency as supervision, providing Hera with an initial device–cloud routing capability. Next, we refine Hera via cost-aware reinforcement learning: across multiple rollouts, we group identical states and generate step-level preferences that favor higher return and, when comparable, fewer future cloud calls, thereby explicitly co-optimizing success and cost. Extensive experiments on ALFWorld, WebShop, and AppWorld demonstrate that Hera consistently outperforms state-of-the-art baselines while invoking the cloud agent in only 46.3% of steps and achieving 92.5% of cloud-only success. We present the broader impact of Hera in Appendix A, and discuss the limitations of Hera and directions for future research in Appendix E.

12 A Broader Impacts 

This work introduces Hera as the first step-level device–cloud coordinator for LLM agents, marking a pivotal shift from coarse, task-level routing to fine-grained, dynamic collaboration. By demonstrating that only a small fraction of critical steps require cloud intervention, Hera significantly lowers the computational and financial barriers to deploying sophisticated agents on user devices. This efficiency democratizes access to advanced agentic capabilities, enabling real-time, latency-sensitive applications—such as autonomous robotic control and interactive assistants—to operate reliably on device hardware without prohibitive costs. Amid the explosive growth of LLM agents, artificial intelligence is undergoing a fundamental paradigm shift from static, single-turn queries to long-horizon autonomous interaction, where agents must sustain coherent reasoning over extended, dynamic trajectories. We believe Hera is essential for realizing scalable, sustainable, and ubiquitous autonomous agents in real-world applications by establishing a strong performance–cost Pareto frontier for this complex mode of operation.

C.2 Details of Experimental Settings 

For ALFWorld benchamrks ,we use a maximum prompt length of 2048 tokens and a maximum response length of 512 tokens. Each episode is capped at 50 environment interaction steps, counting both device- and cloud-executed actions. In WebShop, the prompt length is capped at 4096 tokens and the response length at 1024 tokens. Each episode terminates after 20 interaction steps. In AppWorld, we set the maximum prompt length to 4096 tokens, the maximum response length to 1024 tokens, and the maximum number of steps to 50. At each execution step, the agent is given access to a bounded history window. The number of visible historical steps is set to 50, 15, and 5 for the three datasets, respectively. To evaluate request-level baselines in long-horizon agent settings, we apply them independently at each interaction step. Specifically, at step t, each baseline receives the same per-step context as Hera —task instruction, current observation, and truncated history—and outputs a binary decision for routing the current step to either the device or the cloud model. The definition of an identical state in the RL training stage depends on the benchmark-specific environment representation. In ALFWorld, two states are considered identical if they correspond to the same textual environment observation and admissible action set. In WebShop, two states are considered identical if they have the same webpage observation, including the current page content and available actions. In AppWorld, two states are considered identical if they produce the same execution observation returned by the Python environment under the same task context. For device-side model deployment, we run the open-source MNN engine (Jiang et al., 2020) on a single NVIDIA RTX 4090 GPU using BF16 precision. Cloud-side closed-source models are invoked exclusively through their official APIs. All training experiments are performed on 2 NVIDIA H100 GPUs. Unless otherwise specified, the imitation learning and reinforcement learning phases are run for 30 and 50 iterations, respectively. In practice, however, we observe that the best-performing 15 checkpoint typically emerges much earlier, after only around 10 IL iterations and 15 RL iterations, highlighting the training efficiency of Hera.

C.4 Effect of Hyperparameters 

We investigate the sensitivity of Hera to the hyperparameter ϵ and β. ϵ governs the trade-off between task success and cloud cost in the RL stage: a small ϵ (e.g., 0) prioritizes success but incurs excessive cloud usage, while a large ϵ (e.g., 0.3) overly restricts offloading, leading to performance degradation. Similarly, β stabilizes RL updates against the IL initialization; removing regularization ( β = 0 )causes policy drift, whereas excessive regularization ( β=0.3) hinders adaptation to RL objectives. Empirical results in Table 4 indicate that the optimal configuration ( ϵ = 0 .05 , β = 0 .1) effectively balances these dynamics, allowing Hera to refine the IL prior into a robust, cost-efficient routing policy. This demonstrates that our design successfully leverages moderate regularization to prevent catastrophic forgetting while utilizing a fine-grained preference margin to identify critical states for cloud intervention, ensuring superior co-optimization of accuracy and efficiency. ## E Limitations and Future Work 

Limitations By studying fine-grained, step-level device–cloud coordination for long-horizon agents, Hera improves efficiency and establishes a favorable performance–cost Pareto frontier. However, several limitations remain. First, Hera assumes a synchronous, always-connected setting with reliable low-latency communication between device and cloud, and does not account for asynchronous execution or intermittent connectivity, which may limit robustness in mobile or remote environments. Second, Hera does not incorporate built-in privacy-preserving mechanisms for offloaded data. Al-though it reduces the frequency of cloud calls, it does not sanitize transmitted context, which may restrict deployment in privacy-sensitive domains such as healthcare or finance. Third, our study focuses on a binary single-device–single-cloud topology. As a result, Hera does not address routing and coordination in more complex multi-agent settings, such as systems with multiple edge devices, specialized services, or hierarchical cloud infrastructure, where coordination overhead can scale non-linearly. 24 Future Directions. To address these limitations, we outline several directions for future research in device–cloud collaboration for LLM agents. Building on Hera, we hope to further broaden the applicability, robustness, and scalability of step-level coordination, ultimately enabling more reliable and efficient autonomous systems in real-world environments. Future systems should move beyond the assumption of stable synchronous interaction and explicitly reason about latency, communication uncertainty, and temporary disconnection, allowing adaptive execution under weaker connectivity. It is also important to improve the safety of offloading itself. Incorporating privacy-aware representations, controllable context abstraction, or secure transmission and computation mechanisms would make such frameworks more suitable for domains where sensitive information cannot be directly shared with external providers. More broadly, the current setting opens up opportunities to study coordination at larger system scales. Extending beyond single-device–single-cloud topology toward heterogeneous networks of edge devices, specialized services, and hierarchical cloud resources will require new methods for routing, scheduling, and distributed planning, particularly when workflow dependencies and coordination costs grow rapidly with system size. 25
