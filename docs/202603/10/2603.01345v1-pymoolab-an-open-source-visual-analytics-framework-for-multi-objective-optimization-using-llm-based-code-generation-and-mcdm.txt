# PymooLab: An Open-Source Visual Analytics Framework for Multi-Objective Optimization using LLM-Based Code Generation and MCDM 

Thiago Santos, Sebasti˜ ao Xavier, Luiz Gustavo de Oliveira Carneiro, and Gustavo de Souza 

Abstract —Multi-objective optimization is now a core paradigm in engineering design and scientific discovery. Yet mainstream evolutionary frameworks, including pymoo , still depend on imper-ative coding for problem definition, algorithm configuration, and post-hoc analysis. That requirement creates a non-trivial barrier for practitioners without strong software-engineering training and often complicates reproducible experimentation. We address this gap through PymooLab, an open-source visual analytics environment built on top of pymoo . The platform unifies con-figuration, execution monitoring, and formal decision support in a single reproducible workflow that automatically records hyper-parameters, evaluation budgets, and random seeds. Its decoupled object-oriented architecture preserves compatibility with the base ecosystem while enabling LLM-assisted code generation for rapid model formulation. The interface also embeds interactive Multi-Criteria Decision Making (MCDM) tools, which reduces the cognitive burden of Pareto-front inspection. For computationally intensive studies, PymooLab relies on the native pymoo accel-eration pathway through JAX, improving scalability in high-dimensional evaluations. Overall, the framework combines visual experimentation, LLM-based modeling, and deterministic orches-tration to narrow the gap between rigorous operations research and practical accessibility for domain experts. Source code is publicly available at https://github.com/METISBR/pymoolab. 

Index Terms —Multi-objective Optimization , Evolutionary Al-gorithms , Python , Open-Source Framework , Computational Experimentation , Visual Analytics

II. R ELATED WORK 

Selecting an optimization framework is more than the mere identification of the fastest solver. That historical baseline is no longer sufficient. Contemporary analysis demands deep decision support, unified interfaces, and strict reproducibility. Today, workflow integrity frequently eclipses raw algorithmic output. Table I maps this evolving ecosystem. A decade ago, componentized execution defined the ecosys-tem. The jMetal libraries [13] established a highly durable object-oriented standard. Yet, coding raw experiments from scratch imposes severe operational friction. Tian et al. [14] recognized this limitation explicitly. By introducing PlatEMO ,they delivered a GUI-driven MATLAB environment that re-moved major onboarding barriers and tangibly accelerated comparative studies. As the scientific community gravitated toward Python, extensibility patterns shifted accordingly. Benitez-Hidalgo et al. [15] successfully translated the jMetal architecture into 

jMetalPy . Subsequently, pymoo [11] formalized the modular abstractions that currently anchor most Python-based empirical pipelines. Even so, a structural barrier remains. Purely code-centric pipelines impose steep operational constraints. They restrict practitioners who lack formal computer science back-grounds by requiring extensive boilerplate programming. Parallel development tracks focused on strictly method-ological domains. Campelo et al. [16] engineered a rigorous decomposition solver in R through MOEADr . Similarly, Pru-vost et al. [17] authored the MOEAD-framework in Python. Both represent highly reproducible pipelines. However, their design specifically targets optimization specialists rather than interdisciplinary decision-makers. Despite these independent advances, the landscape remains highly fragmented. Most toolchains optimize exclusively for a single dimension—whether that is algorithmic breadth, decomposition rigor, or execution speed—leaving prior and subsequent lifecycle stages entirely unmanaged. Consider platforms engineered strictly for throughput. The 

pagmo library [18] distributes parallel runs efficiently across heterogeneous hardware. ParMOO [20] bypasses heavy sim-ulation bottlenecks. Yet, their foundational designs treat algo-rithmic convergence as the final endpoint. This logic inherently produces a critical operational gap precisely when the analyst requires an interactive decision interface. Interactive environments actively address this gap. Misitano et al. [19] explicitly embedded human-in-the-loop workflows within their solvers via DESDEO . Recently, JECDM [21] di-rectly wired evolutionary search algorithms into formal multi-criteria evaluation protocols. This progression establishes a hard methodological mandate: computing mathematically op-timal trade-offs holds little practical value if the practitioner lacks the visual machinery to differentiate them. Huang et al. [22] demonstrated this explicitly: visual an-alytics routinely expose system convergence behaviors that scalar metrics fundamentally obscure. Addressing uncertainty, Forouli et al. [23] reached an identical conclusion via AUGMECON-Py. Raw numerical outputs simply do not con-stitute robust decision support. When examining broad ecosystem reviews [12], [24], the same systemic pathologies persist. Interoperability remains highly uneven. Reproducibility practices fluctuate erratically across laboratories, and reporting infrastructures stay severely fragmented. Han et al. [25] extended this critique, demon-strating empirically that software architecture acts as a core predictive variable. We can no longer classify implementation logic as a secondary detail. Consequently, the primary bottleneck in contemporary optimization does not stem from a shortage of available algorithms. The fundamental flaw resides in disconnected toolchains. The field lacks unified environments deliberately constructed to guide a problem from its analytical formulation, through deterministic execution, and directly into an auditable visual analysis. PymooLab addresses this exact rupture. We engineered it to retain the algorithmic core of pymoo , but removed the syntactic modeling friction by incorporating an LLM-assisted prompt agent. The framework enforces reproducibility by logging deterministic metadata during every individual execution. Finally, it couples natively with visual MCDM evaluation layers. A. Test Module 

We begin with a deliberately constrained experimental unit: one algorithm, one problem, and one run. This restriction is methodological rather than merely ergonomic. By suppressing early combinatorial variation, we can validate parameteriza-tion, backend compatibility, and indicator sensitivity before allocating computational budget to campaign-scale studies. Execution is asynchronous and event-driven, so the interface remains responsive while optimization is in progress. Interme-diate callbacks expose runtime progress and population/front updates in real time, while finalized payloads preserve seed, backend, and execution metadata for downstream reuse and reproducibility auditing. In this way, the Test Module functions as a pre-campaign validation gate rather than a simplified launcher. The diagnostic layer is objective-aware. Convergence is tracked in FE space through metric histories, two-objective outcomes are inspected in objective-space scatter plots, three-objective cases are rendered in interactive 3D views (when available), and higher-dimensional fronts are examined through parallel-coordinate visualizations. This multimodal evidence is practically important because non-convergence, front distortion, and indicator instability can be identified while corrective intervention is still inexpensive. B. Experiment Module 

The Experiment Module extends the interactive workflow from single-run diagnosis to controlled experimental cam-paigns involving multiple algorithms, multiple benchmark problems, and repeated runs under explicit seed policies. In contrast to conventional pymoo usage, where such campaigns are typically assembled through custom scripts, we provide an integrated GUI layer for campaign orchestration, execution governance, and result consolidation. We configure global execution parameters (e.g., number of runs and FE budget) once, while preserving per-problem overrides for population size and dimensional settings (M and D). The same benchmark can also be instantiated as multiple variants, which is particularly useful in synthetic quality studies when we need to compare algorithm behav-ior across objective-space and decision-space configurations without fragmenting the protocol into separate scripts. This integrated campaign-management capability is not natively provided by the original pymoo package, which is intentionally code-centric. Our design therefore targets a re-current bottleneck in empirical optimization research: the gap between algorithm implementation and reproducible, large-scale comparative experimentation. We also retain algorithm-native defaults during execution to reduce configuration-induced bias in baseline comparisons. At the same time, reproducibility is strengthened through explicit seed plans (random, fixed, or sequence) and bounded parallel execution, allowing us to run repeated experiments with predictable resource usage and stable runtime behavior. C. LLM Prompt Agent 

Although prompt engineering has established reliable struc-tural precedents in varied domains [26], their direct application to multi-objective optimization remains a developing frontier. We treat the prompt-agent interaction as a strict interface con-tract, where the semantic intent of the domain expert must be translated into an executable problem class without violating the structural constraints of the underlying framework. Traditional modeling bottlenecks are increasingly bypassed through high-capacity models such as the Qwen2.5-Coder and the emerging Qwen3 architectures, both developed by the Alibaba Group. These models, particularly those leveraging billions of parameters and specialized instruction-tuning for structural semantics [27], enable the autonomous conversion of natural-language specifications into vectorized Python code. Our im-plementation exploits this capability by establishing a veri-fication chain that enforces syntax stability, immediate com-pilation, and native compatibility with the pymoo vectorized evaluation contract. This multi-stage validation is essential because code that appears semantically plausible often fails at execution runtime—a recurring pathology in LLM-driven workflows that we mitigate through iterative feedback loops similar to those explored by Aytekin et al. [28]. When an artifact successfully traverses the verification gate, it is instantly integrated into the local execution registry and made available via hot reload. By automating the mechanical translation from mathematical spec to executable topology, PymooLab effectively collapses the latency traditionally asso-ciated with the modeling phase of the optimization lifecycle. 

D. Extensibility Module 

Modularity represents a non-negotiable prerequisite rather than a cosmetic feature in high-performance optimization environments. We prioritize extensibility as a core architectural mandate, enabling the seamless integration of methodological innovations—such as novel search operators or indicator vari-ants—without compromising the stability of the underlying search loops. Our implementation establishes explicit author-ing and validation pathways. Researchers should note, however, that while the inter-face materially reduces syntactic impediments, effective us-age remains contingent on a foundational proficiency in Python. Grasping the underlying mechanics of the pymoo 

core—particularly concerning objective vectorization and con-straint handling—remains essential to ensure that custom-formulated models remain computationally efficient and com-patible with accelerated backends. 

E. MCDM Module 

Our proposal currently implements a posteriori decision support through an embedded MCDM workspace integrated into the Analysis interface, rather than through a separate standalone tab. This design choice is deliberate. The same workspace already centralizes run filtering (algorithm, prob-lem, and trial), Pareto-front visualization, and statistical sum-maries, which allows decision support to operate directly on the inspected solution set without duplicating selectors or visualization logic. Within this integrated workflow, the user selects a completed run and applies an MCDM rule to the stored Pareto approx-imation. The current implementation supports TOPSIS and a normalized weighted-sum baseline with optional user-provided weights. PymooLab computes the compromise point over the final front, reports the selected index and score, highlights the chosen point in the two-objective Pareto chart, and stores a decision snapshot in JSON sidecar format for traceability. This capability positions MCDM as an operational continuation of analysis rather than an isolated post-processing script. The present scope remains intentionally focused on lightweight, auditable a posteriori selection. Objectives are treated as minimization targets and normalized before scoring, which is appropriate for comparative exploratory use but does not yet cover richer preference-elicitation protocols or interactive what-if sensitivity studies. Those extensions remain part of our forward development roadmap. IV. O PERATIONAL USABILITY AND WORKFLOW

A. Empirical Benchmarking and Statistical Synthesis 

The fundamental bottleneck in comparative benchmarking typically emerges during data collation and statistical format-ting. The Experiment Module systematically dismantles this barrier. After configuring an evaluation matrix—comprising multiple algorithms, specific benchmarks, allocated evaluation budgets, and fixed seed policies—the framework comprehen-sively executes the designated batch. Instead of mandating external data extraction scripts to parse disparate output files, PymooLab synthesizes the raw outcome tensors directly into dissemination-ready artifacts. The platform generates an integrated statistical summary table that reports indicator central tendencies and standard dispersions across all independent runs for each algorithm-problem pair. Crucially, researchers can capture this consoli-dated matrix instantaneously as raw CSV data or structurally complete L ATEX table code, inherently circumventing manual transcription vulnerabilities. B. Low-Overhead Formulation via the LLM Agent 

Today it is well-known that introducing novel conceptual definitions into an optimization environment frequently inter-rupts the analytical continuum, imposing severe imperative-coding constraints on practitioners. We bypass this limita-tion through an integrated LLM Prompt Agent. Researchers establish semantic parameters—such as decision variables, equality bounds, and mathematical objectives—using direct natural language within the graphical interface. Our agent subsequently parses these unstructured require-ments. It synthesizes a formally valid Python class that rig-orously adheres to the pymoo vectorized evaluation contract. Before local deployment, we subject the instantiated code to a stringent internal validation sequence that corroborates syntax stability and execution mechanics. Once verified, the artifact integrates into the local execution registry using a hot-reload architecture. The newly formulated model thus becomes avail-able across all experimental modules immediately, effectively eliminating standard script-and-restart disruption cycles. C. A Posteriori Selection (MCDM) 

Theoretical Pareto fronts remain abstract manifolds until decision-facing interpretation resolves them into actionable choices. To bridge this, we embedded MCDM controls directly into the Analysis & MCDM workspace. It was a deliberate move to prioritize workflow continuity over the use of isolated scripts. By keeping these selection tools within the same substrate where run-level data resides, we guarantee that the jump from discovery to final decision is auditable. The sequence is straightforward. After selecting an algo-rithm and run, a researcher applies rules like TOPSIS to the persisted Pareto set. Figure 4 demonstrates this using the canonical ZDT1 benchmark. For bi-objective or tri-objective problems, the compromise point is visually anchored on the manifold. But many-objective optimization presents a different barrier. In scenarios where dimensionality exceeds three, the platform abandons Cartesian projection in favor of a dense scalar report. This report maps the chosen point’s values against each objective, keeping the decision logic transparent even when the geometry is invisible. This isn’t just a display; it’s an active decision gate. Because the workspace operates on persisted payloads, every selection remains traceable. We store these decisions in JSON sidecar records, preserving the chosen point as well as the weights and methods used to derive it. This aligns with the reproducibility standards we set for the rest of the lifecycle. At this stage, our focus is on traceability rather than interactive preference-learning, although richer protocols and sensitivity analyses remain part of our forward pipeline. V. C ONCLUSION 

The academic community has reached a crossroads where another search heuristic, however sophisticated, creates dimin-ishing returns without a rigorous ecosystem to support it. In architecting PymooLab, we consciously abandoned the chase for yet another niche solver. Our work centers on something else entirely: a unified, auditable lifecycle. We believe that if research workflows are to survive the complexity of modern industrial studies, the distance between modeling and analysis must be collapsed into a single substrate. This isn’t a minor refinement; it is a fundamental rejection of the fragmented, code-heavy prototyping that has slowed operations research for decades. Static code blocks and manual transcription—these are where research continuity goes to die. By forcing an LLM-led agent into the very center of the search envelope, our framework effectively destroys the latency that usually par-alyzes problem formulation. But speed is just the surface.
